# test-design

単体テストを実装する前に **「何を、どのようにテストすべきか」** を言語化するためのスキル。テスト対象を理解し、**ブラックボックス・ホワイトボックスの両観点** から具体的なテストケース一覧を導出する。

---

## ワークフロー全体像

1. Step 1: テスト対象の理解
2. Step 2: テスト設計 → テストケース一覧出力
3. テスト実装（別スキル: test-implementation）

Step 1 からテストケース一覧の出力まで、レビューで区切らず一気通貫で進める。

---

## Step 1: テスト対象の理解

### 目的

テスト対象（System Under Test, 以下 SUT）が **何を入力に取り、どう振る舞い、何を出力するか** を把握する。実装コードがある場合は、加えて **内部の分岐構造** も把握する。前者はブラックボックステストの、後者はホワイトボックステストの材料になる。

### Step 1-1: 入力ソースの決定

分析の根拠となる情報を以下の優先順で確保する。

1. **仕様書がある場合** — 仕様書を主、実装コードがあれば補助的に参照
2. **仕様書がない場合** — 実装コードがあれば、コードから振る舞いを推定
3. **仕様書も実装もない場合** — ユーザーに振る舞いをヒアリングしてから進む

仕様書と実装が矛盾している、または仕様が曖昧なときは **推測せずユーザーに確認する**。テストは仕様の代弁者なので、推測で書かれたテストは将来の仕様変更で誤警報を出すか、バグを見逃す。

### Step 1-2: 整理する項目

以下を整理する。「コード構造」は **実装コードがある場合のみ**。

| 区分 | 整理する内容 | 用途 |
|---|---|---|
| 入力 | 引数・受け取る状態・依存（DB/外部）の取り得る値の範囲 | ブラックボックス |
| 出力 | 戻り値・副作用（DB書込み・イベント発火・例外）の期待 | ブラックボックス |
| 振る舞い | 「この入力ならこう振る舞う」という仕様レベルの記述 | ブラックボックス |
| コード構造 *(コードがある時のみ)* | 分岐（if/case/guard）・条件式・ループ・早期return・例外パス | ホワイトボックス |

実装コードが無い場合は「コード構造」を空にし、Step 2 のホワイトボックス工程はスキップする（ブラックボックスのみで設計する）。

---

## Step 2: テスト設計 → テストケース一覧

### 目的

Step 1 の理解を、**具体的な入力値・期待結果を持つテストケース** に変換する。

### Step 2-1: ブラックボックスでケース導出

仕様（入力・出力・振る舞い）をもとに、`./reference.md` の **「ブラックボックス技法」** から観点の性質に合った技法を選び、テストケースを導出する。仕様だけでも実施できる。

### Step 2-2: ホワイトボックスで網羅ギャップを埋める（コードがある時のみ）

Step 2-1 のケースで実装コードの分岐をどれだけ通せるかを確認し、**未到達の分岐・条件を通すケースを追加する**。網羅基準は `./reference.md` の **「ホワイトボックス網羅基準」** を参照し、以下の優先順で適用する。

- **分岐網羅（C1）を基本目標** にする（各 if/case/guard の真偽両方を通す）
- 複合条件（`a && b` 等）が絡む箇所は **条件網羅** まで踏み込む
- パス網羅は組合せが爆発しがちなので、ビジネス上意味のある経路に絞る

**重要**: ホワイトボックスは **ケースを導くため** に使う。検証対象（アサーション）は **振る舞いベースのまま** にする。内部状態・private メソッド・呼び出し回数をアサーションにしない — リファクタリングのたびに壊れる誤警報源になる。検証するのは「外から見える出力・状態」。

### Step 2-3: 重複を統合してテストケース一覧を出力

ブラックボックスとホワイトボックスで同じになるケースは1つにまとめ、以下の形式で出力する。

```markdown
## テストケース一覧: `クラス名#メソッド名`

| #  | テストケース名 | 前提条件 | 入力 | 期待結果 | 技法/網羅 |
|----|--------------|---------|------|---------|----------|
| 1  | amount=1（最小有効値）で残高が減る | 残高=1000 | amount=1 | 残高=999, 戻り値=999 | 境界値 / C1分岐T |
| 2  | amount=残高ぴったりで残高=0になる | 残高=1000 | amount=1000 | 残高=0, 戻り値=0 | 境界値 |
| 3  | amount=0 で ArgumentError | 残高=1000 | amount=0 | ArgumentError, 残高=1000（不変） | 同値分割 / C1分岐F |
| 4  | 残高不足で例外、残高は変化なし | 残高=500 | amount=501 | InsufficientFundsError, 残高=500 | 同値分割 / C1分岐 |
```

「技法/網羅」列に、各ケースがどのブラックボックス技法・どのホワイトボックス網羅（例: C1分岐の真/偽）に対応するかを書く。網羅性が一目で分かり、抜け漏れをレビューしやすくなる。コードが無い場合はブラックボックス技法のみを記す。

## テスト可用性が低い場合の中断

設計の途中で SUT のテスト可用性が著しく低いと判断したら、**作業を中断してユーザーに SUT 自体の見直しを提案する**。無理にテストを設計しても、実装段階で破綻するか、過剰なモック・複雑なセットアップで保守困難なテストが生まれる。

### 中断を検討すべき兆候

- **責務が多すぎる** — 1メソッド/クラスが複数の独立した責務を担っており、テストケースが爆発する
- **セットアップが過剰** — テストを成立させる前提条件が多すぎる
- **テストケースが多過ぎる**

### 中断時の振る舞い

1. 現時点までの理解・設計結果をユーザーに提示する
2. テスト可用性が低いと判断した **具体的箇所** と、**なぜ問題か** を明示する
3. SUT の見直し方向を提案する
4. ユーザーの判断を仰ぐ。選択肢として「SUTを直す」「テスト粒度を変える（結合テストに格下げ等）」「このまま無理して進める」を提示する

- テスト実装に進まない: 実装は test-implementation スキルの担当。
- 仕様の曖昧さを推測で埋めない: 「たぶんこういう仕様だろう」と思った瞬間にユーザー確認に切り替える。
- ホワイトボックスで内部実装をアサーションにしない: コード構造はケース導出に使うだけ。検証対象は振る舞い。
- テスト可用性の低さを無視して突き進まない: 上記「テスト可用性が低い場合の中断」を発動する。

# ブラックボックス技法（仕様ベース）

各観点に対して、**観点の性質** を見極めて技法を選ぶ。1つの観点で複数の技法が同時に当てはまることが多い。

## 技法選択早見表

各観点を以下の表で分類し、対応する技法を選ぶ。

| 観点の性質                            | 第一選択                  | 補完          |
|----------------------------------|------------------------|--------------|
| 数値・日付の範囲チェック              | 境界値分析 + 同値分割       | エラー推測     |
| 列挙型・カテゴリの入力                | 同値分割                  | エラー推測     |
| 複数条件の組み合わせで結果が変わる     | デシジョンテーブル          | エラー推測     |
| 状態を持つオブジェクトの振る舞い       | 状態遷移テスト              | エラー推測     |
| 独立パラメータが多い設定機能           | Pairwise / 直交表          | 境界値分析     |
| **すべての観点**                    | —                       | **エラー推測（必ず最後に実施）** |

## 各技法の判定基準と適用法

### 1. 同値分割 + 境界値分析

**判定基準**: 入力ドメインに「同じ振る舞いをするグループ（同値クラス）」が存在する場合。特に数値範囲・日付範囲・文字列長など連続値を受け取るときは必須。

**手順**:
1. 入力を同値クラスに分割（有効クラス + 各種無効クラス）
2. 各クラスから代表値を1つ選ぶ
3. 境界の **内側・境界そのもの・境界の外側** をテスト

**例（年齢: 0〜150 が有効）**:

| 入力値 | 分類             | 期待結果 |
|--------|-----------------|---------|
| -1     | 無効（下境界の外）  | invalid |
| 0      | 有効（下境界）     | valid   |
| 1      | 有効（境界+1）     | valid   |
| 75     | 有効(同値クラス代表) | valid |
| 149    | 有効（境界-1）     | valid   |
| 150    | 有効（上境界）     | valid   |
| 151    | 無効（上境界の外）  | invalid |
| "abc"  | 無効（型違い）     | invalid |

**重要ルール**: 同値分割を適用するときは **境界値分析を必ず併用する**。境界の前後はバグが集中する場所。

### 2. デシジョンテーブル

**判定基準**: 結果を決める条件が複数あり、それらの AND/OR の組み合わせでロジックが分岐する場合。単一の if 文ではなく、独立した複数条件が絡むときに効く。

**手順**:
1. 条件を列挙する
2. 全組合せをテーブル化
3. ビジネス上意味のある組合せに絞ってテストケース化（論理的に不可能な組合せは除外）

**例（割引判定: 会員ランク × 購入金額）**:

| 条件 / 結果              | C1   | C2   | C3  | C4  |
|------------------------|------|------|-----|-----|
| プレミアム会員？          | Y    | Y    | N   | N   |
| 購入金額 ≥ 10,000 円？   | Y    | N    | Y   | N   |
| **割引率**              | 20%  | 10%  | 5%  | 0%  |

### 3. 状態遷移テスト

**判定基準**: SUT が状態を持ち、操作によって状態が変化し、かつ **状態によって同じ操作の振る舞いが変わる** 場合。

**手順**:
1. 状態と遷移イベントを書き出す（状態遷移図/表）
2. **正常な遷移** と **無効な遷移（許可されない状態変化）** の両方をテスト
3. 状態に応じた振る舞いの違いも観点に加える

**例（注文の状態）**:

| 初期状態     | イベント       | 期待結果                    |
|------------|--------------|---------------------------|
| 受付         | 支払い完了    | 支払済（正常遷移）            |
| 支払済       | 出荷         | 出荷済（正常遷移）            |
| 受付         | 出荷         | エラー（無効: 支払前は出荷不可）|
| 完了         | キャンセル    | エラー（無効: 完了済は不可逆）  |

**重要ルール**: **無効な遷移を必ず入れる**。正常系だけ書くと状態管理のバグを検出できない。

### 4. Pairwise / 直交表

**判定基準**: 独立したパラメータが3つ以上あり、全組合せ（直積）をテストすると現実的でない数になる場合。「2因子間の全組合せ」までカバーすればバグの大半が検出できる、という経験則を活用する。

**手順**:
1. パラメータと取り得る値を列挙
2. PICT・Allpairs などのツール、または直交表 L_n を使って組合せを縮約
3. 全組合せでなく「2因子間の網羅」を達成する最小ケース集を生成

**例**: ブラウザ（3種）× OS（3種）× 言語（4種）× 通貨（5種）= 全 180 組合せ → Pairwise で約 20 ケースに縮約。

### 5. エラー推測

**判定基準**: 上記の体系的技法で拾えない、経験的に「バグが潜みそう」な箇所を補強したいとき。**常に最後の補完として実施する**。これ単独で網羅性は担保できない。

| カテゴリ    | 推測すべき値                                                       |
|-----------|------------------------------------------------------------------|
| null / 空 | nil, null, undefined, ""（空文字）, []（空配列）, {}（空ハッシュ）        |
| 数値       | 0, -0, -1, Integer の最大/最小, 浮動小数点誤差 (0.1 + 0.2 ≠ 0.3)        |
| 文字列     | 全角文字, 絵文字, 改行コード (CR/LF/CRLF), 制御文字, 極端に長い文字列, BOM |
| 日時       | 閏年 2/29, 月末日, 夏時間切替, タイムゾーンまたぎ, Unix epoch (0/負数)   |
| 並行性     | 同時実行, リトライ, 競合条件, デッドロック                            |
| エンコード | UTF-8/UTF-16, サロゲートペア, 不正なバイト列                           |

# ホワイトボックス網羅基準（コードベース）

実装コードがある場合に、ブラックボックスで出したケースが **コードのどの分岐を通っているか** を確認し、未到達の分岐・条件を通すケースを追加するための基準。

**位置づけ**: ホワイトボックスは **ケースを導くため** に使う。検証対象（アサーション）は振る舞いベースのままにする。「内部でこのメソッドが呼ばれた」ではなく「外から見える出力・状態」を検証する。コード構造に依存したアサーションはリファクタリングで壊れる誤警報源になる。

## 網羅基準の種類と使い分け

| 網羅基準 | 何を網羅するか | 使いどころ |
|---|---|---|
| 命令網羅 (C0) | すべての文を1回以上実行 | 最低限。これだけでは分岐の偽側を見逃す |
| 分岐網羅 (C1) | すべての分岐の真・偽の両方を実行 | **基本目標**。if/case/guard の両側を通す |
| 条件網羅 | 複合条件の各条件式の真・偽を網羅 | `a && b` 等、複数条件が絡む分岐 |
| パス網羅 | 実行経路の組合せを網羅 | 経路が少なく重要な箇所に限定（全網羅は爆発する） |

**適用の優先順**:
1. 分岐網羅（C1）を基本目標にする
2. 複合条件が絡む箇所は条件網羅まで踏み込む
3. パス網羅は、組合せが現実的かつビジネス上意味のある経路に絞る

## 適用例

次のメソッドを考える。

```ruby
def shipping_fee(weight, premium_member)
  return 0 if premium_member
  if weight > 10
    1000
  else
    500
  end
end
```

分岐を洗い出す:
- B1: `premium_member` の真 / 偽
- B2: `weight > 10` の真 / 偽（B1が偽のときのみ到達）

分岐網羅（C1）を満たす最小ケース:

| # | 入力 | 通る分岐 | 期待結果 |
|---|------|---------|---------|
| 1 | premium_member=true, weight=任意 | B1真 | 0 |
| 2 | premium_member=false, weight=11 | B1偽, B2真 | 1000 |
| 3 | premium_member=false, weight=5 | B1偽, B2偽 | 500 |
