# test-design スキル再設計: 契約ベース → ブラックボックス/ホワイトボックス

- 日付: 2026-05-23
- 対象スキル: `.claude/skills/test-design/`（`SKILL.md` + `reference.md`）
- 種別: 既存スキルの方法論再設計

## 背景と動機

現行の test-design スキルは Phase 1（テスト分析）の土台に **Design by Contract（DbC: 事前条件・事後条件・不変条件）** を採用している。これが「大げさ・学術的」で、テスト観点を洗い出すにはオーバースペックという課題がある。

より一般的でなじみのある **ブラックボックス/ホワイトボックス** というテスト分類に土台を差し替え、フローも簡素化する。

## 変更後の全体像

### フロー（2ステップ・一気通貫）

```
Step 1: テスト対象の理解
  ├─ 入力ソース確保（仕様書 / コード / ヒアリング）
  ├─ 仕様の整理（入力・出力・前提・期待される振る舞い）= ブラックボックスの材料
  └─ コードがある場合のみ: 構造の把握（分岐・条件・パス）= ホワイトボックスの材料

Step 2: テスト設計 → テストケース一覧
  ├─ ブラックボックス観点でケース導出（技法ガイド参照）
  ├─ コードがある場合のみ: ホワイトボックス観点で網羅ギャップを埋める
  ├─ 重複を統合し1つのテストケース一覧に
  └─ 出力

（テスト実装は test-implementation スキル: スコープ外）
```

- **レビューゲートなし・モード分岐なし**で一気通貫に実行する。
- ホワイトボックスは **実装コードがある時のみ**実施。無ければブラックボックスのみ（コードの有無で自動切り替え）。

### 残すもの（現行から踏襲）

- テスト可用性が低い場合の中断（責務過多・セットアップ過剰・ケース過多の検知）
- 出力先 `./tmp/test-design/`
- 入力ソースの優先順（仕様書 > コード > ヒアリング）
- 「仕様の曖昧さを推測で埋めない」原則（矛盾・曖昧時はユーザー確認）
- 「実装詳細をテストしない」原則（置き場所は変更。下記参照）

### 消すもの（現行から削除）

- 契約の明文化（事前/事後/不変条件の表）
- 「契約に含める/含めない」の線引き表
- 中間成果物「テスト観点リスト」と、そのレビューゲート
- 対話/非対話モードの概念とトリガー判定
- 各 Phase 末尾のユーザーレビューゲート

## Step 1: テスト対象の理解

BB と WB 両方の材料を集める。

| 区分 | 整理する内容 | 用途 |
|---|---|---|
| 入力 | 引数・受け取る状態・依存（DB/外部）の取り得る値の範囲 | BB |
| 出力 | 戻り値・副作用（DB書込み・イベント・例外）の期待 | BB |
| 振る舞い | 「この入力ならこう振る舞う」の仕様レベルの記述 | BB |
| コード構造 *(コードがある時のみ)* | 分岐（if/case/guard）・条件式・ループ・早期return・例外パス | WB |

入力ソースの優先順:
1. 仕様書あり → 仕様書を主、コードは補助
2. 仕様書なし・コードあり → コードから振る舞いを推定
3. 両方なし → ユーザーにヒアリング

仕様とコードが矛盾、または仕様が曖昧なときは **推測せずユーザーに確認**する。

## Step 2: テスト設計 → テストケース一覧

### 導出フロー

1. **ブラックボックスでケース導出** — 仕様（入力・出力・振る舞い）から技法ガイド（境界値・同値分割・デシジョンテーブル・状態遷移・Pairwise・エラー推測）を使ってケースを出す。仕様だけでも実施可能。

2. **ホワイトボックスで網羅ギャップを埋める**（コードがある時のみ）— BB で出したケースでコード構造をどれだけ通せるか確認し、未到達の分岐・条件を通すケースを追加する。網羅基準の優先順:
   - **分岐網羅（C1）を基本目標**にする（各 if/case/guard の真偽両方を通す）
   - 複合条件（`a && b` 等）が絡む箇所は **条件網羅**まで踏み込む
   - パス網羅は全パス爆発しがちなので、ビジネス上意味のある経路に絞る

3. **重複を統合**して1つの一覧にする。BB と WB で同じケースになるものはまとめる。

中間表（分岐 → 到達ケースのマッピング表）は **出さない**（軽量重視）。最終のケース一覧のみを出力する。

### 「実装詳細をテストしない」原則の置き場所

現行の「契約に含める/含めない表」は廃止するが、原則自体はこのステップの注意書きとして残す:

> ホワイトボックスはケース導出に使うが、検証するのは振る舞い。内部状態・private メソッド・呼び出し回数をアサーションにしない（リファクタで壊れる誤警報源になる）。

### 出力フォーマット

現行のケース一覧表に **「技法/網羅」列を追加**する。各ケースがどの技法・どの分岐網羅に対応するかを明示し、網羅性を一目で分かるようにする。

```markdown
## テストケース一覧: `クラス名#メソッド名`

| #  | テストケース名 | 前提条件 | 入力 | 期待結果 | 技法/網羅 |
|----|--------------|---------|------|---------|----------|
| 1  | amount=1（最小有効値）で残高が減る | 残高=1000 | amount=1 | 残高=999, 戻り値=999 | 境界値 / C1分岐T |
```

### 出力先

- `./tmp/test-design/<SUT名>-cases.md` — テストケース一覧

`<SUT名>` はテスト対象のクラス・メソッド名から派生（例: `account-withdraw`）。
（観点リストの中間ファイルは廃止のため、出力は原則ケース一覧のみ。）

## reference.md（技法ガイド）の改修

2部構成にする。

```
# テスト技法選択ガイド

## ブラックボックス技法（仕様ベース）
  - 既存の早見表 + 各技法（同値分割/境界値、デシジョンテーブル、状態遷移、Pairwise、エラー推測）
  - ※ほぼ現状維持（見出しを「ブラックボックス技法」配下に整理）

## ホワイトボックス網羅基準（コードベース）★新規
  - 命令網羅(C0) / 分岐網羅(C1) / 条件網羅 / パス網羅 の定義と使い分け
  - 「分岐網羅を基本目標、複合条件は条件網羅まで」の指針
  - 簡単な適用例（if/case/guard を含む小さな Ruby メソッド → どのケースで各分岐を通すか）
  - 「ケース導出に使い、検証は振る舞いベース」の再掲
```

ブラックボックス部分は既存内容をほぼ維持する。ホワイトボックス部分を新規追加する。

## SKILL.md frontmatter（description）への影響

`description` から「契約による設計」「テスト分析とテスト設計の2フェーズ」という表現を削除し、「ブラックボックス・ホワイトボックス両面でテストケースを設計する」旨に更新する。トリガー文言（「テスト設計して」「テスト観点を洗い出して」等）は維持する。

## スコープ外

- テスト実装（test-implementation スキルの担当）
- description のトリガー精度最適化（必要なら別途）

## 成功基準

- DbC 由来の用語・表（事前/事後/不変条件、契約の含める/含めない表）が SKILL.md から消えている
- ブラックボックス + ホワイトボックスの2軸でケースが導出される
- コードがある場合は分岐網羅ベースのケースが追加され、出力一覧に「技法/網羅」列で根拠が示される
- コードが無い場合はブラックボックスのみで破綻なく動作する
- 検証対象が振る舞いベースに保たれている（内部状態のアサーションを推奨しない）
