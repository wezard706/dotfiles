# ddd-modeling スキル再設計

**作成日:** 2026-03-21
**対象スキル:** `ddd-modeling`
**目的:** 成果物をテーブル設計・クラス設計の入力として使えるレベルまで引き上げる

---

## 背景と問題

現行の ddd-modeling スキルは戦略的設計（BC識別・ユビキタス言語・コンテキストマップ）を成果物として出力するが、粒度が粗すぎて次のステップ（テーブル設計・クラス設計）に直接入れない。また、ワークフローの順序に以下の問題がある：

1. **Event Storming の位置が遅すぎる** — BC・ユビキタス言語を Event Storming 前に確定させているが、実際には ES によって境界や用語が精錬される
2. **ユビキタス言語を BC より先に定義できない** — BC が決まる前は「どのコンテキストの言語か」が不明
3. **BC を暫定・確定の2段階に分けていない** — ES の結果で境界が変わるのが正常なため、手戻りが大きくなる

---

## 設計方針

- ワークフローを DDD の実践的な順序に再構築する
- Design Level Event Storming をコアドメイン・重要支援ドメインに適用し、集約・コマンド・ドメインイベント・ポリシー・リードモデルを明確化する
- 成果物に「戦術的設計への入力」セクションを追加し、クラス設計・テーブル設計の起点となる情報を提供する
- 規模によるショートカットは設けず、常に8ステップを踏む

---

## 新ワークフロー（8ステップ）

```
1. ドメイン理解
   ビジネス目的・主要アクター・主な業務フローを把握

2. サブドメイン分類
   コア / 支援 / 汎用 に分類
   判断基準：競争優位性の源泉か、既製品で代替可能か

3. BC識別（暫定）
   サブドメインをもとに境界を仮定
   「1サブドメイン = 1BC」にならないケースを意識的に検討

4. Design Level Event Storming
   コアドメインと重要な支援ドメインに適用
   汎用ドメイン（Auth・決済等）はスキップ
   展開順序：アクター → コマンド → ドメインイベント → 集約 → ポリシー → リードモデル

5. BC境界の見直し・確定
   ES で判明した「この BC の境界、違くない？」を反映
   暫定 BC からの変更点と理由を記録

6. ユビキタス言語定義
   確定した BC 別に用語を整理
   同一用語が複数 BC で異なる意味を持つ場合は明示

7. コンテキストマップ作成
   全 BC が確定してから BC 間の関係をモデル化
   ACL / CF / OHS-PL / Shared Kernel / Customer-Supplier 等のパターンを適用

8. 成果物出力
   下記フォーマットで出力
```

## 成果物フォーマット

```markdown
# DDDモデリング成果物

## 1. ドメイン概要
ビジネス目的と主要業務フローのサマリー

## 2. サブドメイン分類
| サブドメイン | 種別 | 理由 |
|---|---|---|
| 注文管理 | コア | 競争優位性の源泉 |
| 在庫管理 | 支援 | 自社固有ルールあり |
| 認証 | 汎用 | Auth0で代替可能 |

## 3. 境界づけられたコンテキスト（確定版）
各 BC について：
- 名前・責務
- 対応するサブドメイン
- 暫定 BC からの変更点（ES で判明した理由）

## 4. Event Storming 結果（BC別）
コアドメイン・重要支援ドメインのみ記載

### [BC名]
| 種別 | 名前 | 補足 |
|---|---|---|
| アクター | 購入者 | |
| コマンド | 注文を確定する | 購入者が実行 |
| ドメインイベント | 注文が確定された | 過去形・不変の事実 |
| 集約 | 注文 | コマンドを処理し一貫性を保証 |
| ポリシー | 在庫を確保する | 注文確定イベント発火時に実行 |
| リードモデル | 注文確認画面 | 購入者が参照 |

## 5. ユビキタス言語（BC別）
| 用語 | BC | 定義 |
|---|---|---|
| 注文 | 注文コンテキスト | 購入者が商品を購入する意思表示と確定された取引 |
| 注文 | 配送コンテキスト | 配送すべき荷物の単位（購入とは無関係） |

## 6. コンテキストマップ
BC 間の関係（パターン名付き）

例：
- 注文コンテキスト → [ACL] → 配送コンテキスト
- 注文コンテキスト → [CF] → 決済コンテキスト（Stripe）

## 7. 戦術的設計への入力

### 集約候補（クラス設計の入力）
| 集約 | 属するBC | 主要コマンド | 不変条件の候補 |
|---|---|---|---|
| 注文 | 注文コンテキスト | 確定・キャンセル | 確定後は変更不可 |
| 在庫 | 在庫コンテキスト | 確保・解放 | 在庫数は0以上 |

### 集約の状態（テーブル設計の入力）
| 集約 | 属性 | 型の候補 | 備考 |
|---|---|---|---|
| 注文 | 注文ID | UUID | PK |
| 注文 | ステータス | Enum | 確定/キャンセル/配送中 |
| 注文 | 購入者ID | UUID | FK |
| 注文 | 確定日時 | Timestamp | |

### ポリシー一覧（ドメインサービス候補）
| ポリシー | トリガーイベント | 実行コマンド |
|---|---|---|
| 在庫を確保する | 注文が確定された | 在庫確保コマンド |

### リードモデル一覧（クエリオブジェクト/ビュー候補）
| リードモデル | 利用するアクター | 必要なデータ |
|---|---|---|
| 注文確認画面 | 購入者 | 注文明細・合計金額・ステータス |
```

## 成功基準

- Event Storming の結果から「どのクラスを作るか」が明確にわかる
- Event Storming の結果から「どのテーブルを作るか」が明確にわかる
- BC境界の変更理由が成果物に記録されている
- ユビキタス言語が BC 別に整理されており、同一用語の異なる定義が明示されている
