# 開発原則

## 基本原則

- **変更容易性（ETC原則）**が最も重要です。将来の変更コストを常に意識して設計して下さい
- **戦略的プログラミング**: 短期的な成果よりも長期的な成果をもたらす理想的な設計を重要視して下さい
- **モデリング**: 良い設計は良いモデルから生まれます。実装を行う前に必ずモデリングを行って下さい。

## 命名原則

- 名前で「仕様を語る」ことを心掛けて下さい
- 「何（What/How）」より「なぜ（Why/Purpose）」を意識した名前にして下さい
- 技術用語よりもビジネス用語を使って下さい

## 重要なプログラミング原則

- 単一責任の原則を意識して下さい
- Tell, Don't Ask を意識して下さい
- 継承よりも集約を優先して下さい
- 機能的凝集を目指して下さい。特に、偶発的凝集、論理的凝集は避けて下さい
- スタンプ結合、データ結合、メッセージ結合を目指して下さい。特に、内容結合、共通結合、外部結合は避けて下さい

## コードコメント規約

**コメントは原則書かない。** 書く場合は「記述してよいコメント」のいずれかに該当することを正当化できる場合のみ。迷ったら書かない。

仕様・意図の表現手段はコメントではなく以下を使う:
- 振る舞いの仕様 → クラス名・メソッド名・テストケースの記述
- 変更理由の Why → コミットメッセージのボディ・PR説明

ただし「記述してよいコメント」に該当する場合は、この一般則より優先してコメントに書く。なお、本規約内の例示コードはコメント可否のみを示すものであり、実装パターンの推奨ではない。

判定基準:
- コードから自明なことは書かない。そのコメントを消して、コードとテストだけでレビュアーが理解できるなら不要
- コメントはコードと同じ抽象度で書かない。コードより精密（単位・境界・不変条件）か、コードより深い（採用しなかった選択肢とその理由）のいずれかであること。同じ抽象度のコメントは言い換えであり禁止

### 記述してよいコメント

設計判断の根拠: 複数の選択肢から自明でないトレードオフで選んだ理由。コードから読み取れない「なぜ他の方法にしなかったか」。

```ruby
# 退会ユーザー除外はサブクエリ（user: User.active）で行う。joins(:user)にしないのは、
# 他のscopeが独自にusersをJOINしており、JOINが衝突するため。
```

非自明な制約: 呼び出し順序・状態・事前条件・副作用・例外など、名前から推測できない暗黙の依存。

```ruby
# mutex.lock より先に呼ぶこと。逆順だとデッドロックする (issue #482)。
```

精密化: 名前や型で表現しきれない精密な情報。単位、境界値の含む/含まない、nil が意味すること、不変条件など。

```ruby
# 単位はミリ秒。0 はタイムアウト無効を意味する
attr_reader :timeout
```

### 記述してはならないコメント

コードの言い換え: コードを自然言語で繰り返すコメントは記述しない。命名やリファクタリングで解決すること。

```ruby
# 不可
user = get_user(id)  # ユーザーを取得する

# 可
user = get_user(id)
```

仕様・ビジネスルールの説明: コードが実現する仕様をコメントで説明しない。「Whyを述べているから良いコメント」ではない——実装の選択理由（トレードオフ）以外のWhyはコメントに書かない。仕様はクラス名・メソッド名・テストが語る。

```ruby
# 不可（仕様のナレーション。メソッド名とテストで表現できる）
# 退会コーチのプランは一覧に出さない
scope :listable, -> { joins(:coach).merge(Coach.active) }

# 不可（メソッドの動作説明 + 仕様の言い換え）
# 退会済みユーザーのプロフィールは直リンクでも参照不可（find_by!でRecordNotFoundを発生させる）
user = User.active.find_by!(id: id)

# 可（コメントなし。仕様はテストケースとして記述する）
user = User.active.find_by!(id: id)
```
