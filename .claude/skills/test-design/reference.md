# テスト技法選択ガイド

Step 2 で各テスト観点をテストケースに変換するときに、適用すべきテスト技法を選ぶための判断ガイド。

テスト技法は2系統ある。

- **ブラックボックス技法（仕様ベース）** — 入力・出力・振る舞いの仕様からケースを導く。仕様だけでも適用できる。
- **ホワイトボックス網羅基準（コードベース）** — 実装コードの分岐構造から、未到達の分岐を通すケースを導く。コードがある時のみ適用する。

ブラックボックスでケースを出し切ったうえで、コードがある場合はホワイトボックスで網羅ギャップを埋める、という順で使う。

# ブラックボックス技法（仕様ベース）

各観点に対して、**観点の性質** を見極めて技法を選ぶ。1つの観点で複数の技法が同時に当てはまることが多い。

## 技法選択早見表

各観点を以下の表で分類し、対応する技法を選ぶ。

| 観点の性質                            | 第一選択                  | 補完          |
|----------------------------------|------------------------|--------------|
| 数値・日付の範囲チェック              | 境界値分析 + 同値分割       | エラー推測     |
| 列挙型・カテゴリの入力                | 同値分割                  | エラー推測     |
| 複数条件の組み合わせで結果が変わる     | デシジョンテーブル          | エラー推測     |
| 状態を持つオブジェクトの振る舞い       | 状態遷移テスト              | エラー推測     |
| 独立パラメータが多い設定機能           | Pairwise / 直交表          | 境界値分析     |
| **すべての観点**                    | —                       | **エラー推測（必ず最後に実施）** |

## 各技法の判定基準と適用法

### 1. 同値分割 + 境界値分析

**判定基準**: 入力ドメインに「同じ振る舞いをするグループ（同値クラス）」が存在する場合。特に数値範囲・日付範囲・文字列長など連続値を受け取るときは必須。

**手順**:
1. 入力を同値クラスに分割（有効クラス + 各種無効クラス）
2. 各クラスから代表値を1つ選ぶ
3. 境界の **内側・境界そのもの・境界の外側** をテスト

**例（年齢: 0〜150 が有効）**:

| 入力値 | 分類             | 期待結果 |
|--------|-----------------|---------|
| -1     | 無効（下境界の外）  | invalid |
| 0      | 有効（下境界）     | valid   |
| 1      | 有効（境界+1）     | valid   |
| 75     | 有効(同値クラス代表) | valid |
| 149    | 有効（境界-1）     | valid   |
| 150    | 有効（上境界）     | valid   |
| 151    | 無効（上境界の外）  | invalid |
| "abc"  | 無効（型違い）     | invalid |

**重要ルール**: 同値分割を適用するときは **境界値分析を必ず併用する**。境界の前後はバグが集中する場所。

### 2. デシジョンテーブル

**判定基準**: 結果を決める条件が複数あり、それらの AND/OR の組み合わせでロジックが分岐する場合。単一の if 文ではなく、独立した複数条件が絡むときに効く。

**手順**:
1. 条件を列挙する
2. 全組合せをテーブル化
3. ビジネス上意味のある組合せに絞ってテストケース化（論理的に不可能な組合せは除外）

**例（割引判定: 会員ランク × 購入金額）**:

| 条件 / 結果              | C1   | C2   | C3  | C4  |
|------------------------|------|------|-----|-----|
| プレミアム会員？          | Y    | Y    | N   | N   |
| 購入金額 ≥ 10,000 円？   | Y    | N    | Y   | N   |
| **割引率**              | 20%  | 10%  | 5%  | 0%  |

### 3. 状態遷移テスト

**判定基準**: SUT が状態を持ち、操作によって状態が変化し、かつ **状態によって同じ操作の振る舞いが変わる** 場合。

**手順**:
1. 状態と遷移イベントを書き出す（状態遷移図/表）
2. **正常な遷移** と **無効な遷移（許可されない状態変化）** の両方をテスト
3. 状態に応じた振る舞いの違いも観点に加える

**例（注文の状態）**:

| 初期状態     | イベント       | 期待結果                    |
|------------|--------------|---------------------------|
| 受付         | 支払い完了    | 支払済（正常遷移）            |
| 支払済       | 出荷         | 出荷済（正常遷移）            |
| 受付         | 出荷         | エラー（無効: 支払前は出荷不可）|
| 完了         | キャンセル    | エラー（無効: 完了済は不可逆）  |

**重要ルール**: **無効な遷移を必ず入れる**。正常系だけ書くと状態管理のバグを検出できない。

### 4. Pairwise / 直交表

**判定基準**: 独立したパラメータが3つ以上あり、全組合せ（直積）をテストすると現実的でない数になる場合。「2因子間の全組合せ」までカバーすればバグの大半が検出できる、という経験則を活用する。

**手順**:
1. パラメータと取り得る値を列挙
2. PICT・Allpairs などのツール、または直交表 L_n を使って組合せを縮約
3. 全組合せでなく「2因子間の網羅」を達成する最小ケース集を生成

**例**: ブラウザ（3種）× OS（3種）× 言語（4種）× 通貨（5種）= 全 180 組合せ → Pairwise で約 20 ケースに縮約。

### 5. エラー推測

**判定基準**: 上記の体系的技法で拾えない、経験的に「バグが潜みそう」な箇所を補強したいとき。**常に最後の補完として実施する**。これ単独で網羅性は担保できない。

| カテゴリ    | 推測すべき値                                                       |
|-----------|------------------------------------------------------------------|
| null / 空 | nil, null, undefined, ""（空文字）, []（空配列）, {}（空ハッシュ）        |
| 数値       | 0, -0, -1, Integer の最大/最小, 浮動小数点誤差 (0.1 + 0.2 ≠ 0.3)        |
| 文字列     | 全角文字, 絵文字, 改行コード (CR/LF/CRLF), 制御文字, 極端に長い文字列, BOM |
| 日時       | 閏年 2/29, 月末日, 夏時間切替, タイムゾーンまたぎ, Unix epoch (0/負数)   |
| 並行性     | 同時実行, リトライ, 競合条件, デッドロック                            |
| エンコード | UTF-8/UTF-16, サロゲートペア, 不正なバイト列                           |

# ホワイトボックス網羅基準（コードベース）

実装コードがある場合に、ブラックボックスで出したケースが **コードのどの分岐を通っているか** を確認し、未到達の分岐・条件を通すケースを追加するための基準。

**位置づけ**: ホワイトボックスは **ケースを導くため** に使う。検証対象（アサーション）は振る舞いベースのままにする。「内部でこのメソッドが呼ばれた」ではなく「外から見える出力・状態」を検証する。コード構造に依存したアサーションはリファクタリングで壊れる誤警報源になる。

## 網羅基準の種類と使い分け

| 網羅基準 | 何を網羅するか | 使いどころ |
|---|---|---|
| 命令網羅 (C0) | すべての文を1回以上実行 | 最低限。これだけでは分岐の偽側を見逃す |
| 分岐網羅 (C1) | すべての分岐の真・偽の両方を実行 | **基本目標**。if/case/guard の両側を通す |
| 条件網羅 | 複合条件の各条件式の真・偽を網羅 | `a && b` 等、複数条件が絡む分岐 |
| パス網羅 | 実行経路の組合せを網羅 | 経路が少なく重要な箇所に限定（全網羅は爆発する） |

**適用の優先順**:
1. 分岐網羅（C1）を基本目標にする
2. 複合条件が絡む箇所は条件網羅まで踏み込む
3. パス網羅は、組合せが現実的かつビジネス上意味のある経路に絞る

## 適用例

次のメソッドを考える。

```ruby
def shipping_fee(weight, premium_member)
  return 0 if premium_member
  if weight > 10
    1000
  else
    500
  end
end
```

分岐を洗い出す:
- B1: `premium_member` の真 / 偽
- B2: `weight > 10` の真 / 偽（B1が偽のときのみ到達）

分岐網羅（C1）を満たす最小ケース:

| # | 入力 | 通る分岐 | 期待結果 |
|---|------|---------|---------|
| 1 | premium_member=true, weight=任意 | B1真 | 0 |
| 2 | premium_member=false, weight=11 | B1偽, B2真 | 1000 |
| 3 | premium_member=false, weight=5 | B1偽, B2偽 | 500 |

ブラックボックス（境界値）でケース2,3の weight を 10/11 の境界に寄せると、分岐網羅と境界値分析を同時に満たせる。このようにブラックボックスのケースを土台に、未到達分岐があればホワイトボックスで補う。
