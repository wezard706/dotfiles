# コメント規約レビュー

コードコメントが規約に従っているかをレビューし、違反をフィードバックする。**修正そのものは行わず、何がどう違反しているかと直し方を返す**のが本スキルの責務である（PostToolUse 起点なら、それを受け取った本体エージェントが自分で直す）。

## 基本思想

コメントとは、**設計者の頭の中にあったが、コードでは表現しきれなかった情報を記録するもの**である。

- コードが表現できるのは「実装（どう動くか）」だけ。
- 「抽象（呼び出し側が知るべきこと）」と「意図（なぜそうしたか）」はコードに現れない。
- この失われた情報は obscurity（不明瞭さ）となり、複雑性の主要因になる。

だから「良いコードは自己文書化される」は誤りであり、コードで表現できない情報を残すのがコメントの役割である。逆に、**コードを読めば分かることをなぞるコメントはノイズ**であり、可読性を下げるので削るべき対象になる。

この「足りない Why を補わせる」と「余分な What を削らせる」の両輪がレビューの軸になる。

## レビュー対象の特定

**変更部分のコメントだけを見る。** 無関係な既存コメントを巻き込んで指摘すると、本来のレビュー目的（いま入った変更の品質を守る）がぼやけ、ノイズになる。

- **PostToolUse フック起点**（「いま編集したコメントを見直せ」と促された場合）: 直前に Edit / Write / MultiEdit したファイルの**追加・変更したコメント**が対象。必要なら `git diff` で変更箇所を確認する。
- **手動起動**（ユーザーが「コメントレビューして」等）: 既定では `git diff main...HEAD` の変更部分。ユーザーがファイルや範囲を指定した場合はそれに従う。

変更行に隣接していて、**変更によって陳腐化した既存コメント**は対象に含める（規約4）。それ以外の既存コメントには触れない。

## 規約チェックリスト

変更部分の各コメント（および「コメントが在るべきなのに無い箇所」）を、以下の観点で照合する。

### 規約1. 自明でないことを書く（足りない Why / 余分な What）

コードから読み取れない次の情報が、必要なのに欠けていないか:

| 種類 | 内容 |
|------|------|
| 意図・理由 | なぜこの実装・この値を選んだか |
| 前提条件 | 呼び出し前に満たすべき条件 |
| 副作用 | 状態変更・I/O・外部への影響 |
| 並行性 | スレッド安全性、呼び出し順序への依存 |
| 例外条件 | どの条件でエラー／例外になるか |
| 制約 | 単位・範囲・境界値など、コードに現れない制限 |
| 性能特性 | 計算量など、非自明なパフォーマンス上の注意 |

逆に、**コードを見れば自明なことを書いているコメントは削除を促す。**

```
# 悪い例：見れば分かる（自明なWhat → 削除）
i += 1  # i に 1 を足す

# 良い例：意図を残す（自明でないWhy）
i += 1  # ヘッダ行をスキップするため先頭をずらす
```

### 規約2. マジックナンバー・定数の根拠を残す

「なぜその値なのか」が書かれているか。根拠の分からない定数は、将来の変更で事故を起こす。

```
// リトライ上限。外部APIのレート制限が 3req/sec のため
const MAX_RETRY = 3;
```

### 規約3. ワークアラウンドは出典をリンクする

バグ回避や非直感的な実装に、理由とリンク（Issue・PR・仕様書・外部のバグ報告）があるか。後任が「なぜこんな書き方なのか」を追えるようにするため。

### 規約4. コードとコメントの同期

**古いコメントは、コメントが無いことよりも有害。** 変更によってシグネチャや挙動が変わったのに、隣のコメントが古い記述のまま取り残されていないか。残せないなら削除も正解。

### 規約5. フォーマット

| 項目 | ルール |
|------|--------|
| 形式 | ドキュメンテーションコメントは言語標準の形式に従う（Python: docstring／JS・TS: JSDoc・TSDoc／Java: Javadoc／Go: doc comment／Ruby: RDoc・YARD 等） |
| 言語 | 日本語 |
| 文体 | 体言止め |

## フィードバックの作り方

修正は本体エージェントの仕事。本スキルは**何が問題で、どう直すべきか**を渡す。違反ごとに次を示す:

```
- `path/to/file.rb:42` [規約N] 何が問題か。→ どう直すべきか。
```

直し方は、**正解が一意に決まるかどうか**で書き分ける:

- **削除・整形（正解が一意）**: 具体的な修正後の文言や「この行のコメントは削除」と明示する。例: 自明なWhatコメント、体言止め違反、doc形式違反。
- **Why・根拠の欠落（正解が一意でない）**: 中身を**こちらで書かない**。「なぜこの値/順序/方法を選んだかを記す」と**指示する**にとどめる。

  これは重要で、レビュアが意図を憶測で埋めると、それ自体が「コードと不一致な古いコメント」（規約4違反）の種になる。ただし PostToolUse 起点で**そのコードを書いた本体エージェント自身**が直す場合、本体は意図を知っているので正確に補える。だからこの場合の指示は「お前はなぜそう書いたか知っているはずだ。それをコメントに残せ」という促しになる。

違反が無ければ、その旨を簡潔に伝えて終わる。無理に指摘を捻り出さない。

## 誤検知を出さないために

ノイズの多いレビューは無視されるようになる。次は**指摘しない**:

- コードを読めば分かる程度のことに、過剰に Why コメントを要求する（短く自明なローカル変数・ゲッター等まで説明を求めない）
- 変更していない行・無関係な既存コメント
- リンタ・フォーマッタ・型チェッカが機械的に直す範囲（インデント等のスタイル）
- 意図的な仕様変更そのものへの異議（コメントの良し悪しがスコープであって、実装の是非ではない）

「このコメントが無い／余計なことで、半年後の自分や他人が困るか」を判断基準にする。困らないなら指摘しない。
