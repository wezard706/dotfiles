# DDD 概念モデリング（機能追加）

既存システムに機能を追加する際に、**Agent Teams**（3人体制）で概念モデリングを行い、テーブル設計・クラス設計に直接入力できる成果物を生成する。

---

## 入力と出力

**入力:** ビジネス要求（既存システムの情報 + 追加したい機能）

ビジネス要求が不足している場合は、ユーザーにヒアリングして補完する。以下の情報が揃うまで先に進まない：
- 既存システムの主要な概念・BC構成
- 追加する機能のビジネス上の目的
- 機能が対象とするアクターと業務フロー

**出力:**
1. 概念モデル図（Mermaid `classDiagram`）
2. イベントストーミング結果（Mermaid `flowchart LR`）
3. ユビキタス言語（BC別テーブル）
4. BC間連携マトリクス
5. ポリシー整合性分類テーブル
6. 集約分離の判断根拠テーブル（集約を複数に分けた場合）

---

## Agent Teams

モデリングは3つのペルソナで議論しながら進める。各ステップで3人が意見を出し合い、合意を形成する。

| ペルソナ | 役割 | 発言の特徴 |
|---|---|---|
| **ファシリテーター（F）** | 議論を進行し、合意を取り付ける。ホットスポットを記録する。安易に合意を取らず、各ペルソナの主張の根拠を問い質す | 「次に〇〇を検討しましょう」「DEの意見とLEの意見が異なりますが、それぞれの根拠を聞かせてください」「ここは意見が割れています。ホットスポットとして記録します」 |
| **ドメインエキスパート（DE）** | 業務の現場実態・例外的な運用・暗黙知を提供する。用語の正確性を検証し、ビジネスルールを明確にする。技術用語は使わない | 「現場では〇〇と呼んでいます」「実際の運用では〇〇のケースが月に数回あります」「このルールには例外があり、〇〇の場合は〇〇します」 |
| **リードエンジニア（LE）** | 技術的実現可能性を検証し、集約境界・トランザクション整合性を助言する | 「この2つは同一トランザクションで整合性が必要なので同一集約にすべきです」 |

**議論の進め方:**
- 各ステップで3人が順に意見を述べる（DE → LE → F の順が基本）
- 意見が割れた場合はファシリテーターがホットスポット（⚠️）として記録し、仮の判断を付ける
- **少なくとも1つの設計判断でDEとLEが異なる見解を持ち、トレードオフを明示した上で判断を下す場面を含めること**（形式的な合意だけでは深い設計にならない）
- 全員が合意したら次のステップに進む

---

## ワークフロー

```
1. ビジネス要求の理解     不足時はユーザーにヒアリング
2. 既存システムとの整合性  新機能がどのBCに属するか/新BCが必要かを判断
3. Event Storming        3人チームでDesign Level ESを実施
4. 概念モデル図の作成     classDiagram形式で集約・値オブジェクトを整理
5. ユビキタス言語の定義   BC別に用語を整理
6. 成果物の保存          Markdownファイルとして保存
```

---

## Step 1: ビジネス要求を理解する

ユーザーから提供されたビジネス要求を読み解き、不足があればヒアリングする。

**確認すべき情報:**
- 追加機能のビジネス上の目的（なぜこの機能が必要か）
- 対象アクター（誰がこの機能を使うか）
- 主要な業務フロー（何をどの順序で行うか）
- 既存システムの構成（どのようなBCが存在するか）
- 既存BCとの関係（新機能は既存BCに収まるか、新BCが必要か）

**ヒアリング例:**
- 「この機能を追加するビジネス上の理由は何ですか？」
- 「既存の〇〇（BC名）とどう関係しますか？」
- 「この操作を行うのは誰ですか？」

---

## Step 2: 既存システムとの整合性を確認する

新機能が既存のどのBCに属するか、または新BCが必要かを判断する。

**判断基準:**
- 新機能が既存BCと同じ言語（用語）を使うか → 同一BCの可能性
- 新機能が独自の言語を持つか → 新BCの可能性
- 新機能のライフサイクルが既存概念と異なるか → 新BCの可能性

**記録する:**
- 新BC（必要な場合）の名前と責務
- 既存BCへの影響（変更が必要な箇所）
- 新BC/既存BC間の関係パターン（以下から選択）

**BC間の関係パターン:**

| パターン | 意味 | 使う場面 |
|---|---|---|
| 準拠者（Conformist） | 上流BCのモデルにそのまま従う | 既存BCのモデルを変更できない場合 |
| 顧客-供給者（Customer-Supplier） | 上流が下流のニーズに応じてモデルを調整する | 既存BCと交渉可能な場合 |
| 腐敗防止層（ACL） | 変換層を設けて上流モデルの影響を遮断する | 既存BCのモデルが新機能に合わない場合 |
| 公開ホストサービス/公開言語（OHS/PL） | 汎用的なインターフェースで複数BCに提供する | 新BCを複数BCから利用される場合 |

**BC間連携マトリクス（成果物に含める）:**

```
| 公開元BC | 公開イベント | 購読BC | 連携パターン | 同期/非同期 |
|---|---|---|---|---|
| 休暇管理 | 休暇が承認された | 勤怠記録 | 準拠者 | 非同期 |
```

---

## Step 3: Design Level Event Storming を実施する

3人チームで議論しながら、新機能に関わるBCにDesign Level ESを適用する。

詳細は `references/event-storming-design-level.md` を参照。

**展開順序（この順で進める）:**

1. **ドメインイベント** — 「何が起きたか」を過去形で列挙する
2. **コマンド** — ドメインイベントを引き起こした操作を特定する
3. **集約** — コマンドを受け取り一貫性を保証するオブジェクトを特定する
4. **ポリシー** — あるイベントが別のコマンドを引き起こすルールを特定する
5. **リードモデル** — アクターが意思決定のために参照する情報を特定する
6. **アクター** — コマンドを実行する人・外部システムを特定する

**深掘りのチェック（ハッピーパスで終わっていないか確認する）:**
- 失敗・エラーケース（バリデーションエラー、外部連携失敗）
- 例外フロー（キャンセル、差戻し、期限切れ）
- タイムベースイベント（一定時間後に自動で何かが起きる場合）
- 集約の状態遷移（「このコマンドが拒否される状態はいつか？」）

ホットスポット（⚠️）は放置せず、不明点として明示的にフローに記録する。

**ポリシーの整合性分類（成果物に含める）:**

各ポリシーについて「強整合性（同一トランザクション）」か「結果整合性（非同期）」かを判断し、根拠とともに記録する。

```
| ポリシー | 整合性 | 根拠 |
|---|---|---|
| 承認されたら残日数を消化する | 強整合性 | 承認後に残日数不足が発覚するのは業務上許容不可 |
| 承認されたら勤怠に反映する | 結果整合性 | 勤怠反映の遅延は業務上許容可能 |
```

**集約を複数に分ける場合の判断根拠（成果物に含める）:**

集約を分離する判断を下したら、以下を明記する。

```
| 集約A | 集約B | 分離の理由 |
|---|---|---|
| 休暇申請 | 有給残日数 | ライフサイクルが異なる（申請は短期の承認フロー、残日数は年度単位）。不変条件が独立しており、結果整合性で許容可能 |
```

### Event Storming の出力形式

Mermaid の `flowchart LR` で表現する。

**要素の記法:**

| 要素 | 記法 | スタイル |
|---|---|---|
| アクター | `([アクター名])` | デフォルト |
| コマンド | エッジのラベル `-->\|コマンド名\|` | — |
| ドメインイベント | `[イベント名]` | `fill:#f97316,color:#fff` |
| 集約 | `[集約名]` | `fill:#fef08a,color:#333` |
| ポリシー | `{{説明}}` | `fill:#c4b5fd,color:#333` |
| リードモデル | `[リードモデル名]` | `fill:#86efac,color:#333` |
| 外部システム | `[[システム名]]` | `fill:#fda4af,color:#333` |
| ホットスポット | `(問題点)` | `fill:#ef4444,color:#fff` |

**ルール:**
- BC ごとに1つのフローチャートを作成する
- 同一フロー内の複数のドメインイベントを左から右に時系列で並べる
- ポリシーはイベントの直後に配置し、次のコマンド/イベントへ繋げる
- リードモデルはコマンドの手前に配置し、アクターからの参照を示す

## Step 4: 概念モデル図を作成する

ESの結果から概念を抽出し、Mermaid `classDiagram` で表現する。

詳細は `references/conceptual-model.md` を参照（書き方のルール・テンプレート・チェックリスト）。

**概念の抽出手順:**

1. **集約から主要概念を抽出する** — 各集約が概念モデルの中核概念
2. **コマンド・イベントから隠れた概念を探す** — 名詞が新しい概念の手がかり
3. **集約の内部構造から付随概念を探す** — 値オブジェクト候補
4. **リードモデルから参照概念を探す** — 既存BCの概念を確認

**概念モデル図のルール（重要）:**
- `<<集約ルート>>` `<<エンティティ>>` `<<値オブジェクト>>` `<<列挙型>>` `<<外部システム>>` のステレオタイプを使う
- エンティティ（独自のライフサイクルを持つ）と値オブジェクト（不変・同一性なし）を区別する
- フィールドはビジネス用語のみ（型名禁止: `String`, `Integer`, `uuid` は書かない）
- 操作はビジネス上の行為（メソッドシグネチャ禁止: `cancel!(reason:)` は書かない）
- `note for` で集約の不変条件を明記する
- 関係線: `*--`（所有）、`-->`（参照）、`..>`（依存）、ラベル必須
- 関係線に多重度を記載する（例: `"1" *-- "*"`, `"1" --> "0..1"`）

**ESと概念モデルの整合性チェック（Step 4の末尾で実施）:**
- ESの全集約が概念モデルに含まれているか
- ESのポリシーが概念モデルの集約間の関係に反映されているか
- ESのリードモデルに登場する概念が概念モデルに含まれているか

---

## Step 5: ユビキタス言語を定義する

**確定した BC 別**に用語を整理する。

```
## ユビキタス言語: [コンテキスト名]

| 用語 | 定義 | 注意（他コンテキストとの差異） |
|------|------|-------------------------------|
| 顧客 | 注文を行う登録ユーザー | ※配送コンテキストでは「受取人」と呼ぶ |
```

## Step 6: 成果物を保存する

すべての成果物は **Markdown ファイルとして保存**する。

**保存先:**
- `docs/ddd/[機能名]-modeling.md`（全成果物を1ファイルにまとめる）

保存後、ファイルパスをユーザーに伝える。
