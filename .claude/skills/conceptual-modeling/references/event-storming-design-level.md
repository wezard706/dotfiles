# Design Level Event Storming ガイド

## 概要

Design Level Event Storming（以下 DLES）は、境界づけられたコンテキスト（BC）の内部構造を設計するための手法。
Big Picture Event Storming が「ドメイン全体の流れ」を把握するのに対し、DLES は「特定の BC 内部の設計」を詳細化する。

**適用対象:** コアドメインと複雑な支援ドメインのみ。汎用ドメイン（Auth・決済・通知等）にはスキップ。

---

## 各要素の定義と見つけ方

### ドメインイベント（Domain Event）

**定義:** BC 内で起きた重要な出来事。過去形で表現し、不変の事実を表す。

**見つけ方の問い:**
- 「このBCで起きる重要な出来事は何か？」
- 「〜された」という過去形で表現できるか？
- 「このシステムが外部に知らせる必要がある出来事は何か？」

**良い例:** 「注文が確定された」「在庫が確保された」「支払いが完了した」
**悪い例:** 「注文を確定する」（命令形 → コマンド）、「注文済み」（状態 → ドメインイベントではない）

---

### コマンド（Command）

**定義:** ドメインイベントを引き起こす操作・意図。命令形で表現する。

**見つけ方の問い:**
- 「このドメインイベントを引き起こしたのは何か？」
- 「誰が、いつ、何をきっかけにこの操作を実行するか？」
- 「アクターが何を依頼しているか？」

**良い例:** 「注文を確定する」「在庫を確保する」「支払いを実行する」

---

### 集約（Aggregate）

**定義:** コマンドを受け取り、ビジネスルール（不変条件）を守りながらドメインイベントを発行するオブジェクト群のまとまり。トランザクションの単位。

**見つけ方の問い:**
- 「このコマンドの一貫性を守る責任を持つのは何か？」
- 「このコマンドが失敗する条件（ビジネスルール）は何か？」
- 「同一トランザクションで必ず一緒に変更されるデータは何か？」

**境界の決め方（ヒューリスティクス）:**
- **小さく始める** — 最初は最小の集約を定義し、必要なら後で統合する
- **トランザクション境界で決める** — 「同時に一貫性が必要なデータ」が同一集約に属する
- **結果整合性を許容できるなら別集約** — BC をまたぐ操作は結果整合性（ポリシー経由）で扱う

---

### ポリシー（Policy / Reaction）

**定義:** あるドメインイベントが発生したときに、自動的に別のコマンドを発行するルール。「〜されたら、〜する」の形で表現する。

**見つけ方の問い:**
- 「このイベントが発生した後、自動的に何が起きるか？」
- 「人間が介在せずに自動実行されるルールは何か？」
- 「このイベントに反応して別のBCに通知や依頼をする必要があるか？」

**実装形態の判断基準:**

| 実装形態 | 適用場面 |
|---|---|
| **ドメインサービス** | 同一BC内・複数の集約をまたぐが、ドメインルールを含む場合 |
| **アプリケーションサービス** | BC間をまたぐ、またはインフラ（メール・キュー等）に依存する場合 |
| **プロセスマネージャー** | 長期実行トランザクション・複数ステップにまたがる場合 |

**例:**
- 「注文が確定された → 在庫を確保する」— BC跨ぎのため **アプリケーションサービス**
- 「支払いが承認された → 請求書を発行する → 配送を手配する」— 複数ステップのため **プロセスマネージャー**

---

### リードモデル（Read Model）

**定義:** アクターがコマンドを実行する前の意思決定のために参照するデータのビュー。

**見つけ方の問い:**
- 「このコマンドを実行する前にアクターは何を見るか？」
- 「アクターが判断を下すために必要な情報は何か？」

**実装形態候補:** クエリオブジェクト、DB View、CQRS のクエリ側、GraphQL クエリ等

---

### アクター（Actor）

**定義:** コマンドを実行する人・外部システム・タイマー等。

**見つけ方の問い:**
- 「誰が（何が）このコマンドを実行するか？」
- 「ユーザーか、外部システムか、タイマーか？」

---

## 展開手順のまとめ

```
Step 1: ドメインイベントを列挙する（「何が起きたか」から始める）
         ↓
Step 2: 各イベントに対してコマンドを追加する（「何がそれを引き起こしたか」）
         ↓
Step 3: コマンドを受け取る集約を特定する（トランザクション境界で決める）
         ↓
Step 4: イベント → 自動実行されるコマンドのポリシーを追加する
         ↓
Step 5: コマンド実行前にアクターが参照するリードモデルを特定する
         ↓
Step 6: 各コマンドにアクターを紐付ける
```

---

## 集約の境界を見直すサイン

DLES 中に以下が起きたら、集約の境界を見直す:

- 1つのコマンドで複数の集約を同時に変更しなければ不変条件を守れない → 同一集約に統合を検討
- 1つの集約が巨大になり、多くの異なるコマンドに応答している → 集約の分割を検討
- 「〜されたら〜する」のポリシーが多すぎる → BC の境界が適切か再検討

---

## BC境界を見直すサイン

DLES 中に以下が起きたら、BC の境界（Step 5）で見直す:

- 同一 BC 内で同じ用語が異なる意味で使われている → 分割を検討
- ポリシーが BC を頻繁にまたぐ → 境界の引き直しを検討
- 2つの BC が常にセットで変更される → 統合を検討

---

## 状態遷移の制約

Event Storming の各要素の接続には決まったルールがある。違反すると設計の意図が伝わらなくなる。

### 有効な遷移パターン

**基本連鎖（Internal State Transition）:**
```
[アクター] → コマンド → [集約] → ドメインイベント
```

**ポリシー経由の自動実行（Bridging via Policy）:**
```
ドメインイベント → [ポリシー] → コマンド → [集約] → ドメインイベント
```

**アクターの意思決定（Bridging via Read Model）:**
```
[リードモデル] ←参照← [アクター] → コマンド
（リードモデルは別のドメインイベントによって更新される）
```

**外部システムトリガー:**
```
[外部システム] → コマンド        （外部がコマンドを発行）
[外部システム] → ドメインイベント  （外部が独自にイベントを発行）
```

**外部システム連携パターン:**

| パターン | フロー | 例 |
|---|---|---|
| 同期API呼び出し | 集約 → 外部システム → 結果を受けてイベント発行 | 決済実行 → Stripe API → 決済完了イベント |
| Webhookコールバック | 外部システム → コマンド → 集約 → イベント | Stripe Webhook → 決済結果を記録する → 決済集約 → 決済完了/失敗イベント |
| ポーリング | タイマー → コマンド → 集約 → イベント | 定期チェック → 外部状態を同期する → 集約 → 状態変更イベント |

外部システムへのコマンド発行は、ポリシーから直接外部システムに繋げるのではなく、**ポリシー → コマンド → 集約 → 外部システム**の順で記述する。集約が外部呼び出しの結果を受けてイベントを発行する責務を持つ。

### 禁止される遷移（よくある誤り）

| NG パターン | 正しいパターン | 理由 |
|---|---|---|
| コマンド → ドメインイベント（直接） | コマンド → 集約 → ドメインイベント | 集約が不変条件を検証してからイベントを発行する |
| ドメインイベント → コマンド（直接） | ドメインイベント → ポリシー → コマンド | 自動実行ルールはポリシーとして明示する |
| ポリシー → ドメインイベント（直接） | ポリシー → コマンド → 集約 → ドメインイベント | ポリシーはコマンドを発行するだけ。集約がイベントを発行する |
| 集約 → コマンド | — | 集約はイベントのみを発行する |
| 集約 → ポリシー | — | 集約はイベントを発行し、ポリシーがイベントに反応する |

---

## 深掘りの視点

ハッピーパス（正常系）だけで終わると、設計が浅くなる。以下の視点で意識的に掘り下げること。

### 1. ハッピーパス以外のシナリオを探す

- **失敗・エラーケース**: 「在庫が不足していた場合は？」「支払いが失敗した場合は？」
- **例外フロー**: キャンセル、返品、差戻し、却下
- **タイムアウト・期限切れ**: 「24時間以内に支払いが完了しなかった場合は？」「承認が一定期間なければ？」

### 2. 集約の状態を問う

コマンドが拒否される条件を問うことで、集約の不変条件（ビジネスルール）が明確になる。

- 「このコマンドが実行できない状態はいつか？」（例: キャンセル済みの注文は再確定できない）
- 「集約が取りうる状態のリストは何か？」（状態機械として考える）
- 「この状態遷移を許可するビジネスルールは何か？」

### 3. アクターの役割を深く問う

- 「このコマンドを実行するのは人間か、外部システムか、タイマーか？」
- 「複数のアクターが同じコマンドを実行できる場合、権限・条件の違いは何か？」
- 「このアクターはどの情報を見てこのコマンドを実行するか？」（リードモデルの手がかり）

### 4. ホットスポットを表面化する

不明・不確か・意見が割れる箇所を放置せず、ホットスポット（⚠️）として記録する。

- ビジネスルールが曖昧な箇所（「どのタイミングで確定とみなすか？」）
- チーム間で意見が割れる箇所
- 技術的に実現が難しいと感じる箇所
- 後回しにしたい複雑な例外ケース

### 5. 時間軸と一貫性の視点

- 「このポリシーはリアルタイムが必要か、非同期（結果整合性）で許容できるか？」
- 「SLA・期限などのビジネス上の時間制約はあるか？」
- 「同時に一貫性が必要なデータはどれか？」（集約の境界の確認）
- 「結果整合性で許容できる箇所はどこか？」（ポリシー経由でよい箇所）

### 6. 外部との連携を問う

- 「このイベントを外部システムに通知する必要があるか？」
- 「外部システムから受け取るイベントはあるか？どのタイミングで来るか？」
- 「外部連携が失敗した場合のフォールバックは？」
