# テストワークフロー（test-workflow）

test-design と test-implementation の2スキルを順次実行し、テスト対象（SUT）に対して「テストケース設計 → RSpec実装 → 検証」を一気通貫で行う。各スキルはサブエージェントで実行し、最後にワークフロー全体の成果を検証する。

---

## 良い単体テストとは

このワークフローのゴールは **良い単体テストを完成させること**。設計・実装・検証の各ステップで、成果物が次の性質に反していないか常に問い直す。設計だけ・実装だけでは判断しきれない性質（壊れにくさ・速さ等）も、ワークフロー全体を通して担保する。

- **バグを見つけられる** — そもそもの目的。バグを検出できないテストは存在価値がない
- **読めば仕様が分かる** — テストコードが仕様の説明になっている
- **壊れにくい** — 実装をリファクタしても、振る舞いが変わらなければ失敗しない（誤警報を出さない）
- **失敗した原因が分かる** — 落ちたとき、どの振る舞いが壊れたか一目で分かる
- **速い・独立している** — 他テストの実行順や状態に依存せず、単体で速く回せる

各サブスキルへの指示（Step 2・Step 3）でも、この性質を満たすよう促す。

---

## 入力と出力

**入力:** テスト対象（SUT）。test-design と同じく、仕様書・実装コードのいずれか、または両方。不足する場合はユーザーにヒアリングする。

**出力:**
1. テスト対象に対応する spec ファイル（test-design が骨格を作り、test-implementation が本体を実装した、実行可能な RSpec）
2. 検証結果の要約（全 `it` 実装済みか・テストがグリーンか・発見したバグ）

spec ファイルのパスは Rails/RSpec の規約に従う（例: `app/models/account.rb` → `spec/models/account_spec.rb`）。プロジェクト直下のスクラッチ領域（`tmp/` 等）には出力せず、必ず `spec/` 配下の実パスに出力する。

---

## ワークフロー概要

```
Step 1: 入力の確認・ヒアリング
Step 2: test-design をサブエージェントで実行 → RSpec アウトライン
Step 3: test-implementation をサブエージェントで実行 → spec を実装
Step 4: レビュー（良いテストの原則に沿った品質レビュー、サブエージェント）
Step 5: 検証（静的: 実装の完全性 / 動的: テスト実行）
  → 要修正あり & iteration < 3 → Step 6 へ
  → 問題なし or iteration = 3 → Step 7 へ
Step 6: 修正イテレーション → Step 4 へ戻る
Step 7: ユーザーへ最終報告
```

---

## Step 1: 入力の確認

テスト対象（クラス・メソッド）と、その仕様書・実装コードの所在を確認する。test-design の入力要件（仕様書 > 実装コード > ヒアリング の優先順）を満たすことを確認する。

この段階で **出力先の spec ファイルパス**を決める。テスト対象のソースパスを Rails/RSpec の規約で `spec/` 配下にミラーする（例: `app/models/account.rb` → `spec/models/account_spec.rb`）。既存 spec があればそのパスを使う。ここではパスを決めるだけで、ファイルはまだ作らない（Step 2 の test-design が設計成功時に作成する。中断した場合は作られない）。

仕様が曖昧な場合は **推測せず確認する**（test-design の「原則1: 仕様を疑う」と同じ）。

---

## Step 2: テスト設計（test-design）

サブエージェントに以下を指示する。

```
あなたはテスト設計の専門家です。以下のタスクを実行してください。

1. スキル定義を読む: .claude/skills/test-design/SKILL.md を読む
2. テスト対象: [SUTのクラス・メソッドと、仕様書/実装コードの所在を貼る]
3. スキルの手順（Step 1〜Step 2）に従ってテストケースを設計する
4. 成果物（describe/context/it の RSpec アウトライン）を spec ファイル [Step 1 で決めた spec パス] に保存する。本体未実装の it 行で構成する（pending 扱い）。既存 spec があれば追記・統合する

重要: ブラックボックスとホワイトボックスの両観点を使うこと（コードがある場合）。
テスト可用性が著しく低いと判断したら、無理に設計せず中断して理由を報告すること。
```

**完了確認:** spec ファイルに describe/context/it のアウトラインが生成され、各 `it` に技法/網羅のコメントが付いていること。

test-design が「テスト可用性が低い」として中断した場合は、ここでワークフローを止めてユーザーに報告する（実装に進まない）。この中断パスでは spec ファイルを作らない（作りかけのアウトラインがあれば残さない）。報告には中断理由と SUT 見直しの提案を含める。

---

## Step 3: テスト実装（test-implementation）

サブエージェントに以下を指示する。

```
あなたはRSpecテスト実装の専門家です。以下のタスクを実行してください。

1. スキル定義を読む: .claude/skills/test-implementation/SKILL.md と reference.md をすべて読む
2. テスト設計書（RSpecアウトライン）を読む: [Step 2 で出力した spec パス]
3. テスト対象コードを読む: [SUTの実装コードの所在]
4. スキルの手順に従い、その spec ファイルを in-place で編集し、各 it の本体を実装する

重要:
- アウトラインの全 it を実装すること（pending を残さない）
- アサーションは振る舞いベース（外から見える出力・状態）に保つこと
- モックは最小限に留めること
- 新しい spec ファイルを作らず、Step 2 のアウトラインと同じファイルを埋めること
```

**完了確認:** アウトラインの全 `it` が実装され、`pending`/`skip` が残っていないこと。

---

## Step 4: レビュー（良いテストの原則）

実装された spec を、冒頭「良い単体テストとは」の5性質に照らして **質的にレビューする**。Step 5 の検証（実装漏れ・テストの合否）が機械的なチェックなのに対し、ここはテストコードの質を人の目線で問う。**レビューはサブエージェント**に行わせ、客観性を確保する。

サブエージェントに以下を指示する。

```
あなたはテストコードのレビュアーです。以下の spec を「良い単体テスト」の観点でレビューしてください。

読むファイル:
- [実装された spec のパス]
- [テスト対象の実装コード]

5つの性質ごとに、違反している箇所を具体的に指摘してください。

1. バグを見つけられるか — アサーションが弱すぎないか（期待値が緩い、実質何も検証していない、ハッピーパスしか見ていない）
2. 読めば仕様が分かるか — it の文と中身が一致しているか、テスト名から仕様が読めるか、意図不明なマジックナンバー/セットアップがないか
3. 壊れにくいか — 内部実装に依存していないか（private メソッド・呼び出し回数・内部状態のアサーション、過剰なモック）。検証は振る舞いベースか
4. 失敗した原因が分かるか — 1つの it に複数観点を詰め込んでいないか（1観点1テスト）、失敗時にどの振る舞いが壊れたか分かるか
5. 速い・独立しているか — テスト間で状態を共有していないか、実行順に依存していないか、不要な DB/外部アクセスがないか

出力形式:
## 要修正（原則違反）
- [性質番号] 該当箇所と、なぜ問題か、どう直すか

## 改善提案
- [性質番号] あれば

要修正がない場合は「要修正: なし」と明記すること。
```

**要修正** があれば Step 6 の修正イテレーションで対応する。**改善提案** は収束をブロックせず、Step 7 でユーザーに提示する。

---

## Step 5: 検証

実装された spec を2軸で検証する。

### 静的検証: 実装の完全性

Step 2 で設計したアウトライン（spec の `it` 一覧）に対し、実装後の spec を確認する。実装は同じファイルを in-place で埋めるため、Step 2 時点の `it` 一覧（サブエージェントの報告や git diff で把握できる）を基準にする。

- アウトラインの全 `context`/`it` が残っているか（設計したケースが消えていないか）
- 各 `it` に本体が実装され、`pending`/`skip`/空ボディが残っていないか
- 設計になかった余計なテストが紛れ込んでいないか

### 動的検証: テストの実行

可能なら実際にテストを実行する。プロジェクトのテストランナーを使う（例: `bundle exec rspec <spec ファイルのパス>`。Bundler を使わない構成なら `rspec <spec ファイルのパス>`）。

- **全グリーン** → 問題なし。Step 4 のレビュー結果とあわせて、要修正がなければ Step 7 へ
- **失敗あり** → 原因を切り分ける（重要）:
  - **テスト側のバグ**（前提・入力・期待値の誤り）→ Step 6 で修正する
  - **プロダクトコード側のバグ**（テストが正しく、実装が仕様に反している）→ **プロダクトコードを勝手に直さない**。これはテストが目的どおりバグを発見した状態。Step 7 でユーザーに報告する

テスト実行環境が無い（依存・DB等が用意できない）場合は、静的検証のみとし、その旨を報告する。

---

## Step 6: 修正イテレーション

Step 4 のレビュー（要修正）と Step 5 の検証で見つかった **テスト側の不足・誤り・原則違反** を修正する。

- アウトラインに対する実装漏れ・`pending` 残り → test-implementation を再実行（同じ spec ファイルの不足分を指示に明記）
- 期待値・前提の誤り → 該当 `it` を修正
- 原則違反（内部実装依存・1テスト多観点・順序依存等）→ レビュー指摘に従って該当 `it` を修正

修正後、Step 4（レビュー）に戻る。**上限3イテレーション**。上限到達時は残課題をユーザーに提示する。

プロダクトコードのバグが疑われるケースはこのループで直さない。Step 7 で報告する。

---

## Step 7: ユーザーへ最終報告

以下を簡潔に報告する。

- 生成物: spec ファイルのパス（テスト対象に対応するパス。test-design が骨格を作り、test-implementation が本体を実装）
- 検証結果: 全 `it` 実装済みか / テスト実行結果（グリーン・赤・未実行）
- レビュー結果: 良いテストの原則に照らした残りの改善提案（あれば）
- **発見したバグ**: プロダクトコード側のバグが疑われる場合は、どの `it` が失敗し、仕様とのどの差異かを明示する
- 残課題: 上限到達で未解決の項目、テスト未実行の理由など

中間成果物（サブエージェントのログ等）はユーザーに提示しない。

---

## やってはいけないこと

- 単体の設計だけ・実装だけの依頼でこのワークフローを起動しない: それぞれ test-design / test-implementation を直接使う。
- テストを通すためにプロダクトコードを勝手に書き換えない: テストが見つけたバグは報告する。テストの目的はバグを見つけること。
- test-design が中断したのに実装へ進まない: テスト可用性の問題はユーザーに委ねる。
