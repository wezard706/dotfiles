# コミットメッセージ規約

git commitを作成する際に適用します。

## 共通のルール
- **重要**: コミットメッセージは日本語で書くこと

## フォーマット

- Conventional Commits形式
- type: subject の形式に従うこと

### Typeの定義

- feat: 新機能の追加
- fix: バグ修正
- refactor:	リファクタリング
- docs: ドキュメントのみの変更
- style: コードの意味に影響を与えない変更（改行、インデント、セミコロン忘れなど）
- test: テストコードの追加・修正
- chore: ビルドプロセス、補助ツール、ライブラリの更新など
- perf: パフォーマンス向上のための変更

## サブジェクト

- 最大50文字
- 体言止めにすること
- 曖昧な言葉を避けること
- 末尾に句点は付けないこと
- コロンの後に半角スペースを入れること
- 何をどうしたのかを書くこと

### 機能追加・変更（feat）の例
「何ができるようになったか」を書く
feat: [ドメイン名]に[新しい振る舞い]を追加
feat: [画面名]における[特定の操作]をサポート

例: feat: 注文モデルにキャンセル予約ステータスを追加

### バグ修正（fix）の例
「何がどう直ったか」を書く
fix: [発生箇所]で[発生現象]が起きる問題を解消
fix: [条件]において[期待しない動作]になるバグを修正

例: fix: 未ログイン時にカート遷移でクラッシュする問題を修正

### リファクタリング（refactor）の例
「構造をどう変えたか」を書く
refactor: [モジュール名]のロジックを[デザインパターン名]へ移行
refactor: [クラス名]から[責務]を分離

例: refactor: 決済処理をStrategyパターンで再構築

## ボディ

変更の意図を詳しく説明する場合に記述する

- サブジェクトの後に空行を入れること
- 各文は改行して始めること
- 72文字で折り返すこと

### 記述内容のガイドライン
「何をしたか（What/How）」はコードを見ればわかるため、**「なぜその変更が必要だったか（Why）」**を重点的に記述してください。

1. 変更の理由・背景: なぜこの変更を行う必要があったのか
2. 判断の根拠: 複数の選択肢がある中で、なぜこの実装を選んだのか
3. 副作用・注意点: この変更によって影響を受ける箇所や、あえて対応しなかったこと

```
feat: ユーザー認証に二要素認証を導入

セキュリティ強化のため、従来のパスワード認証に加えてOTP認証を必須化。
当初はSMS認証も検討したが、コストと到達率の観点からGoogle Authenticator
によるアプリ認証を先行して実装する。

ログインフローのAPI仕様が変わるため、フロントエンド側での追従が必要。
```
