# ユーザーストーリーを用いた要件定義 Knowledge

このドキュメントは、ユーザーストーリーを用いた要件定義を行う際に参照する知識ベースです。
思想・原則・実務的な書き方のすべてを含みます。

---

## 1. 思想と前提

### 1.1 ユーザーストーリーとは何か

ユーザーストーリーは、Kent Beck が XP (Extreme Programming) の中で提唱した概念です。
重要なのは、ユーザーストーリーは **「要件仕様書」ではなく「会話のきっかけ」** だということ。

Ron Jeffries が定式化した「3つのC」が本質を表しています。

- **Card** (カード): 要点を短く書いたもの。ポストイット1枚程度
- **Conversation** (会話): カードを起点にチームで詳細を会話する
- **Confirmation** (確認): 受け入れ基準で完成を確認する

ストーリーは詳細を網羅した文書ではなく、関係者の対話を促進するための触媒です。
Jeff Patton の言葉を借りれば「**ドキュメントは旅行の写真のようなもの**」で、
思い出を呼び起こすためのものであり、旅行そのものを再現するものではありません。

### 1.2 アンチパターン: テンプレートゾンビ

ユーザーストーリーの3要素テンプレート（ロール・機能・利益形式など）に
機械的に埋めるだけの作業に陥ることを、Jeff Patton は「テンプレートゾンビ」と呼んで
警鐘を鳴らしています。

形式を満たすことが目的ではなく、**チームが共通理解を持つこと** が目的です。
テンプレートを使う場合も、「ロール」「機能」「理由」のそれぞれが
意味のある情報を持っているかを常に確認する。
特に「理由」が「機能」の言い換えになっていないか注意する。

### 1.3 ストーリーマッピングの考え方

バックログを縦一列に並べると、全体像(big picture)が見えなくなります。
ストーリーマッピングは、ユーザーの行動の流れに沿って横軸に並べ、
詳細度に応じて縦に展開する2次元の地図です。

```
[ユーザーの行動の流れ →]

大ストーリー(エピック): 商品を買う
  ├── 中ストーリー: 商品を探す
  │     ├── 小ストーリー: キーワード検索
  │     ├── 小ストーリー: カテゴリ絞り込み
  │     └── 小ストーリー: 価格でソート
  ├── 中ストーリー: 商品を選ぶ
  │     ├── 小ストーリー: 商品詳細を見る
  │     └── 小ストーリー: レビューを読む
  └── 中ストーリー: 購入する
        ├── 小ストーリー: カートに追加
        ├── 小ストーリー: 配送先入力
        └── 小ストーリー: 決済
```

横軸は「**ナラティブフロー**」(ユーザーがプロダクトを使う物語の流れ)、
縦軸は詳細度の階層です。これにより、機能の抜け漏れや、リリースの切り出し方が
直感的に把握できます。

## 2. ストーリーの基本フォーマット

### 2.1 標準テンプレート (ロール・機能・利益形式 / Role-Feature-Reason)

ユーザーストーリーは **ロール・機能・利益形式（Role-Feature-Reason）** で記述する。
Mike Cohn が『User Stories Applied』で紹介した、Connextra のシンプルな別表記。

```
[ロール] として、
[機能] が欲しい。
なぜなら [理由] だから。
```

例:
```
登録ユーザーとして、
パスワードをリセットする機能が欲しい。
なぜなら ログインできなくなっても自力で復旧したいから。
```

3つの要素はすべて埋める。

- **ロール**: 誰のためのストーリーか。「ユーザー」のような曖昧な表現ではなく、
  「登録ユーザー」「管理者」「ゲストユーザー」のように具体化する。
- **機能**: 何ができるようになるか。実装手段ではなく、ユーザーが達成する
  行動を書く。
- **理由**: なぜそれが必要か。「ログインできるようにしたい」のような
  機能の言い換えではなく、ユーザーが得る価値（利益）を書く。

### 2.2 ストーリーに含める要素

最低限:
- タイトル
- ロール / 機能 / 理由（ロール・機能・利益形式）
- 受け入れ基準

任意で追加:
- 背景・補足
- メモ(未確定事項、技術的な留意点)
- 関連ストーリー、依存関係
- 見積もり(ストーリーポイント)
- 優先度(MoSCoW)

## 3. INVEST原則

良いストーリーが満たすべき6つの性質。Bill Wake が提唱しました。

### I - Independent (独立している)
他のストーリーに依存せず、単独でリリース可能であること。
依存があると優先順位が固定化され、計画が硬直します。
完全に独立できない場合は、依存を最小化する。

### N - Negotiable (交渉可能である)
ストーリーは契約書ではなく、対話の出発点。
詳細は実装前に交渉して詰める余地があるべき。
完全に固められた仕様書になっていたら、それはストーリーではない。

### V - Valuable (価値がある)
ユーザーまたはビジネスに価値を提供すること。
「DBにインデックスを追加する」は技術タスクであり、ストーリーではない。
ストーリーは「ユーザーが○○できる」という形で価値を表現する。

### E - Estimable (見積もり可能である)
チームが工数を見積もれる程度に明確であること。
見積もれないなら、情報不足か粒度が大きすぎる。

### S - Small (小さい)
1スプリント(通常1〜2週間)で完了するサイズであること。
大きすぎるものはエピックなので、分割する。

### T - Testable (テスト可能である)
完成したかどうかを客観的に判定できること。
受け入れ基準が書けない = テスト不可能 = ストーリーとして未完成。

## 4. 受け入れ基準の書き方

### 4.1 基本原則

受け入れ基準は **チェックリスト形式** で記述する。
1項目1動作で、テスターが○×を判定できる粒度で書く。

悪い例:
- ユーザーが使いやすい
- 高速に動作する
- 適切にエラーハンドリングされている

良い例:
- [ ] 検索結果が3秒以内に表示される
- [ ] 不正なメールアドレス形式の場合、入力欄の下に赤字でエラーメッセージが表示される
- [ ] 連続5回ログインに失敗すると、アカウントが10分間ロックされる

### 4.2 書き方のルール

- 主語と目的語を省略しない
- 数値・時間・回数など測定可能な要素を含める
- 実装方法ではなく、観察可能な動作を記述する
- 1項目に複数の判定を入れない(「○○して、かつ△△する」は2項目に分ける)
- チェックボックス(`- [ ]`)で書き、完了時にチェックする運用にする

### 4.3 シナリオ網羅

正常系だけでなく、異常系・境界値も漏らさず書く。
書く際は「正常系」「異常系」「境界値」のラベルを明示すると抜け漏れを防げる。

例: パスワードリセット

```markdown
### 受け入れ基準

正常系:
- [ ] 登録済みメールアドレスを入力すると、リセットリンクを含むメールが送信される
- [ ] リセットリンクから新パスワードを設定できる
- [ ] 新パスワードでログインできる

異常系:
- [ ] 未登録のメールアドレスを入力すると「該当アドレスは見つかりません」と表示される
- [ ] 不正な形式のメールアドレスは入力検証エラーになり、送信処理は実行されない
- [ ] 1分以内に同じアドレスへ再送信を試みた場合、レート制限のメッセージが表示される

境界値:
- [ ] 発行から24時間ちょうど経過したリンクは有効である
- [ ] 発行から24時間1秒経過したリンクは「有効期限切れ」と表示される
- [ ] 新パスワードが7文字以下の場合は受け付けない(8文字以上が必須)
```

### 4.4 受け入れ基準のチェックリスト(メタチェック)

書いた受け入れ基準自体が要件を満たしているかの確認項目:

- [ ] チェックボックス形式(`- [ ]`)で書かれているか
- [ ] 主語と目的語が省略されていないか
- [ ] 数値・時間・回数など測定可能な要素が含まれているか
- [ ] 正常系・異常系・境界値をカバーしているか
- [ ] 1項目に1動作になっているか(複合判定になっていないか)
- [ ] 実装方法ではなく、観察可能な動作を記述しているか
- [ ] テスターが○×判定できる粒度か

## 5. ストーリー分割のテクニック

大きなストーリー(エピック)を、INVEST を満たすサイズに分割する方法。
以下8つの分割パターンを **エピックの性質に応じて使い分ける** 。
判断順序は次の通り。

1. ユーザーが複数のステップを踏む作業か → **5.1 ワークフロー分割**
2. 適用ルールが複数あるか → **5.2 ビジネスルール分割**
3. 同じ目的に複数の手段があるか → **5.3 操作バリエーション分割**
4. 扱うデータの種類が複数あるか → **5.4 データタイプ分割**
5. 入出力チャネルが複数あるか → **5.5 入出力チャネル分割**
6. 機能の複雑度に段階があるか → **5.6 シンプル/複雑分割**
7. データの操作種別が揃っているか → **5.7 CRUD分割**
8. 上記いずれにも当てはまらない場合 → **5.8 ハッピーパス先行分割**

複数のパターンが当てはまる場合は、上から順に検討して最初に当てはまるものを採用する。

### 5.1 ワークフロー分割
ユーザーの一連の作業を、ステップごとに分ける。

例: 「会員登録できる」
→ メールアドレス入力 / 確認メール送信 / 認証 / プロフィール入力 / 完了

### 5.2 ビジネスルール分割
適用するルールごとに分ける。

例: 「割引が適用される」
→ 会員割引 / クーポン割引 / まとめ買い割引 / セール期間割引

### 5.3 操作バリエーション分割
同じ目的の異なる手段ごとに分ける。

例: 「ログインできる」
→ メール+パスワード / Googleログイン / Appleログイン / 二要素認証

### 5.4 データタイプ分割
扱うデータの種類ごとに分ける。

例: 「ファイルをアップロードできる」
→ 画像 / PDF / 動画 / その他

### 5.5 入出力チャネル分割
インターフェースごとに分ける。

例: 「通知を受け取れる」
→ メール通知 / プッシュ通知 / SMS通知 / アプリ内通知

### 5.6 シンプル/複雑分割
まずシンプル版を作り、後から複雑版を追加する。

例: 「商品を検索できる」
→ Step1: キーワード単純検索のみ
→ Step2: 絞り込みフィルタ追加
→ Step3: ファセット検索追加

### 5.7 CRUD分割
作成・読み取り・更新・削除を分ける。

例: 「コメント機能」
→ コメント投稿 / コメント一覧表示 / コメント編集 / コメント削除

### 5.8 ハッピーパス先行分割
最も基本的な成功パスを先に作る。エラー処理は後回し。

例: 「決済できる」
→ Step1: 正常な決済フロー(エラーなし)
→ Step2: カード情報不正のエラーハンドリング
→ Step3: ネットワークエラー時のリトライ
→ Step4: タイムアウト処理

## 6. 優先順位付け (MoSCoW法)

ユーザーストーリーの優先順位付けには **MoSCoW法** を採用する。
すべてが「必須」だと優先順位がないのと同じ。明示的に4つのカテゴリに分類する。

### 6.1 4つのカテゴリ

- **Must have** (必須):
  これがないとリリースできない。提供価値の中核を成す機能。
  欠けるとプロダクトとして成立しない。
- **Should have** (重要):
  重要だが、なくてもリリースは可能。可能な限り入れたいもの。
  代替手段やワークアラウンドがあれば後回しにできる。
- **Could have** (あれば嬉しい):
  時間や余力があれば入れたい機能。なくても大きな影響はない。
  Must/Should が先に進んだ後に着手する。
- **Won't have** (今回は対象外):
  今回のリリースでは明示的に対象外とする機能。
  「やらないこと」を宣言することで、スコープクリープを防ぐ。

### 6.2 運用の指針

**Must have の比率に注意する**

MoSCoW法を機能させるコツは、**Must have を全体の60%以下に抑える** こと
(目安として工数ベース)。「全部 Must です」となった時点で優先順位付けは
機能していない。

**Won't have を必ず明示する**

「やらないこと」を書かないリストは MoSCoW ではない。
Won't have を書くことで以下の効果がある:
- ステークホルダー間の期待値調整ができる
- 後から「あれもやってほしい」と要求が膨らむのを防げる
- 次回以降のリリースに繰り越す候補が明確になる

**カテゴリの再評価**

スプリントやリリースごとに見直す。状況が変われば、
Could だったものが Must に昇格することも、その逆もある。
固定的なものではなく、生きたリストとして扱う。

### 6.3 判断のための問いかけ

ストーリーをどのカテゴリに分類するか迷ったときの問いかけ:

- このストーリーがなくても、ユーザーは目的を達成できるか?
  - できない → Must have
  - できるが大きく不便 → Should have
  - 不便だが許容できる → Could have
- 競合や代替手段と比較して、これがないと選ばれないか?
- 法規制や契約上、必須となっている要件か?
- 今回のリリースで本当に必要か、次回でも間に合うか?

### 6.4 出力フォーマット例

```markdown
## リリース優先順位

### Must have
- USR-001: ユーザー登録ができる
- USR-002: ログイン/ログアウトができる
- USR-005: 商品を検索できる
- USR-010: 商品を購入できる

### Should have
- USR-003: パスワードリセットができる
- USR-007: 商品をお気に入り登録できる

### Could have
- USR-008: お気に入りをカテゴリ分けできる
- USR-012: 購入履歴をCSVでダウンロードできる

### Won't have (今回は対象外)
- USR-020: 商品レビュー投稿機能 → 次回リリース候補
- USR-021: ソーシャルログイン → 検討中
- USR-022: 多言語対応 → 来期検討
```

## 7. 実務的なフォーマット例

### 7.1 機能追加のストーリー

```markdown
## ストーリー: 商品をお気に入り登録できる

**ロール**: ログイン済みユーザー
**機能**: 気になる商品をお気に入りに追加できる
**理由**: 後で見返して購入を検討できる

### 受け入れ基準

正常系:
- [ ] 商品詳細ページにハートアイコンが表示される
- [ ] ログイン済みユーザーがハートアイコンをタップすると、お気に入りに追加される
- [ ] お気に入り追加済みの商品はアイコンが塗りつぶし表示になる
- [ ] 塗りつぶし表示のハートアイコンを再度タップするとお気に入りから解除される
- [ ] 解除後はアイコンが枠線のみの表示に戻る

異常系:
- [ ] 未ログイン状態でハートアイコンをタップするとログイン画面に遷移する
- [ ] ログイン後は元の商品詳細ページに戻る

### メモ
- 一覧表示は別ストーリー(USR-103)で対応
- お気に入り上限数は要相談(PMに確認中)

### 見積もり
3ポイント

### 優先度
Should have
```

### 7.2 非機能要件のストーリー

```markdown
## ストーリー: 商品検索が高速に応答する

**ロール**: 商品を探しているユーザー
**機能**: 検索結果が素早く表示される
**理由**: ストレスなく買い物を続けられる

### 受け入れ基準

性能:
- [ ] 通常負荷時、検索リクエストの95パーセンタイルが500ms以内に応答する
- [ ] 通常負荷時、24時間にわたるタイムアウト発生率が0.1%以下である

精度:
- [ ] 改善前と同じ検索キーワード集合で、検索結果の適合率(precision)が改善前と同等以上である

### 技術メモ
- 現状は平均1.8秒、95%tileで4.2秒
- 詳細な実装方針は設計フェーズで決定

### 優先度
Must have
```

### 7.3 バグ修正のストーリー

```markdown
## ストーリー: ログイン後にカートが空になる問題を修正

**ロール**: 買い物中のユーザー
**機能**: ログインしてもカートの中身が保持される
**理由**: 買い物を続けられる

### 再現手順
1. 未ログイン状態で商品をカートに入れる
2. ログインボタンを押す
3. ログイン完了後、カートを確認

### 期待動作
入れた商品が残っている

### 実際の動作
カートが空になる

### 受け入れ基準

正常系:
- [ ] 未ログイン時にカートに入れた商品が、ログイン後も保持される
- [ ] ログイン前のカートとログインユーザーの既存カートに別々の商品があった場合、両方がマージされる
- [ ] マージ時に同一商品が両方にあった場合、数量が合算される

### 優先度
Must have
```

## 8. 対話を通じて要件を引き出すときの質問例

要件定義の対話で使える質問のストック。
段階に応じて使い分ける。

### 8.1 コンテキスト把握フェーズ

- このシステム/機能で解決したい課題は何ですか?
- 今、ユーザーは何に困っていますか?
- もしこの機能ができなかったら、何が起こりますか?
- このプロジェクトの成功を、どうやって測りますか?
- 半年後に「やってよかった」と言える状態は、どんな状態ですか?

### 8.2 ユーザー把握フェーズ

- 主なユーザーはどんな人ですか? (年齢、職業、技術レベル)
- ユーザーは1日のうち、どんなタイミングでこの機能を使いますか?
- ユーザーが今、似た目的で使っているツールはありますか?
- 使う頻度はどれくらいですか? (毎日 / 週次 / 年に数回)

### 8.3 機能スコープ把握フェーズ

- 今回のリリースに必ず入れたい機能は何ですか?
- 逆に、今回は対象外にしたい機能はありますか?
- 既存の機能で、これに関連するものはありますか?
- 連携が必要な外部システムはありますか?

### 8.4 制約把握フェーズ

- いつまでにリリースが必要ですか?
- 予算や人数の制約はありますか?
- 法規制やコンプライアンス上の要件はありますか?
- 必須の技術スタックはありますか?

### 8.5 受け入れ基準を引き出すフェーズ

- どうなったら「完成」と判断できますか?
- 想定外の入力があった場合はどうしたいですか?
- データが0件のときの表示は?
- 通信エラーが起きたときは?
- パフォーマンスはどれくらい必要ですか? (具体的な数値で)

## 9. 最後に: 要件定義の進め方の心得

1. **完璧を目指さない**
   要件は対話と実装を通じて精度が上がる。最初から完璧な要件は存在しない。

2. **書きながら見せる**
   書き上げてからレビューではなく、書いている過程を見せる。

3. **未確定を未確定として残す**
   決まっていないことを「決まったことにする」と後で必ず破綻する。
