# ユーザーストーリーを用いた要件定義

ユーザーと対話しながら要件を引き出し、**ユーザーストーリー形式のバックログ**として
`./tmp/user-stories/backlog.md` に書き出すスキル。

ストーリーは「契約書」ではなく「会話のきっかけ」（Ron Jeffries の3つのC）。
テンプレートを機械的に埋める「テンプレートゾンビ」（Jeff Patton）に陥らないよう、
対話を通じてユーザー・チームと共通理解を作ることを最優先する。

詳細な思想・原則・書き方はすべて `references/knowledge.md` に集約してある。
判断に迷ったとき、フォーマット例を確認したいとき、分割パターンを選びたいとき、
受け入れ基準のメタチェックをしたいときなどは、該当セクションを必ず参照すること。

---

## 入力と出力

**入力**
ユーザーの要望。粒度は問わない。例えば：
- 「ECサイトに商品レビュー機能を追加したい」
- 「管理画面の通知機能を作りたい」
- 「ログイン後にカートが消える不具合を直したい」
- 「お気に入り機能を作りたいけど何から考えればいい？」

**出力**
`./tmp/user-stories/backlog.md` に以下を含む Markdown ドキュメントを生成する：
- 背景・目的（簡潔なコンテキスト）
- ユーザーストーリー一覧（ロール・機能・利益形式 + 受け入れ基準）
- リリース優先順位（MoSCoW法、Won't have を含む）
- 未確定事項（TBD/要相談リスト）

ファイルは1本にまとめる方針。複数のテーマを扱う場合も、見出しで区切って同一ファイルに記述する。
既に `backlog.md` が存在する場合は、上書きせず追記するか別名にするかをユーザーに確認する。

## このスキルの守備範囲

含む：
- ストーリー作成（ロール・機能・利益形式 + 受け入れ基準）
- エピックの分割支援（`knowledge.md` セクション5の8パターン）
- MoSCoW優先順位付け
- バグ修正・非機能要件のストーリー（`knowledge.md` セクション7.2, 7.3）
- **サブエージェントによる自動レビューループ**（INVEST原則・受け入れ基準メタチェック・MoSCoW妥当性・形式チェック）。詳細は後述「ストーリーレビュー」セクション参照

含まない（必要なら別スキルや別ツールに委譲）：
- ストーリーマッピング（横軸ナラティブフロー、縦軸詳細度）の図化
- 概念モデリング・テーブル設計（`conceptual-modeling` / `table-design` を案内）
- 「真の課題は何か」レベルの要件発見の深掘り（後述「他スキルとの連携」参照）

## 対話の進め方

### 基本姿勢: 柔軟な対話

`knowledge.md` セクション8 には5つのフェーズ（コンテキスト把握 → ユーザー把握 → 機能スコープ → 制約 → 受け入れ基準）が定義されているが、これらは**スキル内部の指針**として使う。
ユーザーの発話・状況に応じて順序や深さを動的に変えてよい。

具体的には：
- ユーザーが既に詳しい要件を提示してきたら、コンテキスト把握は最低限にしてストーリー化に進む
- ふわっとした入口の場合は、コンテキスト〜ユーザー把握に時間をかける
- 受け入れ基準を書くフェーズに入ったら、正常系・異常系・境界値を意識的に網羅する

質問は **一度に1〜3個まで** に絞る。質問攻めにしない。

### 節目で「決まったこと / 未確定」をサマリする

柔軟な対話はフェーズの抜け漏れが起きやすい。これをカバーするため、
対話の節目（次のフェーズに移るとき、または5〜10往復ごと）で必ず以下を提示する：

```markdown
ここまでで決まったこと：
- 〜
- 〜

未確定 / 要相談：
- 〜
- 〜
```

この提示でユーザーが「あ、これ忘れてた」と気づける機会を作る。
ユーザーから「これでOK」「次へ」と確認が取れたら次のフェーズに進む。

### ストーリー作成中もリアルタイムに見せる

書き上げてからレビューではなく、書きながら見せる（`knowledge.md` セクション9-2）。
1ストーリーずつ「こんな感じで書こうとしていますが、どうですか？」と提示し、
ユーザーの反応を見て修正してから次に進む。

### 未確定を未確定のまま残す

決まっていないことを「決まったことにする」と後で破綻する（`knowledge.md` セクション9-3）。
`TBD` や `要相談` を積極的に使い、後で誰に確認すべきかも書く。

例：
```markdown
### メモ
- お気に入り上限数: TBD（PMに確認）
- 通知のタイミング: 要相談（UXチームと議論）
```

### 対話なしで呼ばれた場合（非対話モード）

このスキルが subagent から呼ばれた / cron や batch から呼ばれた / ユーザーが一発の入力だけで silent になった、といった**リアルタイム対話できないケース**では、対話前提のステップを degrade する。

| 通常モード | 非対話モード |
|---|---|
| 「節目で決まったこと/未確定をサマリ」を提示 | サマリ提示は省略。代わりに最終提示で全 TBD/要相談を一覧化する |
| 「1ストーリーずつ見せる」 | 一気に書く。ただし各ストーリーに「メモ」を設けて discretionary な前提を明示する |
| 不明点を質問で引き出す | 不明点は **すべて TBD / 要相談 として残す**。業務常識レベルで明らかなものは「推測値 + 要相談」を併記して進める |
| ファイル衝突時にユーザーに諮る | 既存ファイルを**上書きせず**、別名（例: `backlog-2.md` の連番）で保存してユーザー判断を待つ |

非対話モードかどうかの判定は「直近の発話履歴に対話の余地があるか」で行う。判定が曖昧なときは**非対話モード扱い**（より安全側）。

非対話モードでも、レビュー（後述「ストーリーレビュー」）は実行する。レビューもサブエージェント dispatch が利用できない場合はインライン self-review にフォールバックする（後述）。

## ストーリーの書き方

### ロール・機能・利益形式 (Role-Feature-Reason)

すべての機能ストーリーは以下の形式で書く（Mike Cohn『User Stories Applied』。
Connextra のシンプルな別表記）：

```
[ロール] として、
[機能] が欲しい。
なぜなら [理由] だから。
```

ファイル内では3要素を箇条書きで記載する：

```markdown
## ストーリー: [タイトル]

**ロール**: [具体的な役割。「ユーザー」のような曖昧な表現は避ける]
**機能**: [ユーザーが達成する行動。実装手段は書かない]
**理由**: [ユーザー/ビジネスが得る価値。機能の言い換えは避ける]
```

書く際、以下が起きていないかチェックする（テンプレートゾンビ警報）：
- 「ロール」が「ユーザー」「人」のように抽象的すぎる
- 「理由」が「機能」の言い換えになっている（例: 機能「ログインしたい」/ 理由「ログインできるようにしたい」）
- 3要素のうちどれかを埋めるのに苦労している（→ そのストーリー自体の存在意義を疑う）

これらが起きたら、ユーザーに「ロールをもう少し具体化できますか？」「この機能で本当に得たい価値は何ですか？」と問い返す。

### 受け入れ基準

チェックリスト形式（`- [ ]`）で書く。書き方の詳細ルールは `knowledge.md` セクション4 を参照。

**必ず守ること**：
- **正常系・異常系・境界値の3カテゴリでラベル分けして書く**
  「正常系のみ」で済ませない。異常系・境界値を意識的に問いかける。
- **数値・時間・回数など測定可能な要素を入れる**
  「速い」「使いやすい」のような主観表現は禁止。
- **1項目1動作**
  「○○して、かつ△△する」は2項目に分ける。
- **実装方法ではなく、観察可能な動作**
  「Reactのstate管理で〜」ではなく「画面に〜が表示される」と書く。

異常系・境界値を引き出すための質問例（`knowledge.md` セクション8.5）：
- 想定外の入力があったらどうしますか？
- データが0件のときの表示は？
- 通信エラーが起きたときは？
- 上限・下限はどこですか？ ちょうど境界値のときの動作は？

### 受け入れ基準のメタチェック

書き上げたら、`knowledge.md` セクション4.4 のチェックリストで自己レビューする。
特に「テスターが○×判定できる粒度か」を最重要項目として確認する。

## エピックの分割

ストーリーが大きすぎる（1スプリントで終わらないサイズ、INVEST の S を満たさない）と判断したら分割する。

### 分割パターンの選び方

`knowledge.md` セクション5 の8パターンを **判断順序に従って上から検討** し、
最初に当てはまるものを採用する：

1. ユーザーが複数のステップを踏む作業か → ワークフロー分割
2. 適用ルールが複数あるか → ビジネスルール分割
3. 同じ目的に複数の手段があるか → 操作バリエーション分割
4. 扱うデータの種類が複数あるか → データタイプ分割
5. 入出力チャネルが複数あるか → 入出力チャネル分割
6. 機能の複雑度に段階があるか → シンプル/複雑分割
7. データの操作種別が揃っているか → CRUD分割
8. 上記いずれにも当てはまらない → ハッピーパス先行分割

判断順序を守る理由：上位パターンの方がユーザー視点での分割になりやすく、各分割後のストーリーが「単独で価値を提供する」状態になりやすい。CRUD分割やハッピーパス分割は便利だが、機械的に使うとリリース価値の薄いストーリーが量産される。

### 分割時にユーザーに見せること

分割案を出したら、必ず以下を提示する：
- 採用した分割パターンと、その理由（なぜこのパターンを選んだか）
- 分割後の各ストーリーが「単独で価値を提供できるか」のチェック
- リリース順序の提案（最初に何を出すと最も価値があるか）

ユーザーが別パターンを希望したり、もっと粗い/細かい粒度を求めたら柔軟に組み直す。

## MoSCoW優先順位付け

ストーリーが揃ったら、4カテゴリ（Must/Should/Could/Won't）に分類する。詳細は `knowledge.md` セクション6 を参照。

### 運用上の注意

- **Must have を全体の60%以下に抑える**ことを目標にする。
  「全部 Must です」となったら、ユーザーに「もし1つだけ削るとしたら？」と問いかけて優先順位を浮き彫りにする。
- **Won't have を必ず明示する**。
  「やらないこと」を書かないリストは MoSCoW ではない。
  対話中に出たが今回スコープ外と判断したアイデアは Won't have に書き、次回繰り越し候補とする。
- 判断に迷うときは `knowledge.md` セクション6.3 の問いかけを使う：
  - このストーリーがなくても、ユーザーは目的を達成できるか？
  - 法規制や契約上、必須となっている要件か？

## 非機能要件・バグ修正のストーリー

機能ストーリーと同じくロール・機能・利益形式で書くが、受け入れ基準のラベル構造が異なる。

### 受け入れ基準のラベル規約（ストーリー種別ごとに必須形式が異なる）

| ストーリー種別 | 必須ラベル | 必要数 |
|---|---|---|
| 機能 | 正常系 / 異常系 / 境界値 | 各 ≥1 項目 |
| バグ修正 | 正常系 / 異常系 / 境界値 | 各 ≥1 項目（境界値にマージ・並行操作・他機能への影響を含める） |
| 非機能要件 | 領域別（性能 / 精度 / 可用性 等） | カテゴリ数1〜複数、各 ≥1 項目 |

**バグ修正は機能ストーリーと同じ「正常系・異常系・境界値」の3カテゴリに揃える**。「成功率」「ログ」のような領域別ラベルで書くと、後述の blocking 定量基準（「3カテゴリそれぞれ ≥1 項目」）で blocking 判定される。

**非機能要件のみ**領域別ラベル（性能・精度等）でよい。`knowledge.md` セクション7.2 の例を参照。

### 非機能要件
- 受け入れ基準のラベルは「性能」「精度」「可用性」など領域名にする
- 必ず数値で書く（「速く」「安定して」は禁止）
- フォーマット例: `knowledge.md` セクション7.2

### バグ修正
- ストーリー本体に加え、「再現手順 / 期待動作 / 実際の動作」を記載する
- 受け入れ基準のラベルは **正常系 / 異常系 / 境界値** の3カテゴリで書く
- 境界値カテゴリには、マージ・並行操作・他機能への影響など、関連する境界ケースを入れる
- フォーマット例: `knowledge.md` セクション7.3

## 出力ファイルの構造

`./tmp/user-stories/backlog.md` の標準構造：

```markdown
# [プロジェクト名 / 機能名] バックログ

## 背景・目的
[2〜5行程度。なぜこの機能/改修が必要か]

## ユーザー像（必要に応じて）
[主要ユーザーの簡潔な説明]

## ユーザーストーリー

### USR-001: [タイトル]
**ロール**: ...
**機能**: ...
**理由**: ...

#### 受け入れ基準
正常系:
- [ ] ...
異常系:
- [ ] ...
境界値:
- [ ] ...

#### メモ
- ...

#### 優先度
Must have

---

### USR-002: ...

## リリース優先順位 (MoSCoW)

### Must have
- USR-001: ...

### Should have
- USR-XXX: ...

### Could have
- USR-XXX: ...

### Won't have (今回は対象外)
- USR-XXX: ... → 次回リリース候補

## 未確定事項 / 要相談
- [ ] お気に入り上限数（PMに確認）
- [ ] 通知タイミング（UXチームと議論）
```

### ID 採番ルール
- `USR-001` から始める3桁連番
- 削除しても番号は再利用しない（トレーサビリティ確保）
- **Won't have に置くストーリーも本編と同じ連番（USR-XXX）を使う**。900番台などに分けない。やらないことも将来のリリース候補として連番管理する

### Won't have ストーリーの記述粒度

Won't have セクションのストーリーは、本編 (Must/Should/Could) よりも**簡略な記述**でよい。

**必須**:
- タイトル
- ロール / 機能 / 理由（次回検討時の出発点として残す）
- 一行メモ（「次回繰り越し」「対象外理由」など）

**省略可**:
- 受け入れ基準（次回 Must/Should に昇格させるときに詳細化する。プレースホルダ表記でも可）
- 見積もり

理由: Won't have の主目的は「やらないこと」の宣言。本編と同じ粒度で書こうとするとテンプレートゾンビ化し、暫定値の量産になる。**blocking 定量基準（受け入れ基準のカテゴリ網羅、ストーリーサイズ等）は Won't have ストーリーには適用しない**。

### ファイル運用
- 既存の `backlog.md` がある場合：
  - 同一プロジェクトの追加なら追記
  - 別プロジェクトなら `backlog-[機能名].md` のような別名を提案して確認
  - **非対話モード時はユーザーに諮れないので、上書きせず別名（`backlog-2.md` などの連番）で保存**する。最終提示で「既存ファイルを保護したため別名で保存しました」と一言伝える。
- ファイル先頭に最終更新日を入れる

## ストーリーレビュー（サブエージェント自動レビュー）

バックログ初稿（または大きな更新）が出来たら、`Agent` ツールでサブエージェントを起動して品質レビューを実行する。
**blocking 指摘の件数が 0 になるまでイテレーションを繰り返す。**

### 発火タイミング
- バックログを最初に書き出した直後（必須）
- 大きな構造変更後（ストーリー追加・分割・MoSCoW再分類など。必須）
- ユーザーから「レビューして」と明示された時（必須）
- 1ストーリーだけの軽微な表現修正の場合は、ユーザーに「再レビュー走らせる？」と確認してから判断（省略可）

### 指摘の分類

| 分類 | 定義 | 収束への影響 |
|---|---|---|
| **blocking** | 修正しないとストーリーとして成立しない（具体基準は下表） | 収束をブロックする |
| **nit** | 改善余地はあるがストーリーとして使える（表現の磨き / 補足情報の追加 / より良い分割パターン候補 / 受け入れ基準のもう一段細分化提案 / 命名改善 など） | 収束をブロックしない。最終提示時にユーザーに諮る |

#### blocking の定量基準（曖昧判定を防ぐためすべて定量化）

| 観点 | 推奨 (nit止まり) | blocking |
|---|---|---|
| 形式の3要素（ロール/機能/理由） | 全項目に意味のある内容 | いずれか1つでも空 or 「ユーザー」のような曖昧表現のみ |
| 受け入れ基準のカテゴリ網羅（機能・バグ修正） | 正常系・異常系・境界値**それぞれ ≥1 項目** | いずれか1カテゴリが0項目 |
| 受け入れ基準のカテゴリ網羅（非機能要件） | 領域別ラベル（性能/精度/可用性等）で各 ≥1 項目 | 領域別ラベルなし or どの領域も 0 項目 |

**Won't have ストーリーは、上記の「受け入れ基準のカテゴリ網羅」「ストーリーサイズ (INVEST: S)」の blocking 判定対象外**。タイトル・3要素・一行メモがあれば成立とみなす（前述「Won't have ストーリーの記述粒度」参照）。
| 受け入れ基準の形式 | 全項目チェックボックス + 観察可能動作 | チェックボックスなし or 全項目が主観表現（"使いやすい"等） |
| MoSCoW Must比率 | 60%以下 | **80%超** |
| Won't have | 1件以上明示 | 0件 |
| ストーリーサイズ (INVEST: S) | 1スプリント（1〜2週間）相当 | 明らかに1スプリントで終わらない巨大さ |
| ストーリーの価値 (INVEST: V) | ユーザー価値を表現 | 純粋な技術タスク（"DBにインデックス追加"等）でユーザー価値の言及なし |

「nit止まり」と「blocking」の境界に該当しないものは nit に倒す（過剰検出を避ける）。

**収束条件**: blocking 件数 = 0

**上限**: 3イテレーション。上限到達時は残存 blocking をユーザーに提示して判断を仰ぐ（自動修正は止める）。

### サブエージェント指示テンプレート

#### 起動方法（環境に応じて選ぶ）

サブエージェントの起動は、**独立コンテキストでエージェントを発火できる仕組み**であれば何でもよい。具体例：
- Claude Code の `Agent` ツール（`subagent_type: general-purpose`）
- `Task` 系ツール（環境によって名前が異なる）
- それに相当する仕組み

#### dispatch 可否の判定手順（必ずこの順で実行する）

1. **top-level 確認**: `Agent` / `Task` 系ツールがそのまま呼べる状態か確認する
2. **deferred 確認**: top-level で見つからなくても、`ToolSearch` 等の deferred tool 機構が利用可能なら、そこで `Agent` / `TaskCreate` 等を検索して schema を取得し、**起動を1回試みる**
3. 起動が成功した場合 → サブエージェントレビューを実行
4. **試行が失敗した場合のみ**（schema 取得不能、呼び出しエラー等）→ インライン self-review にフォールバック
5. fallback した場合は、最終提示で「サブエージェントは利用不可だったため self-review で実施した」とユーザーに一言伝える

**NG**: dispatch 試行をせずに「機構が見つからない」と即判断して self-review に倒すこと。**判定前に必ず1回は実起動を試みる**こと。「ツール一覧に見えない」だけでは fallback 理由にならない。

インライン self-review に倒した場合でも、収束条件・出力形式は同じものを使う。すなわち、自分自身が一旦バックログ作成者の立場を離れ、ゼロから読み直す姿勢でレビュアーを演じる。

バックログのファイルパスと、参照すべき knowledge.md のパスは**絶対パス**で渡す（サブエージェントは別コンテキストなので相対パスでは到達できない）。

#### プロンプト本体

```
あなたはユーザーストーリー要件定義のレビュアーです。以下のタスクを実行してください。

1. 知識ベースを読む:
   ~/.claude/skills/user-story-requirements/references/knowledge.md
   特に以下のセクションを重点参照:
   - セクション2: ロール・機能・利益形式の3要素
   - セクション3: INVEST原則
   - セクション4.4: 受け入れ基準のメタチェック
   - セクション5: エピック分割の判断順序（8パターン）
   - セクション6: MoSCoW（Must比率の目安、Won't have明示）

2. 業務コンテキスト（原ユーザー要望のサマリ）:
   [ここに2〜5行で原ユーザー要望のサマリを挿入する。
    例:
    - 機能名: 商品お気に入り機能
    - 対象ユーザー: 登録済み会員のみ
    - スコープ: MVP最小限（登録/解除/一覧）
    - 明示的なスコープ外: SNSシェア、フォルダ分け
    - 制約: 上限件数は要相談（TBD）]
   レビューにはこの要望への「忠実度」も含めること（要望に明示された機能が漏れていないか、
   要望に「スコープ外」と明示された項目が Won't have に入っているか、
   要望にない要素を勝手に Must have に入れていないか）。

3. レビュー対象を読む: [絶対パス: 例 /Users/shun/.../tmp/user-stories/backlog.md]

4. 以下の6観点でレビューする。各観点で blocking と nit を区別する基準は、
   呼び出し元スキルの「blocking の定量基準」表を参照する。
   (1) 業務要望への忠実度（NEW）
       - 原ユーザー要望に明示された機能が漏れていないか
       - 原ユーザー要望に「スコープ外」と明示された項目が Won't have に入っているか
       - 原ユーザー要望にない要素を勝手に Must have に入れていないか（推論で追加した
         ストーリーは Should have 以下に置くか、メモで明示すべき）
   (2) 形式チェック（ロール・機能・利益形式）
       - 全ストーリーの **ロール / 機能 / 理由** 3要素が埋まっているか
       - ロールが「ユーザー」のように曖昧でないか（「登録ユーザー」「管理者」のように具体的か）
       - 理由が機能の言い換えになっていないか（テンプレートゾンビ警報）
   (3) INVEST原則（特に V / S / T を重視）
       - V: 技術タスクではなくユーザー価値を表現しているか
       - S: 1スプリント（1〜2週間）で終わるサイズか。エピックのままになっていないか
       - T: 受け入れ基準が客観的に判定可能か
   (4) 受け入れ基準のメタチェック（knowledge.md セクション4.4 のチェックリスト全項目）
       - チェックボックス形式（`- [ ]`）か
       - 主語と目的語が省略されていないか
       - 数値・時間・回数など測定可能な要素が含まれているか
       - **ストーリー種別ごとのラベル必須形式（0項目は blocking）**:
         - 機能ストーリー: **正常系・異常系・境界値の3カテゴリが、それぞれ ≥1 項目**
         - バグ修正ストーリー: **機能と同じ3カテゴリ（正常系・異常系・境界値、各 ≥1 項目）**。「成功率」「ログ」のような領域別ラベルで書かれている場合は blocking
         - 非機能要件ストーリー: 領域別ラベル（性能・精度・可用性等）で各 ≥1 項目
         - **Won't have ストーリーはこの判定の対象外**（タイトル + 3要素 + 一行メモがあればOK、受け入れ基準は省略・プレースホルダ可）
       - 1項目1動作になっているか
       - 実装方法ではなく観察可能な動作を記述しているか
       - テスターが○×判定できる粒度か
   (5) MoSCoW分類の妥当性
       - Must比率: 60%以下が推奨。**80%超は blocking**。60〜80% は nit。
       - Won't have が**1件以上**明示されているか（0件は blocking）
   (6) エピック分割の妥当性
       - 大きすぎるストーリー（S違反）が残っていないか
       - 分割している場合、knowledge.md セクション5の判断順序通りに上位パターンから
         検討されているか
       - 各分割が単独で価値を提供しているか（CRUD・ハッピーパス分割を機械的に使って
         価値の薄いストーリーを量産していないか）

5. 出力形式:
   ## blocking
   - [USR-XXX] 観点名 — 具体的な問題と該当箇所の引用 — 修正提案
   - ...

   ## nit
   - [USR-XXX] 観点名 — 改善提案の内容
   - ...

   blocking がない場合は「## blocking\nなし」と明記すること。
   レビュー対象が空 or 不在の場合はその旨を報告すること。

注意:
- 推測ではなく、対象ファイルの具体的な記述を引用して指摘すること
- 同一の問題を複数のストーリーで見つけた場合、各ストーリーごとに1件ずつ報告する（まとめない）
- レビューに徹し、ファイルを編集しないこと（修正は呼び出し元が行う）
- 推奨ライン（nit）と blocking ラインを混同しない。「Must比率 75%」は nit であって blocking ではない。
```

### イテレーションのロジック

1. **Iteration N 開始**: ユーザーに「Iteration N: レビューを開始します」と一行で伝える
2. **サブエージェント起動**: 上記「起動方法（環境に応じて選ぶ）」に従って起動。dispatch機構が無ければインライン self-review
3. **結果受領**: blocking と nit の件数を確認
4. **収束判定**:
   - blocking = 0 → **収束**（最終提示へ）
   - blocking > 0 かつ N < 3 → 修正フェーズへ
   - blocking > 0 かつ N = 3 → **上限到達**（残存blockingをユーザーに提示して判断を仰ぐ）
5. **修正フェーズ**: blocking 指摘を1件ずつ `backlog.md` を直接編集して修正
   - 修正方針が複数ありえる場合（分割するか縮小するか等）は、ユーザーに簡潔に確認してから編集する
   - 修正完了後、ユーザーに「Iteration N 完了: blocking N件を修正」と一行で報告
6. **次イテレーションへ**: Step 1 に戻る

### 最終提示

収束（または上限到達）後：
1. 最終バックログのパス（`./tmp/user-stories/backlog.md`）を提示
2. nit 指摘を「改善候補」として一覧で提示し、対応するか1問で確認
3. 上限到達で残存 blocking がある場合は、その内容をリストで提示してユーザー判断を仰ぐ

### 注意事項

- **サブエージェントには現在の対話コンテキストが見えない**。ファイルパス・参照知識ベースのパスは絶対パスで渡し、レビュー観点もすべてプロンプトに含める。
- レビュー結果はユーザーには提示しない（中間成果物）。提示するのは「件数」と最終的な「nit改善候補」「残存blocking」のみ。
- 修正の根拠が指摘に対して明らかでない場合（例: 同じblockingを複数回繰り返してしまう場合）、無理に自動修正せずユーザーに方針を確認する。

## 他スキルとの連携

このスキルは **要件のドキュメント化が主役**。発見系の深掘りは他スキルが得意。
以下のサインを察知したら、対応スキルの起動をユーザーに **提案** する（勝手に起動はしない）。

| ユーザーの状況 | 提案するスキル |
|---|---|
| 「そもそも何を解決すべきかが曖昧」「要求の前提が不明」 | `requirements-analysis`（要件問題の診断） |
| 「アイデア発散から始めたい」「どんな機能があり得るか出したい」 | `brainstorming`（superpowers） |
| 「ドメインモデルを整理したい」「集約・エンティティを決めたい」 | `conceptual-modeling` / `ddd-modeling` |
| 「業務フローをイベント単位で洗い出したい」 | `event-storming` |
| 「ストーリーが固まったので、次はテーブル/クラス設計したい」 | `table-design` / `class-design` |

提案するときの言い回し例：
> 「ここから先は『真の課題は何か』レベルの深掘りが必要そうです。`requirements-analysis` スキルの方が向いていますが、起動しますか？ それともこのまま要件をストーリー化に進めますか？」

ユーザーがこのスキルを続けると判断したら、深掘りはせずに既知の情報でストーリー化を進める。
