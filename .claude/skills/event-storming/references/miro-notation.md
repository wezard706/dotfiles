# Miro記法（Event Storming）

Miro MCP の `mcp__claude_ai_Miro__diagram_create` で **DSL テキスト**を渡して Event Storming 図を出力する。

---

## 基本原則（v2: クラスタ主導）

1. **BCごとに1 diagram** — 1つの diagram には1つのBCのフローだけを入れる。BC間遷移はイベントを両 diagram に重複配置する
2. **クラスタ = 1ループ** — クラスタは必ずドメインイベントで終わる。内部は左→右に水平配置（下位ループまたは上位ループのパターンに従う）
3. **クラスタ内部に矢印は引かない** — 近接配置と並び順で意味を伝える
4. **クラスタ間の矢印は「イベント → ビュー」または「イベント → ポリシー」のみ** — イベント同士を直接つながない（これは旧ルールで廃止）
5. **凡例は専用 diagram として1枚だけ作る** — BC diagram には含めない
6. **ホットスポットは Miro に描かない** — 会話内報告のみで扱う

---

## DSL 取得フロー（必須）

`diagram_create` は DSL テキストを受け取る。DSL の正式な文法は Miro 側で管理されているため、**必ず `diagram_get_dsl` を先に呼ぶ**。この呼び出しで返ってきた仕様に従って DSL を組み立てる。

```
# Step A: DSL仕様を取得（セッション中1回で良い）
mcp__claude_ai_Miro__diagram_get_dsl(
  miro_url: "<ボードURL>",
  diagram_type: "flowchart"
)

# Step B: 取得した仕様に従ってDSLテキストを構築し、diagram_createに渡す
mcp__claude_ai_Miro__diagram_create(
  miro_url: "<ボードURL>",
  diagram_type: "flowchart",
  title: "ES: <BC名>",
  x: 0,
  y: 0,
  diagram_dsl: "<構築したDSLテキスト>"
)
```

セッション内で同じ diagram_type を繰り返し作る場合、`diagram_get_dsl` は1回で良い（仕様を記憶して流用する）。

## 色マッピング（7要素）

Zenn記事（bitkey_dev `0220513ab0569d`）の慣例に準拠。ホットスポットは廃止したため7要素。

| 要素 | 色（fillColor） | 備考 |
|---|---|---|
| ドメインイベント | `#f8d3af` オレンジ | クラスタの末尾 |
| コマンド | `#c6dcff` 青 | 集約の直前 |
| 集約 | `#fff6b6` 黄 | コマンドとイベントの間 |
| ポリシー | `#dedaff` 紫 | 上位ループの先頭 |
| リードモデル | `#adf0c7` 緑 | 下位ループのビューと同義（詳細は下記） |
| ビュー | `#ccf4ff` 水色 | UI画面系を区別したい場合 |
| アクター | `#e7e7e7` グレー | 下位ループのビューの直後 |

**リードモデルとビュー:** 概念的には同じ（アクターがコマンド発行前に参照する情報）。慣例として、UI画面として具現化されたものを「ビュー（水色）」、内部的なデータビュー／クエリ結果を「リードモデル（緑）」として色分けする。どちらか一方だけ使っても良い。

## クラスタ構文（2ループ構造）

### 下位ループクラスタ（人が駆動する）

内部構成（左→右、矢印なし）:
```
[ビュー] [アクター] [コマンド] [集約] [イベント]
```

DSLでは1つのサブグラフ／グループとして表現する（`diagram_get_dsl` で取得した構文に従う）。

### 上位ループクラスタ（ポリシーが駆動する）

内部構成（左→右、矢印なし）:
```
[ポリシー] [コマンド] [集約] [イベント]
```

### クラスタ間の矢印

イベントから次クラスタの「起点要素」にのみ矢印を引く：

| 接続 | 意味 |
|---|---|
| イベント → ビュー | 次の下位ループを人が開始する |
| イベント → ポリシー | 次の上位ループを自動で起動する |

**禁止:** イベント同士の直接接続／クラスタ内部の矢印／集約→イベント の明示矢印（近接配置で表す）

## 命名規則（ノードID）

ID は後続の edge で参照されるため、一貫した規則で付ける。

| 要素 | プレフィックス | 例 |
|---|---|---|
| アクター | `actor_` | `actor_applicant` |
| コマンド | `cmd_` | `cmd_apply_leave` |
| 集約 | `agg_` | `agg_leave_request` |
| ドメインイベント | `ev_` | `ev_leave_applied` |
| ポリシー | `pol_` | `pol_consume_balance` |
| リードモデル | `rm_` | `rm_leave_balance` |
| ビュー | `view_` | `view_apply_form` |
| 凡例 | `legend_` | `legend_event` |

クラスタ自体にも ID を振る（例: `cluster_apply`, `cluster_approve`）。

## 凡例 diagram（1枚）

全BC diagramに共通の凡例として、別 diagram を1枚作成する。配置はボードの左端または上部。

**凡例 diagram の内容:**

1. **色見本セクション** — 7要素の付箋を縦または横に並べ、それぞれに要素名を書く
2. **下位ループのサンプルクラスタ** — `[ビュー][アクター][コマンド][集約][イベント]` を水平配置
3. **上位ループのサンプルクラスタ** — `[ポリシー][コマンド][集約][イベント]` を水平配置

サンプルクラスタは読み手が「この形が1ループなんだ」と即座に理解できる教材として機能させる。

---

## BC間遷移の表現

送信側BC・受信側BCは別 diagram。矢印は diagram を跨げないため、**同じイベントを両 diagram に重複して配置する**。

**送信側BC diagram:**
- 発生元クラスタの末尾にイベントを置く（普通通り）
- テキストに `（→ <受信側BC名>へ）` と追記するか、会話内報告で対応関係を明示

**受信側BC diagram:**
- 同じイベントを先頭（独立した要素）に置く
- テキストに `（← <送信側BC名>から）` と追記
- そのイベントから上位ループクラスタ（`[ポリシー][コマンド][集約][イベント]`）へ矢印を引く

---

## ボード配置と既存要素との共存

- **書き出す前に** `mcp__claude_ai_Miro__context_get` または `board_list_items` で既存要素を確認する
- `x`, `y` 座標で各 BC diagram を分離する（例: BC1 は y=0、BC2 は y=800、凡例は y=-600 など）
- 同じボードに複数回書き出す場合、ユーザーに **追記 / 新規作成 / 差し替え** のどれかを必ず確認する
- 座標の微調整APIはないため、間隔を十分に取る（最低 600〜800 px の余白）

## 旧ルールとの違い（v1 → v2）

| 観点 | v1（旧） | v2（現行） |
|---|---|---|
| 矢印 | ドメインイベント同士 | クラスタ間のみ（イベント→ビュー／イベント→ポリシー） |
| クラスタ単位 | フェーズ | 1ループ（下位／上位） |
| BC の表現 | 同一 diagram 内でクラスタタイトルに併記 | BCごとに diagram を分離 |
| ホットスポット | Miro に ⚠ として描画 | Miro には描かず、会話内のみ |
| 凡例 | 各 diagram の左端 | 専用 diagram を1枚 |
| DSL 構文 | pseudo nodes/edges 配列 | `diagram_get_dsl` で取得した DSL テキスト |

v1 からの移行時は、既存の diagram は手動で削除／移動してから新規作成するのが確実。
