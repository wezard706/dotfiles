# Event Storming (Design Level)

ビジネス要求を入力として、**Agent Teams**（3人体制）でDesign Level Event Stormingを実施し、**Miroボード上に可視化されたESフロー図**を生成する。

---

## 入力と出力

**入力:**
1. ビジネス要求（既存システムの情報 + 追加したい機能）
2. **Miroボード URL（必須）** — 例: `https://miro.com/app/board/xxxxxxxxxx/`

Miroボード URL が指定されていない場合、ユーザーに必ず確認する。指定がなければエラーとして停止する（勝手にボードを作らない）。

ビジネス要求が不足している場合も、以下が揃うまで先に進まない：
- 既存システムの主要な概念・BC構成（判明している範囲で）
- 追加する機能のビジネス上の目的
- 機能が対象とするアクターと業務フロー

**出力:**
1. **Miro上のESフロー図**（diagram_create による flowchart DSL）
2. **BC名と責務リスト**（Markdown テーブル、会話内に出力）
3. ホットスポット一覧（⚠️として図と会話内の両方に残す）

---

## Agent Teams

モデリングは3つのペルソナで議論しながら進める。各ステップで3人が意見を出し合い、合意を形成する。

| ペルソナ | 役割 | 発言の特徴 |
|---|---|---|
| **ファシリテーター（F）** | 議論を進行し、合意を取り付ける。ホットスポットを記録する。安易に合意を取らず、各ペルソナの主張の根拠を問い質す | 「次に〇〇を検討しましょう」「DEの意見とLEの意見が異なりますが、それぞれの根拠を聞かせてください」「ここは意見が割れています。ホットスポットとして記録します」 |
| **ドメインエキスパート（DE）** | 業務の現場実態・例外的な運用・暗黙知を提供する。用語の正確性を検証し、ビジネスルールを明確にする。技術用語は使わない | 「現場では〇〇と呼んでいます」「実際の運用では〇〇のケースが月に数回あります」「このルールには例外があり、〇〇の場合は〇〇します」 |
| **リードエンジニア（LE）** | 技術的実現可能性を検証し、集約境界・トランザクション整合性を助言する | 「この2つは同一トランザクションで整合性が必要なので同一集約にすべきです」 |

**議論の進め方:**
- 各ステップで3人が順に意見を述べる（DE → LE → F の順が基本）
- 意見が割れた場合はファシリテーターがホットスポット（⚠️）として記録し、仮の判断を付ける
- **少なくとも1つの設計判断でDEとLEが異なる見解を持ち、トレードオフを明示した上で判断を下す場面を含めること**（形式的な合意だけでは深い設計にならない）
- 全員が合意したら次のステップに進む

---

## ワークフロー

```
1. 入力の確認           ビジネス要求 + Miro URL を確認
2. BC識別（最小限）      新機能が属するBC/新BCを特定し名前と責務を書き出す
3. Event Storming展開   ドメインイベント → コマンド → 集約 → ポリシー → リードモデル → アクター
4. 深掘り              ハッピーパス以外を検討し、ホットスポットを記録する
5. Miroへ出力          flowchart DSL でボードに書き出す
6. 結果報告            ボード内のノードID/URL、BC責務、ホットスポット一覧を会話で返す
```

---

## Step 1: 入力の確認

- ビジネス要求を読み、不足があれば `AskUserQuestion` でヒアリングする
- Miroボード URL が未指定であれば、ユーザーに URL を必ず問い合わせる（例: 「書き出す Miro ボードの URL を教えてください」）
- Miro URL を得たら、`mcp__claude_ai_Miro__context_get` などで既存内容を軽く確認する（上書き事故を避ける）

---

## Step 2: BC識別（最小限）

新機能が属するBCを特定する。厚い記述は不要で、**BC名と一行の責務**だけを出す。

```
| BC | 責務 |
|---|---|
| 休暇管理 | 休暇の申請・承認・残日数管理 |
| 勤怠記録 | 勤務時間・休暇反映の記録 |
```

**判断基準（最小限）:**
- 新機能が既存BCと同じ言葉を使っているか → 同じBC
- 新機能が独自の言葉・ライフサイクルを持つか → 新BC

BC間の連携パターン（ACL、準拠者 等）の詳細分析はこのスキルでは行わない。必要なら `conceptual-modeling` スキルを使うこと。

---

## Step 3: Event Storming を展開する

要素の定義・見つけ方は `references/elements.md` を参照。

**展開順序（この順で進める）:**

1. **ドメインイベント** — 「何が起きたか」を過去形で列挙する
2. **コマンド** — ドメインイベントを引き起こした操作を特定する
3. **集約** — コマンドを受け取り一貫性を保証するオブジェクトを特定する
4. **ポリシー** — あるイベントが別のコマンドを引き起こすルールを特定する
5. **リードモデル（ビュー）** — アクターが意思決定のために参照する情報を特定する
6. **アクター** — コマンドを実行する人・外部システムを特定する

展開中は各ステップで3ペルソナが必ず発言する。

---

## Step 4: 深掘り

ハッピーパスで終わっていないか確認する。以下の観点で問いを立てる：

- **失敗・エラーケース** — バリデーションエラー、外部連携失敗
- **例外フロー** — キャンセル、差戻し、期限切れ
- **タイムベースイベント** — 一定時間後に自動で何かが起きるか
- **集約の状態遷移** — 「このコマンドが拒否される状態はいつか？」

ホットスポット（⚠️）は放置せず、**図と会話内の両方に明示的に残す**。

状態遷移の有効/禁止パターンは `references/state-transitions.md` を参照。

---

## Step 5: Miroへ出力

`mcp__claude_ai_Miro__diagram_create` で flowchart DSL を生成する。

記法・色・クラスタ分割などのルールは `references/miro-notation.md` を参照。

**要点だけここに再掲:**
- 図形は **ホットスポット以外すべて長方形**（flowchart-process）、ホットスポットのみ flowchart-data
- **矢印はドメインイベント同士だけ** を結ぶ（LRレイアウトを維持し、視認性を高めるため）
- クラスタは **フェーズ単位**（BC単位ではない）
- 左端に **凡例クラスタ** を必ず入れる
- 色は 8 要素でZennの慣例に従う（詳細は `references/miro-notation.md`）

Miro MCP は要素位置の微調整APIを持たない。レイアウトは自動配置に任せる。

---

## Step 6: 結果報告

会話内に以下を返す：

1. **Miro 図へのリンク**（`moveToWidget=<nodeId>` 付き）
2. **BC名と責務** のテーブル
3. **ホットスポット一覧** — 図にも残しつつ、判断保留点として会話でも明示する
4. **DE/LE で意見が割れた論点** — 最終判断と根拠を1〜2行で
