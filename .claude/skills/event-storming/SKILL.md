# Event Storming (Design Level)

ビジネス要求を入力として、**Agent Teams**（3人体制）でDesign Level Event Stormingを実施し、**Miroボード上にBCごとのESフロー図**を生成する。

図の構造は「下位ループ（人がコマンドを発行するフロー）」と「上位ループ（ポリシーがコマンドを発行するフロー）」の**クラスタ**で表現する。クラスタは必ずイベントで終わり、クラスタ間はイベント → ビュー／ポリシーの矢印だけで繋ぐ。

---

## 入力と出力

**入力:**
1. ビジネス要求（既存システムの情報 + 追加したい機能）
2. **Miroボード URL（必須）** — 例: `https://miro.com/app/board/xxxxxxxxxx/`

Miroボード URL が指定されていない場合、ユーザーに必ず確認する。指定がなければエラーとして停止する（勝手にボードを作らない）。

ビジネス要求が不足している場合も、以下が揃うまで先に進まない：
- 既存システムの主要な概念・BC構成（判明している範囲で）
- 追加する機能のビジネス上の目的
- 機能が対象とするアクターと業務フロー

**出力:**
1. **Miro 上のESフロー図**（BCごとに1 diagram）
2. **凡例 diagram**（色見本 + 下位ループ例 + 上位ループ例）
3. **BC名と責務リスト**（Markdown テーブル、会話内）
4. **未解決事項（ホットスポット）リスト**（会話内のみ。Miroには描画しない）
5. **DE/LE で意見が割れた論点と最終判断**（会話内）

---

## Agent Teams

モデリングは3つのペルソナで議論しながら進める。各ステップで3人が意見を出し合い、合意を形成する。

| ペルソナ | 役割 | 発言の特徴 |
|---|---|---|
| **ファシリテーター（F）** | 議論を進行し、合意を取り付ける。未解決事項を記録する。安易に合意を取らず、各ペルソナの主張の根拠を問い質す | 「DEとLEの意見が異なりますが、それぞれの根拠を聞かせてください」「ここは判断保留。未解決事項として記録します」 |
| **ドメインエキスパート（DE）** | 業務の現場実態・例外的な運用・暗黙知を提供する。用語の正確性を検証し、ビジネスルールを明確にする。技術用語は使わない | 「現場では〇〇と呼んでいます」「実際の運用では〇〇のケースが月に数回あります」 |
| **リードエンジニア（LE）** | 技術的実現可能性を検証し、集約境界・トランザクション整合性を助言する | 「この2つは同一トランザクションで整合性が必要なので同一集約にすべきです」 |

**議論の進め方:**
- 各ステップで3人が順に意見を述べる（DE → LE → F の順が基本）
- 意見が割れた場合はファシリテーターが未解決事項として記録し、仮の判断を付ける
- **少なくとも1つの設計判断でDEとLEが異なる見解を持ち、トレードオフを明示した上で判断を下す場面を含めること**（形式的な合意だけでは深い設計にならない）
- 全員が合意したら次のステップに進む

## 重要な設計概念（必ず理解すること）

### 2ループ構造（Zenn bitkey_dev 記事準拠）

ESフローは以下の2種類のループで構成される。

**下位ループ（人が駆動する1サイクル）:**
```
ビュー → アクター → コマンド → 集約 → イベント
```
- アクターがビューを見て意思決定し、コマンドを発行する
- 集約がコマンドを受け取り不変条件を検証してイベントを発行する
- **1つのクラスタ = 1つの下位ループ = 必ずイベントで終わる**

**上位ループ（ポリシーが駆動する1サイクル）:**
```
ポリシー → コマンド → 集約 → イベント
```
- 先行イベントに反応してポリシーがコマンドを自動発行する
- 集約がコマンドを受け取りイベントを発行する
- **1つのクラスタ = 1つの上位ループ = 必ずイベントで終わる**

### 発見順序 ≠ 配置順序

- **発見順序** — 設計時に概念を洗い出す順番（イベント → コマンド → 集約 → ポリシー → リードモデル → アクター）
- **配置順序** — Miro 上での視覚的な並び順（上記2ループ構造に従って左から右へ水平配置）

この2つを混同してはならない。発見は思考の順、配置は読みやすさの順。

## ワークフロー

```
1. 入力の確認           ビジネス要求 + Miro URL を確認
2. BC識別（最小限）      新機能が属するBC/新BCを特定し名前と責務を書き出す
3. ES発見               各BCごとに要素を列挙する（発見順序）
4. クラスタ構造化        発見した要素を下位ループ／上位ループのクラスタに組み上げる
5. BC間遷移の扱い        BC境界を跨ぐイベントは両BCに重複して配置する
6. Miro DSL取得          diagram_get_dsl で現行の flowchart DSL 仕様を取得する
7. Miroへ出力           BCごとに diagram_create、最後に凡例 diagram を1枚
8. 結果報告              ボードリンク、BC責務、未解決事項を会話で返す
```

---

## Step 1: 入力の確認

- ビジネス要求を読み、不足があれば `AskUserQuestion` でヒアリングする
- Miroボード URL が未指定であれば、ユーザーに URL を必ず問い合わせる
- Miro URL を得たら、`mcp__claude_ai_Miro__context_get` で既存内容を軽く確認する（上書き事故を避ける）
- 既存図がある場合、**新規 diagram として別座標に配置する**か**追記するか**をユーザーに確認する

---

## Step 2: BC識別（最小限）

新機能が属するBCを特定する。厚い記述は不要で、**BC名と一行の責務**だけを出す。

```
| BC | 責務 |
|---|---|
| 休暇管理 | 休暇の申請・承認・残日数管理 |
| 勤怠記録 | 勤務時間・休暇反映の記録 |
```

**判断基準（最小限）:**
- 新機能が既存BCと同じ言葉を使っているか → 同じBC
- 新機能が独自の言葉・ライフサイクルを持つか → 新BC

BC間の連携パターン（ACL、準拠者 等）の詳細分析はこのスキルでは行わない。必要なら `conceptual-modeling` スキルを使うこと。

---

## Step 3: ES 発見（要素を列挙する）

BCごとに、以下の**発見順序**で要素を洗い出す。これは思考の順序であり、Miro上の配置順ではない。

1. **ドメインイベント** — 「何が起きたか」を過去形で列挙する
2. **コマンド** — 各イベントを引き起こした操作を特定する
3. **集約** — コマンドを受け取り一貫性を保証するオブジェクトを特定する
4. **ポリシー** — あるイベントが別のコマンドを引き起こすルールを特定する
5. **リードモデル / ビュー** — アクターが意思決定のために参照する情報を特定する
6. **アクター** — コマンドを実行する人・外部システムを特定する

要素の定義・見つけ方は `references/elements.md` を参照。

各ステップで3ペルソナが必ず発言する。DEとLEで意見が割れた場合は未解決事項として記録し、ファシリテーターが仮の判断を下す。

### 深掘り（発見時に意識的に問う）

ハッピーパスだけで終わらせない：
- **失敗・エラーケース** — バリデーションエラー、外部連携失敗
- **例外フロー** — キャンセル、差戻し、期限切れ
- **タイムベースイベント** — 一定時間後に自動で起きる遷移
- **集約の状態遷移** — このコマンドが拒否される状態はあるか

状態遷移の有効/禁止パターンは `references/state-transitions.md` を参照。

---

## Step 4: クラスタ構造化

Step 3 で列挙した要素を、**クラスタ = 1ループ** の単位に組み上げる。

### クラスタの作り方

**クラスタの定義:** イベントで終わる1ループ。内部の要素は左から右へ水平配置する。クラスタ内部に矢印は引かない（近接配置で順序を示す）。

**下位ループクラスタの内部構成（左から右）:**
```
[ビュー] [アクター] [コマンド] [集約] [イベント]
```

**上位ループクラスタの内部構成（左から右）:**
```
[ポリシー] [コマンド] [集約] [イベント]
```

### クラスタ間の接続

**矢印は「イベント → ビュー」または「イベント → ポリシー」のみ。**

- イベントが次の下位ループを引き起こす場合: `イベント → 次クラスタのビュー`
- イベントが次の上位ループを引き起こす場合: `イベント → 次クラスタのポリシー`
- イベント同士を直接矢印で結んではならない（これは旧ルールで、時系列が読めなくなるため廃止）
- クラスタ内部の要素間にも矢印を引かない

### クラスタ配置のルール

- 各クラスタは Miro 上で1つのグループ（サブグラフ/cluster）として表現する
- クラスタ自体は左から右へ時系列順に並べる
- BC単位では diagram を分けるため、1つの diagram 内には同一BCのクラスタだけが並ぶ

---

## Step 5: BC間遷移の扱い

BC 境界を跨ぐイベントは、**送信側BCの最後と受信側BCのクラスタ起点の両方に同じイベントを重複して配置する**。

- 送信側BC diagram: 発生元クラスタの末尾にイベントを置く
- 受信側BC diagram: そのイベントを先頭に置き、次のクラスタ（上位ループ）の起点として `イベント → ポリシー → コマンド → 集約 → イベント` を展開する
- diagram 間を跨ぐ矢印は描かない（Miro の diagram は独立しているため）
- 会話内の報告テーブルに「BC間遷移イベント」として一覧を明示する

例:
```
| 送信元BC | イベント | 受信先BC | 受信後の動作 |
|---|---|---|---|
| 注文 | 注文が確定された | 在庫 | ポリシー「注文確定時は在庫を引き当てる」→ 在庫引当コマンド |
```

---

## Step 6: Miro DSL 仕様の取得

`mcp__claude_ai_Miro__diagram_create` は DSL テキストを受け取る。DSL の正式な文法は Miro 側で管理されているため、**必ず `mcp__claude_ai_Miro__diagram_get_dsl` を先に呼んで現行仕様を取得する**。

```
mcp__claude_ai_Miro__diagram_get_dsl(
  miro_url: "<ユーザー指定のURL>",
  diagram_type: "flowchart"
)
```

返ってきた仕様に沿って DSL テキストを構築する。色コード・クラスタ構文・矢印構文の具体的な書き方は `references/miro-notation.md` を参照。

---

## Step 7: Miro へ出力

以下の順で diagram を作成する。座標は BC ごとに縦方向にずらし、重ならないようにする。

1. **BCごとに diagram_create を1回ずつ呼ぶ**
   - `diagram_type: "flowchart"`
   - `title: "ES: <BC名>"`
   - `x`, `y` で配置位置を分ける（例: BC1 は y=0、BC2 は y=800 など）
   - `diagram_dsl` に Step 4 で組んだクラスタ構造を流し込む

2. **最後に凡例 diagram を1枚作る**
   - `title: "凡例"`
   - 含める内容:
     - 7要素の色見本（イベント／コマンド／集約／ポリシー／リードモデル／ビュー／アクター）
     - 下位ループのサンプルクラスタ（`[ビュー][アクター][コマンド][集約][イベント]`）
     - 上位ループのサンプルクラスタ（`[ポリシー][コマンド][集約][イベント]`）
   - 位置は全BC diagram の左端または上部（読み手が最初に見る場所）

**未解決事項（ホットスポット）は Miro に描画しない。** 会話内の報告でのみ一覧化する（視覚的ノイズを避け、判断保留事項であることを明確にするため）。

---

## Step 8: 結果報告

会話内に以下を返す：

1. **Miro ボードリンク** — 各 BC diagram への `?moveToWidget=<frame_id>` 付きリンク（取得できれば）
2. **BC名と責務** のテーブル
3. **BC間遷移イベント** のテーブル（Step 5 のもの）
4. **未解決事項（ホットスポット）一覧** — 「ビジネスルール〇〇が未確定」「DE/LEで判断が割れた」等
5. **DE/LE で意見が割れた論点と最終判断** — 1〜2行で根拠を添える
