# テーブル設計（table-design）

conceptual-modelingスキルの出力を入力として、**Agent Teams**（3人体制）で議論しながら、イミュータブルデータモデリングを指向したRailsテーブル設計を生成する。

---

## 入力と出力

**入力:** conceptual-modelingスキルの出力（必須） + class-designスキルの出力（任意）

conceptual-modelingの出力が提供されていない場合は、ユーザーに先にconceptual-modelingの実行を促す。以下の成果物が揃っていることを確認する：
1. 概念モデル図（Mermaid `classDiagram`）
2. イベントストーミング結果（Mermaid `flowchart LR`）
3. ユビキタス言語（BC別テーブル）
4. BC間連携マトリクス
5. ポリシー整合性分類テーブル
6. 集約分離の判断根拠テーブル（該当する場合）

**出力:**
1. ER図（Mermaid `erDiagram`）
2. テーブル定義一覧（カラム名・型・制約・NOT NULL・デフォルト値・説明）
3. インデックス戦略（各インデックスの目的・種類）
4. マイグレーション順序（依存関係に基づく実行順）
5. 設計判断記録（イミュータブル/ミュータブル選択、正規化/非正規化の根拠等）

---

## Agent Teams

設計は3つのペルソナで議論しながら進める。DB設計の最適解とRailsの規約の間の緊張関係が、より良い設計判断を生む。

| ペルソナ | 役割 | 発言の特徴 |
|---|---|---|
| **DBA（D）** | 正規化・インデックス戦略・制約設計・パフォーマンス・イミュータブル設計を担う。データの物理的な効率と整合性を守る | 「このカラムにはユニーク制約が必要です」「履歴テーブルにすべきか、UPDATEで十分か」「複合インデックスの順序はカーディナリティの高い方を先に」 |
| **Railsエンジニア（RE）** | Rails規約・ActiveRecord互換性・マイグレーションパターンを検証する。Railsの規約から不必要に逸脱しない | 「Rails規約では`_id`サフィックスです」「polymorphicならtypeカラムも必要です」「delegated_typesの方がSTIより拡張しやすい」 |
| **レビュアー（R）** | 変更容易性・データ整合性・ドメインモデルとの整合性を検証する。過剰設計と過少設計の両方を指摘する | 「このテーブル分割は将来の要件変更に耐えますか？」「イミュータブルにするコストに見合いますか？」「概念モデルと乖離していませんか？」 |

**議論の進め方:**
- 各ステップでD → RE → R の順に意見を述べる（DB設計の理想 → Railsの現実 → 変更容易性の検証）
- 意見が割れた場合はレビュアーがトレードオフを整理し、判断を「設計判断記録」に残す
- **少なくとも1つの設計判断でDとREが異なる見解を持ち、トレードオフを明示した上で判断を下す場面を含めること**（例: イミュータブル vs ミュータブル、正規化 vs 非正規化、CHECK制約 vs アプリケーションバリデーション）
- 全員が合意したら次のステップに進む

---

## 設計思想: イミュータブルデータモデリング

このスキルはイミュータブルデータモデリングをデフォルトの設計指向とする。データベースに記録するのは「事実」であり、事実は変更されないという考え方に基づく。ただし、全テーブルをイミュータブルにするのではなく、トレードオフを踏まえて使い分ける。

詳細は `references/immutable-data-modeling.md` を参照。

**判断の基本方針:**
- 業務イベント（注文した、支払った、承認した）→ イミュータブル（INSERT-only）
- 状態管理で履歴が必要 → 状態テーブル + 履歴テーブル or 有効期間モデル
- 確定後に変更されないデータ（請求書等）→ スナップショット
- 現在の状態で履歴が不要（ユーザー設定等）→ ミュータブル（UPDATE）

---

## ワークフロー

```
1. 入力の検証                conceptual-modeling出力が揃っているか確認
2. イミュータブル/ミュータブル分類  各エンティティの更新パターンを3人で議論
3. マッピング方針の決定        DDD概念→テーブル構造の方針を決定
4. ER図の作成                Mermaid erDiagramでテーブル間関連を設計
5. テーブル定義の詳細化        カラム・型・制約を定義
6. インデックス戦略            検索パターンに基づくインデックス設計
7. class-designとの整合性確認  （入力がある場合）
8. 成果物の保存               Markdownファイルとして保存
```

---

## Step 1: 入力の検証

conceptual-modelingの出力が揃っているか確認する。不足がある場合はユーザーに先にconceptual-modelingの実行を促す。

**確認項目:**
- 概念モデル図に `<<集約ルート>>` が含まれているか
- ESに集約・コマンド・ドメインイベントが含まれているか
- ユビキタス言語がBC別に定義されているか
- ポリシー整合性分類テーブルがあるか

class-designの出力がある場合は、クラス構造も読み込む。

---

## Step 2: イミュータブル/ミュータブル分類

3人チームで、各エンティティ・ドメインイベントの更新パターンを分類する。

**分類テーブルを作成する:**

```
| エンティティ/イベント | 分類 | パターン | 根拠 |
|---|---|---|---|
| 注文イベント | イミュータブル | イベントテーブル | 業務の事実記録、変更不可 |
| サブスクリプション | ハイブリッド | 状態テーブル + 履歴 | 現在状態の高速参照が必要だが履歴も重要 |
| 料金プラン | イミュータブル | 有効期間モデル | プラン変更は新世代の追加 |
| ユーザー設定 | ミュータブル | UPDATE | 履歴不要、現在値のみ重要 |
```

この分類がStep 3以降の全ての設計判断の基盤になる。

---

## Step 3: マッピング方針の決定

3人チームで、DDD概念からテーブル構造へのマッピング方針を議論する。

詳細は `references/ddd-to-table-mapping.md` を参照。

**議論すべきポイント:**

1. **集約ルート → メインテーブル** — テーブル名、主キー戦略（auto increment vs UUID）
2. **エンティティ → 子テーブル** — FK制約、カスケード削除の方針
3. **値オブジェクト → カラム群 / JSON / 参照テーブル** — 永続化パターンの選択
4. **列挙型 → enum / CHECK制約 / 参照テーブル** — 値の可変性で判断
5. **ドメインイベント → イベントテーブル** — INSERT-onlyの設計
6. **BC境界 → テーブル名プレフィックス** — BC間参照はFK制約なしのIDのみ
7. **正規化 / 非正規化** — パフォーマンスと整合性のトレードオフ

各判断の根拠を「設計判断記録」に残す。

---

## Step 4: ER図を作成する

Step 2-3の方針に基づき、Mermaid `erDiagram` でテーブル間関連を設計する。

詳細は `references/er-diagram-guide.md` を参照。

**ER図のルール:**
- テーブル名はRails規約に従う（複数形スネークケース）
- BC境界はテーブル名プレフィックスで表現（例: `billing_subscriptions`）
- カラムは型付きで記載（例: `bigint id PK`）
- FK制約は集約境界内に限定する
- BC間の参照はFK制約なしのIDカラム（コメントで明示）
- イミュータブルテーブルはコメントで `"INSERT-only"` や `"immutable"` を明示
- 関連ラベルは英語の動詞句

**概念モデル図との対応確認（Step 4の末尾で実施）:**
- 概念モデルの全 `<<集約ルート>>` がテーブルとして存在するか
- 概念モデルの全 `<<エンティティ>>` が子テーブルとして存在するか
- 概念モデルの全 `<<値オブジェクト>>` が適切な永続化方式で反映されているか
- BC境界がテーブル名プレフィックスとして反映されているか

---

## Step 5: テーブル定義の詳細化

ER図の各テーブルについて、詳細なカラム定義を作成する。

**テーブル定義の形式:**

```
### billing_subscriptions（サブスクリプション）

集約ルート / ミュータブル（状態テーブル）

| カラム名 | 型 | NULL | デフォルト | 制約 | 説明 |
|---|---|---|---|---|---|
| id | bigint | NO | auto | PK | |
| plan_id | bigint | NO | | FK | 契約プラン |
| status | string | NO | | CHECK | pending/active/canceled |
| anchor_date | date | NO | | | 契約基準日 |
| canceled_at | datetime | YES | | | 解約日時 |
| lock_version | integer | NO | 0 | | 楽観ロック |
| created_at | datetime | NO | | | |
| updated_at | datetime | NO | | | |
```

**各テーブルに明記する情報:**
- テーブルの日本語名
- DDD上の役割（集約ルート / エンティティ / イベント / 参照テーブル）
- イミュータブル/ミュータブルの分類
- 楽観ロックの要否とその理由

---

## Step 6: インデックス戦略

検索パターンに基づいて、各テーブルのインデックスを設計する。

**インデックス定義の形式:**

```
| テーブル | インデックス | 種類 | 目的 |
|---|---|---|---|
| billing_subscriptions | (user_id) | INDEX | ユーザー別サブスクリプション検索 |
| billing_subscriptions | (status, anchor_date) | COMPOSITE | ステータス別の期日順表示 |
| billing_subscriptions | (user_id, status) | COMPOSITE + UNIQUE (partial) | アクティブサブスク一意制約 (WHERE status = 'active') |
| billing_subscription_events | (subscription_id, occurred_at) | COMPOSITE | 時系列イベント取得 |
```

**インデックス設計の原則:**
- 主要なWHERE句とORDER BY句をカバーする
- 複合インデックスはカーディナリティの高い方を先に
- partial indexは条件付きユニーク制約に活用
- イベントテーブルは（FK, 時刻）の複合インデックスを基本とする
- 不要なインデックスは追加しない（書き込みコスト考慮）

---

## Step 7: class-designとの整合性確認

class-designの出力がある場合のみ実施する。

**確認項目:**
- ActiveRecord model がテーブルと1:1で対応しているか
- `has_many` / `belongs_to` がFK構造と一致しているか
- 値オブジェクトの実装方式がテーブル設計と整合しているか（`composed_of` → 複数カラム、`store_model` → JSON型カラム）
- STI / delegated_types の使用がテーブル設計と一致しているか
- 状態遷移図の状態値がenumカラムの値と一致しているか
- イミュータブルテーブルに対応するモデルがreadonly設計になっているか

**不整合がある場合:**
- table-design側の変更で解決できるか検討
- class-design側の変更が必要な場合は「class-designへのフィードバック」として記録
