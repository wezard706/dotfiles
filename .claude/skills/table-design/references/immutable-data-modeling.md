# イミュータブルデータモデリングガイド

## このドキュメントの目的

テーブル設計においてイミュータブルをデフォルトの指向としつつ、トレードオフを理解して適切に使い分けるための判断材料を提供する。

---

## イミュータブルデータモデリングとは

### 核心的な考え方

- データベースに記録するのは「事実」であり、事実は変更されない
- UPDATEやDELETEは事実の書き換え — 本来は新しい事実のINSERTで表現すべき
- 「注文がキャンセルされた」は「注文を削除する」ではなく「キャンセルイベントを記録する」

### 動機

| 動機 | 説明 |
|---|---|
| 監査証跡 | 変更履歴が自然に得られる。別途audit_logテーブルを設計する必要がない |
| データ整合性 | 過去の事実が消えない。誤UPDATEによるデータ喪失がない |
| 時系列分析 | 「いつ何が起きたか」を時系列で分析可能 |
| デバッグ容易性 | バグ調査時に「何が起きたか」を完全に追跡可能 |
| 並行処理の安全性 | 行ロックの競合が減る（INSERT同士は行レベルでは衝突しない）。ただしドメイン不変条件（例: 残高が0以上）の保証には楽観ロック・一意制約等の別の仕組みが必要 |

## パターンカタログ

### イベントテーブル（INSERT-only）

最も基本的なイミュータブルパターン。業務上の出来事を事実として記録する。

**テーブル例: `subscription_events`**

| カラム | 型 | 説明 |
|---|---|---|
| id | bigint | PK |
| subscription_id | bigint | FK |
| event_type | string | created / renewed / canceled / ... |
| payload | jsonb | イベント固有データ |
| occurred_at | datetime | イベント発生日時 |
| created_at | datetime | レコード作成日時 |

**特徴:**
- UPDATE/DELETEしない
- `occurred_at` と `created_at` を分離する（イベント発生 ≠ 記録タイミング）
- `event_type` でイベントの種類を判別する

**`payload: jsonb` の注意点:**

payloadをjsonbで設計すると以下のリスクがある:
- スキーマレスになるため、イベントの型安全性が失われる
- 時間が経つとpayloadの構造が不明になる
- RDBMSの制約（NOT NULL、CHECK、FK）が効かない

対策として以下を検討する:
- アプリケーション層でのバリデーション（JSONスキーマ等）でpayloadの構造を保証する
- **代替案:** イベント種別ごとに専用カラムを持つ設計（型安全性とDB制約を活かせる）

**適用場面:** 注文、支払い、承認、ステータス変更など、業務上の出来事すべて

---

### 状態テーブル + 履歴テーブル

現在状態を高速に参照しつつ、変更履歴を保持するパターン。

**構成:**
- `subscriptions` — 現在の状態（ミュータブル、UPDATEあり）
- `subscription_status_changes` — 状態変更の履歴（イミュータブル）

**テーブル例: `subscription_status_changes`**

| カラム | 型 | 説明 |
|---|---|---|
| id | bigint | PK |
| subscription_id | bigint | FK |
| from_status | string | 変更前の状態 |
| to_status | string | 変更後の状態 |
| reason | string | 変更理由 |
| changed_at | datetime | 変更日時 |
| created_at | datetime | レコード作成日時 |

**トレードオフ:**
- 現在状態の参照はO(1)で高速
- 2テーブル間の整合性維持が必要（トランザクションで束ねる）
- イベントテーブル単体より実装がやや複雑

**適用場面:** 現在状態への頻繁なアクセスがあり、かつ変更履歴も必要な場合

---

### 有効期間モデル（Temporal Model）

`valid_from` / `valid_to` で世代管理するパターン。

**テーブル例: `plans`（料金プラン）**

| カラム | 型 | 説明 |
|---|---|---|
| id | bigint | PK |
| plan_code | string | プラン識別子 |
| name | string | プラン名 |
| price | decimal | 月額 |
| valid_from | date | 適用開始日 |
| valid_to | date | 適用終了日（null = 現行） |

**変更操作:** 現行レコードの `valid_to` をUPDATEし、新レコードをINSERT。

> **注意:** 有効期間モデルはセミイミュータブルである。新世代のINSERTは不変だが、旧世代のクローズには `valid_to` のUPDATEが必要。完全にイミュータブルにしたい場合は、`valid_to` を持たず「次のレコードの `valid_from` が前のレコードの終了を暗示する」隣接リスト方式も検討する。

**クエリ:** `WHERE valid_from <= NOW() AND (valid_to IS NULL OR valid_to > NOW())`

**適用場面:** 料金プラン、契約条件、組織構造など、期間ごとに値が変わるマスタデータ

---

### スナップショット

確定時点のデータを丸ごと保存するパターン。

**テーブル例: `invoice_line_items`（請求明細）**

| カラム | 型 | 説明 |
|---|---|---|
| id | bigint | PK |
| invoice_id | bigint | FK |
| product_id | bigint | FK（参照元） |
| product_name | string | 確定時の商品名 |
| unit_price | decimal | 確定時の単価 |
| tax_rate | decimal | 確定時の税率 |
| quantity | integer | 数量 |

**特徴:**
- 請求確定時の商品名・単価・税率をスナップショットとして保存
- 元テーブル（`products`）が後で変更されても請求データは不変
- `product_id` は参照用に保持するが、表示には使わない

**適用場面:** 請求書、契約書、注文確認など、確定後に参照元の変更が影響してはならないデータ

---

### ミュータブル（従来型UPDATE）

イミュータブルが不適切な場合の選択肢。

**適用場面:**
- ユーザープロフィール（ただしドメインによる。HR系で姓が変わる場合は「事実」。**ビジネスドメインがその変更を事実として扱うかどうか**で判断する）
- システム設定値
- キャッシュ・集計テーブル
- マスタデータの軽微な修正

---

## イミュータブル vs ミュータブルの判断フロー

```
このデータは「事実の記録」か「現在の状態」か？
├── 事実の記録（注文した、支払った、承認した）
│   └── イベントテーブル（INSERT-only）
├── 現在の状態で履歴が必要
│   ├── 頻繁に現在値を参照する
│   │   └── 状態テーブル + 履歴テーブル
│   └── 期間ごとに値が変わる
│       └── 有効期間モデル
├── 確定後に変更されないデータ
│   └── スナップショット
└── 現在の状態で履歴が不要
    └── ミュータブル（UPDATE）
```

## Railsでの実装パターン

### INSERT-onlyモデル

UPDATEとDELETEをコードレベルで防止する。

```ruby
class SubscriptionEvent < ApplicationRecord
  belongs_to :subscription

  # UPDATEとDELETEを防止
  after_initialize :readonly!, if: :persisted?
  before_destroy { raise ActiveRecord::ReadOnlyRecord }
end
```

### 有効期間モデルのscope

```ruby
class Plan < ApplicationRecord
  scope :current, -> {
    where("valid_from <= ? AND (valid_to IS NULL OR valid_to > ?)", Date.current, Date.current)
  }

  scope :at, ->(date) {
    where("valid_from <= ? AND (valid_to IS NULL OR valid_to > ?)", date, date)
  }
end
```

### 状態テーブル + 履歴テーブルの同期

```ruby
class Subscription < ApplicationRecord
  has_many :status_changes, class_name: "SubscriptionStatusChange"

  # 状態更新 → 履歴記録の順で実行する。
  # 理由: 状態テーブルの更新が失敗した場合に不整合な履歴が残らないようにするため。
  # （トランザクションでロールバックされるとはいえ、コードの意図を明示する）
  def change_status!(new_status, reason: nil)
    ActiveRecord::Base.transaction do
      old_status = status
      update!(status: new_status)
      status_changes.create!(
        from_status: old_status,
        to_status: new_status,
        reason: reason,
        changed_at: Time.current
      )
    end
  end
end
```

## トレードオフ分析

| 観点 | イミュータブル | ミュータブル |
|---|---|---|
| ストレージ | 多い（履歴を保持） | 少ない |
| クエリ複雑性 | 高い（最新値の取得にJOIN/サブクエリ） | 低い |
| 監査証跡 | 自然に得られる | 別途実装が必要 |
| 並行処理 | 競合しにくい | ロックが必要 |
| データ整合性 | 高い | 更新異常のリスク |
| 実装コスト | 初期コスト高い | 低い |
| デバッグ | 容易（全履歴あり） | 困難（現在値のみ） |

## アンチパターン

### 過剰なイミュータブル化

ユーザー名の変更を「イベント」として記録する必要はない（通常は）。全テーブルをイミュータブルにすると複雑性が爆発する。

**判断基準:** 「この変更履歴にビジネス上の価値があるか？」を問う。価値がなければミュータブルで十分。

### 現在状態テーブルの欠如

イベントテーブルだけでは「今の状態」の取得が遅い。頻繁に参照する現在状態は、イベントから導出したマテリアライズドビューか状態テーブルを持つ。

**症状:** `SELECT ... ORDER BY occurred_at DESC LIMIT 1` が頻発する。

### 不完全なイミュータブル化

イミュータブルと宣言しつつ、例外的にUPDATEするケースが散在する。「基本イミュータブルだが修正可能」は最悪のパターン — どちらかに決める。

**対策:** `readonly!` や `before_destroy` で物理的にUPDATE/DELETEを防止する。規約だけに頼らない。

### 適切な粒度の見極め失敗

- イベントの粒度が細かすぎる — フィールドごとの変更イベントを個別に記録すると、1つの業務操作を復元するのに多数のレコードが必要になる
- イベントの粒度が粗すぎる — 全フィールドを含む巨大イベントは、何が変わったのか判別しにくい

**判断基準:** 1つの業務操作（コマンド）= 1つのイベントを基本とする。

## データ保持・アーカイブ戦略

イミュータブルテーブルはデータが単調増加するため、運用面での計画が不可欠。

### アーカイブ手法

| 手法 | 説明 | 適用場面 |
|---|---|---|
| テーブルパーティショニング | `occurred_at` や `created_at` で月次/年次パーティション | イベントテーブル全般 |
| コールドストレージ移行 | 古いパーティションをS3等に移動 | 参照頻度が低い過去データ |
| 集約テーブル | 古いイベントを日次/月次集計に変換 | 分析用途 |

### 考慮事項

- **法的保持期間**: ドメインによって保持義務期間が異なる（例: 会計データは7年）。アーカイブ前に確認する
- **パフォーマンス**: データ増加に伴うインデックスサイズの肥大化に注意。パーティショニングにより古いデータのインデックスを分離できる
- **見積もり**: テーブル設計時にデータ増加量を見積もる（例: 1日あたりのイベント数 × 保持期間）

## チェックリスト

テーブル設計時に確認する:

- [ ] 各テーブルがイミュータブル/ミュータブルのどちらかに明確に分類されている
- [ ] 分類の根拠が判断フロー（事実の記録 / 現在の状態 / 確定データ）に基づいている
- [ ] イミュータブルテーブルにはUPDATE/DELETE防止策が実装されている
- [ ] イベントテーブルの `occurred_at` と `created_at` が分離されている
- [ ] 頻繁に参照される現在状態には状態テーブルまたはマテリアライズドビューがある
- [ ] 有効期間モデルには `current` / `at(date)` のscopeが定義されている
- [ ] スナップショットには確定時点の値がすべて含まれている（FKだけに依存していない）
- [ ] イベントの粒度が業務操作（コマンド）と1:1で対応している
- [ ] イミュータブルテーブルのデータ増加見積もりとパーティショニング/アーカイブ戦略がある
- [ ] payload（jsonb）を使う場合、その構造の検証方法が定められている
- [ ] 有効期間モデルの重複期間（overlap）を防ぐ制約がある
