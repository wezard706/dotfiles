# 正確性（バグ・ロジックエラー）レビュールーブリック

## 問い

このdiffは意図どおりに動くか？壊れるのはどんな入力・タイミング・並行実行のときか？

セキュリティ・パフォーマンス・設計の観点ではなく、**純粋な正しさ**（ロジック・エラー処理・原子性・並行性）だけを見る。
diffの各変更に対して「この行が壊れるとしたらどういう条件か」を自問しながら読むこと。

## チェックリスト

### 1. ロジックエラー

- off-by-one: 境界値（0件・1件・最終要素・ページ境界）で正しいか。`<` と `<=`、`between?` の端点
- 条件式の論理誤り: ド・モルガン間違い、`&&`/`||` の取り違え、否定の重ね掛け
- nil / undefined 処理: `&.` チェーンの途中でnilになったときの挙動、`dig` の結果、Hashのキー欠落、TypeScriptの `!`（non-null assertion）が実際に保証されているか
- 早期return漏れ: ガード節を通過した後の前提が本当に成立しているか
- デフォルト値・フォールバックの妥当性: `|| 0` や `?? ""` が業務的に正しい既定値か（0と「未入力」の混同）

### 2. エラーハンドリング

- 例外の握りつぶし: `rescue => e` して何もしない / ログだけ出して正常系として続行
- rescue範囲の過大: `rescue StandardError` が意図しない例外（プログラミングエラー）まで飲み込んでいないか
- エラー時のリソースリーク: ファイル・コネクション・ロックが例外時に解放されるか（ensure / using パターン）
- 部分失敗の扱い: バッチ・ループ内の1件の失敗が全体をどう扱うか意図されているか（全ロールバック？スキップ？）

### 3. トランザクション境界

- 複数の書込みが原子的であるべきなのに transaction で囲まれていないか
- トランザクション内の外部API呼び出し・メール送信: ロールバックしても取り消せない副作用 / 長時間ロック保持
- `after_commit` と `after_save` の取り違え（コミット前に非同期ジョブへIDを渡してレコード未可視）
- ジョブへの引数: レコード本体を渡してデシリアライズ時に古い状態にならないか（IDを渡して再取得が原則）

### 4. 並行性

- check-then-act の競合: `exists?` チェック後の `create`、`find_or_create_by` の競合（一意制約 + リトライで保護されているか）
- 一意性がバリデーション（`validates :x, uniqueness: true`）だけでDB制約がない
- カウンタ・残高の read-modify-write（`update(count: count + 1)` は `increment!` / UPDATE文 / ロックへ）
- 楽観ロック・悲観ロックの要否: 同一レコードを複数プロセスが更新するパスか

### 5. フロントエンド固有（.ts / .tsx 変更時）

- 非同期処理のエラー処理漏れ: await の reject が未処理 / `.then` だけで `.catch` なし / エラー時のUI状態（ローディングが止まらない等）
- `useEffect` の依存配列: 参照している値が配列から漏れて古い値を掴む（stale closure）/ オブジェクトを依存に入れて毎回発火
- クリーンアップ漏れ: 購読・タイマー・AbortControllerの解除。アンマウント後のsetState
- 競合状態: 連続リクエストで古いレスポンスが後着して新しい状態を上書き（リクエストのキャンセル・順序保証）
- 状態更新の関数形式: 前の状態に依存する更新で `setCount(count + 1)` を連打していないか（`setCount(c => c + 1)`）

## 指摘の規律

- 「壊れる具体的な条件」を必ず添える（どんな入力・タイミングで何が起きるか）。条件を提示できない指摘はしない
- 変更行に起因する問題のみ。既存コードの潜在バグは対象外
