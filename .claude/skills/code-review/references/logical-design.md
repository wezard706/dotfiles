# 論理設計（テーブル設計）レビュールーブリック

## 問い

テーブル・カラムの論理設計は、事実を無駄なく正しく記録できる構造か？
偶有的複雑性（論理削除・テーブル分割・時系列など、業務概念ではなく実装上の関心事）を適切なパターンで処理しているか？

**適用範囲**: `db/migrate/`・`db/schema.rb`（または structure.sql）の変更。
概念レベルの妥当性（命名・イベント抽出・存在依存）は別観点「モデリング」が扱う。

## チェックリスト

### 1. 正規化〔達人に学ぶDB設計 / zenn〕

- One fact in one place。同じ事実が複数の場所に記録されていないか（導出可能な値の重複保存）
- JOIN・パフォーマンス懸念を理由にした安易な非正規化はないか。**非正規化は最後の手段**であり、採用するなら理由と整合性維持の方法が明示されているべき

### 2. NULL排除〔kawasima / 達人〕

- 追加された nullable カラムは本当に「値が無い」ことがある事実か。**nullableは設計不備のサイン**として代替を検討したか
  - 未確定の関連（配属前の社員の `department_id`）→ 交差テーブル（所属）に分離
  - 時系列で後から埋まる値（`shipped_at`, `billing_id`）→ 別イベントテーブルに分離
  - 状態によって使わないカラム → 状態別テーブル分割（active/leave）
- 「後でUPDATEして埋める」前提のカラムは時系列逆転のサイン

### 3. UPDATE最小化〔kawasima〕

- イベント系テーブルに複数の日時属性がないか（`ordered_at` と `confirmed_at` と `shipped_at` の同居）→ テーブル分解でINSERT-onlyにできる
- 更新ルールが暗黙に生まれる構造（「確認時は confirmed_at と confirmed_by のみ更新可」のような、スキーマから読み取れない制約）を作っていないか

### 4. 制約の過不足〔SQLアンチパターン / 失敗から学ぶRDB〕

- 外部キー制約の欠如（Keyless Entry）: `add_reference` に `foreign_key: true` があるか
- NOT NULL / UNIQUE / CHECK 制約の不足: アプリ側バリデーションだけに頼っていないか（並行実行で破れる）
- 逆に強すぎる制約: 業務上ありうる状態を制約が禁止していないか

### 5. 削除フラグ・論理削除〔失敗から学ぶRDB / そーだい〕

- `is_deleted` / `deleted_at` / `disabled` 等の追加を検出したら指摘する
  - UNIQUE制約が実質無効化される（削除済みと同じ値が登録できない/できてしまう）
  - すべてのクエリに `WHERE deleted_at IS NULL` が漏れなく必要になる（漏れ＝事故）
- 代替を提案する: **状態遷移としてのモデリング**（削除ではなく退会・アーカイブという状態）/ **状態別テーブル分割**（`users` + `active_users` / `withdrawn_users`）/ **削除イベントテーブル**
- 機微情報（個人情報）は物理削除できる設計になっているか

### 6. フラグ・ステータスの闇〔失敗から学ぶRDB / そーだい期間設計〕

- boolean フラグの乱立（`is_active`, `is_premium`, `is_locked` ...）: 組合せ爆発と不正な組合せの混入。状態やサブタイプへの整理を提案
- 単一 `status` カラムの上書きによる履歴喪失: いつ・誰が・どの状態からの遷移かが失われる。遷移をイベントテーブルに記録する構造を検討したか
- 「キャンセル」と「終了」を同じカラム・同じ値域で表現していないか（意味の異なる終了理由の混同）

### 7. 期間・履歴・世代〔そーだい / kawasima〕

- `start_at` / `end_at` 方式で `end_at` が NULL 許容になっていないか（未終了・キャンセル・無期限の区別がつかない）
- **スナップショット**（過去時点の再現: 注文時の商品価格はイベント側に焼き込む）と**世代管理**（マスタの適用開始/終了: 未来の価格改定）を混同していないか。目的別にモデルを分離する
- 世代管理の実装パターンを比較検討したか: シングル世代テーブル（過去分のみ分離）/ 有効世代ビュー（最新のみMaterialized View）/ 世代バージョンタグ付け（Gitのversion+tagに似た構造）
- 全クエリに期間BETWEEN条件を強いる素朴な世代テーブルは開発負荷が高い

### 8. サブタイプ実装の選択〔kawasima〕

スーパータイプ/サブタイプ（区分）の実装が要件に合っているか:

| パターン | 適する条件 |
|---|---|
| シングルテーブル継承 | 区分横断の検索が主 / 区分間の属性差が小さい |
| 具象テーブル継承 | 横断検索がほぼ無い / 属性差が大きい |
| クラステーブル継承 | 横断検索あり / 属性差も大きい |

### 9. SQLアンチパターン検出〔SQLアンチパターン / 達人 / 失敗から学ぶRDB〕

- **Jaywalking**: カンマ区切り値・配列的文字列カラム
- **EAV**: `entity_id, attr_name, attr_value` 形式の汎用属性テーブル
- **ポリモーフィック関連**: `commentable_type` + `commentable_id`（FK制約が張れない）。Railsの `polymorphic: true` は要注意サイン — 交差テーブルや排他的FKの検討
- **Multicolumn Attributes**: `tag1, tag2, tag3` のような繰り返しカラム
- **Metadata Tribbles**: 年度別・月別テーブル（`sales_2024`, `sales_2025`）
- **ダブルミーニング**: 1カラムが文脈によって別の意味を持つ
- **単一参照テーブル（汎用マスタ）**: あらゆるコード値を1つの `codes` テーブルに集約
- **田分け**: 意味なく分割されたテーブル（水平・垂直分割の濫用）

### 10. テーブル・カラムの責務〔zenn〕

- ファットテーブル: 責務の異なるカラム群が1テーブルに同居していないか
- 複数カラムの組合せで初めて意味が成立するカラムはないか（`type` によって `value` の意味が変わる等）
- ドメインを無視した画一的な `id, name` テーブルの量産

### 11. JSON/JSONBの濫用〔失敗から学ぶRDB / そーだい〕

- 必須情報・検索条件になる情報がJSONカラムに埋まっていないか（部分更新に弱い・制約が張れない）
- JSONが妥当なのは「真にスキーマレスな任意情報」のみ。カラムに切り出せるものは切り出す

### 12. DBリファクタリング安全性〔そーだい「DBリファクタリング」〕

既存テーブルの構造変更（カラム移設・分割・型変更）を含む場合:

- 段階的移行の計画があるか: ダブルライト（新旧両構造への書込み）→ 検証 → 切替 → 旧構造削除
- 旧インターフェース互換のビュー提供を検討したか
- データ整合性チェック（新旧の突合）の仕組みがあるか
- ビッグバンリリース（一発切替）になっていないか
