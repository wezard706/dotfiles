# パフォーマンス レビュールーブリック

## バックエンド（Rails）

- **N+1クエリ**: ループ・map内でのDBクエリ発行、association参照時の eager loading（`includes` / `preload`）の欠如。部分テンプレート内のassociation参照も対象
- **不要な処理**: 同じ計算・同じクエリの繰り返し、メモ化・キャッシュできる重い処理
- **メモリ効率**: 大量データの一括ロード（`Model.all.each` → `find_each`）、ページネーション欠如
- **インデックス**: 新規クエリのWHERE/ORDER BY/JOINキーに対応するインデックスが存在するか（migrationとセットで判断できる場合のみ）
- **同期・非同期**: 重い処理（外部API・メール送信・集計）をリクエスト内で同期実行していないか（ActiveJobへ）
- **SELECT範囲**: 大量行に対する `select("*")` 相当で不要カラム（巨大text等）まで取得していないか

## フロントエンド（Next.js / React）

- **不要な `"use client"`**: インタラクションのないコンポーネントのクライアント化（バンドル肥大・サーバーレンダリング利点の喪失）
- **データ取得のウォーターフォール**: 独立したデータの直列 `await`（`Promise.all` で並列化できないか）、親子コンポーネントでの逐次フェッチ
- **バンドルサイズ**: 巨大ライブラリの全量import（`import _ from "lodash"` → 個別import）、クライアントバンドルに不要な依存が混入していないか
- **画像・動的import**: `<img>` ではなく `next/image`（サイズ最適化・遅延読み込み）、初期表示に不要な重いコンポーネントの `next/dynamic` 化
- **過剰な再レンダリング**: レンダー内での新規オブジェクト・関数・配列生成がメモ化された子に渡っていないか、巨大リストの仮想化欠如、Context値の毎レンダー再生成
- **リクエストの重複**: 同じデータを複数コンポーネントが個別にfetchしていないか（fetchのdedupe・キャッシュ設定・React Query等）

## 指摘の規律

- 計測なしの推測で「遅そう」と言わない。**データ量・呼び出し頻度から影響を見積もれるもののみ**指摘する（例: 一覧画面のN+1は行数分のクエリ増加が確実）
- 早すぎる最適化の要求はしない。明らかなアンチパターン（N+1・全件ロード・直列await）に絞る
