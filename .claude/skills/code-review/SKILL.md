# コードレビュー

以下の観点に基づき、包括的なコードレビューを実施する。

## ワークフロー

### 1. 差分の取得と概要把握

```bash
git diff main...HEAD
git log main..HEAD --oneline
```

変更がない場合はその旨を伝えて終了する。

### 2. CLAUDE.mdの収集

プロジェクトルートおよび変更されたファイルのディレクトリにある CLAUDE.md をすべて読み込む。

### 3. 並列レビュー

以下の6つの観点で並列にレビューを実行する。各観点はTaskツールでSonnetエージェントに委譲する。

#### 1. アーキテクチャ（レイヤード設計）

「presentation → application → domain」の層構造に対する逆方向依存を最優先で検出する。Modelの肥大化よりも逆方向依存の方がテスタビリティとリファクタ容易性を毀損する。

**Critical（レイヤー境界違反 — マージ前必修正）**

- Modelが`Current.user` / `Current.account`等のpresentation層の文脈に依存していないか（バックグラウンドジョブで壊れる）
- Serviceがrequest / paramsオブジェクトを受け取っていないか（presentation層の関心事の漏れ込み）
- Controllerにビジネス計算ロジック（金額計算・割引適用など）が埋め込まれていないか
- Viewが単純なassociation参照を超えてDBクエリを発行していないか
- Mailer・Notification・外部APIへの副作用がModelのcallback（after_commit等）から直接起動されていないか

**Warning（修正または明示的合意が必要）**

- 新規コールバックが5段階スコアで4点以上か（業務プロセスの一手順 = 1〜2点は抽出推奨）
- `skip_*` / `unless: :flag` などの制御フラグでコールバックを抑制していないか
- Concernが振る舞いベースで単独テスト可能か（`Validations`・`Scopes` のような成果物タイプでのcode-slicingは避ける）
- Concernが50行を超えて肥大化していないか
- ドメインロジックがServiceへ流出してModelが貧血化していないか
- Serviceが単なる薄いラッパー（モデルメソッドを呼ぶだけ）になっていないか
- 高頻度変更 × 高複雑度のGod Object兆候はないか

#### 2. コード品質

この観点を実行する前に、以下のローカル rules ファイルを Read で読み込み、各規約に従っているかを確認する。プロジェクトの `.claude/rules/` を優先し、無い場合は `~/.claude/rules/` を確認する。

**必須読み込み（rules がローカルに存在する場合）**

- `.claude/rules/development-principles.md` — 開発原則
- `.claude/rules/rails-principles.md` — Ruby on Rails のコーディング原則

ファイルがどちらの場所にも存在しない場合は、その旨を報告に含めた上で当該 rules に依存する判定をスキップする。

**補助参考（必要に応じて参照、必須ではない）**

- [Ruby Style Guide](https://github.com/rubocop/ruby-style-guide)

#### 3. セキュリティ

- インジェクション脆弱性: SQLインジェクション、コマンドインジェクション、XSSなど
- 認証・認可: アクセス制御の漏れ、権限チェックの不備
- 機密情報の扱い: パスワード・APIキー・トークンのハードコーディングや不適切なログ出力
- 入力バリデーション: 外部入力（ユーザー入力・外部API）の検証漏れ
- 依存ライブラリ: 既知の脆弱性を持つバージョンの使用

#### 4. パフォーマンス

- N+1クエリ: ループ内でのDBクエリ発行、eager loadingの欠如
- 不要な処理: 同じ計算の繰り返し、キャッシュできる重い処理
- メモリ効率: 大量データを一括ロードしているケース（バッチ処理・ページネーション欠如）
- インデックス: クエリに対応するDBインデックスが存在するか（明らかな場合のみ）
- 同期・非同期: 重い処理を同期で実行していないか

#### 5. テスト

- カバレッジ: 変更したロジックに対応するテストが存在するか
- テストの質: 正常系だけでなく異常系・境界値のテストがあるか
- テストの独立性: 他のテストに依存するテストや順序依存がないか
- モックの適切さ: 過度なモックによってテストが意味を失っていないか
- テストの明確さ: テスト名がテストの意図を適切に表しているか

#### 6. ドキュメンテーション

- 複雑なロジックへのコメント: なぜそうしているかの説明が必要な箇所にコメントがあるか
- 公開API・メソッドのドキュメント: 外部から使われるものに説明があるか
- TODOコメント: 放置されたTODOがないか、またはチケット番号が紐付いているか
- CHANGELOG・コミットメッセージ: 変更の意図が追跡できるか（指摘は最小限に）

### 4. 信頼度スコアリング

各指摘に0〜100の信頼度スコアを付与する。

- **0**: 誤検知。精査に耐えない、または既存の問題
- **25**: やや自信あり。実際の問題かもしれないが誤検知の可能性もある
- **50**: 中程度の自信。実際の問題だが、軽微またはニッチ
- **75**: 高い自信。検証済みで、実際に影響がある問題
- **100**: 確実。二重チェック済みで、頻繁に発生する問題

スコア75未満の指摘はフィルタリングする。

### 5. 誤検知の除外

以下は誤検知として除外する:

- 変更前から存在していた問題
- リンター・型チェッカー・コンパイラが検出する問題
- CLAUDE.mdで明示的に抑制されている問題（lint ignoreコメント等）
- 意図的な機能変更に対する指摘
- 変更していない行に対する指摘
- シニアエンジニアが指摘しないような些細なnitpick

### 6. 結果の出力

以下の形式でターミナルに出力する。

指摘がある場合:

```markdown
## コードレビュー結果

N件の指摘:

### 1. [観点] 指摘の要約

**ファイル**: `path/to/file.rb:42`
**信頼度**: 85/100
**詳細**: 具体的な問題の説明と改善案

### 2. ...
```

指摘がない場合:

```markdown
## コードレビュー結果

指摘なし。アーキテクチャ、コード品質、セキュリティ、パフォーマンス、テスト、ドキュメンテーションを確認済み。
```
