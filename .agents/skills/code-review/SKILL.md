# コードレビュー

mainブランチからのローカル差分を、観点別の並列サブエージェントでレビューし、検証済みの指摘のみを報告する。

## ワークフロー

### 1. 差分の取得と概要把握

```bash
git diff main...HEAD
git log main..HEAD --oneline
git diff main...HEAD --name-only
```

変更がない場合はその旨を伝えて終了する。

### 2. CLAUDE.md / AGENTS.md の収集

プロジェクトルートおよび変更されたファイルのディレクトリにある CLAUDE.md と AGENTS.md をすべて読み込む。

### 3. 対象領域の判定

変更ファイル一覧から、実行する観点を決める。

| 領域 | 判定条件 |
|---|---|
| バックエンド | `*.rb`・`Gemfile` の変更 |
| フロントエンド | `*.ts`・`*.tsx`・`package.json` の変更 |
| モデル/スキーマ | `app/models/`・`db/migrate/`・`db/schema.rb`（structure.sql）の変更 |
| スキーマのみ | `db/migrate/`・`db/schema.rb`（structure.sql）の変更 |
| テスト | `spec/`・`test/`・`__tests__/`・`*.test.*`・`*.spec.*` の変更。**または実装変更にテスト変更が伴わない場合**（欠落を指摘するため） |

### 4. 並列レビュー

以下の観点のうち、実行条件を満たすものをすべて**並列で**サブエージェント（Claude Code の `Task`/`Agent` ツール、Codex の `collaboration.spawn_agent` 等）に委譲する。サブエージェント機構が利用できない場合は、同じ観点構成を1観点ずつインラインで評価する。

モデル列の指定はモデル選択に対応した環境でのみ適用し、未対応の環境では無視してよい。

| # | 観点 | 実行条件 | モデル | ルーブリック / 委譲先 |
|---|------|---------|--------|---------------------|
| 1 | アーキテクチャ（Rails） | バックエンド | inherit（指定なし） | エージェント `layered-rails:layered-rails-reviewer` が利用可能ならそれに委譲。不在なら汎用エージェント + `references/architecture-rails.md` |
| 2 | アーキテクチャ（Next.js） | フロントエンド | inherit | `references/architecture-nextjs.md` |
| 3 | モデリング（概念） | モデル/スキーマ | inherit | `references/modeling.md` |
| 4 | 論理設計（テーブル） | スキーマのみ | inherit | `references/logical-design.md` |
| 5 | 正確性 | 常時 | inherit | `references/correctness.md` |
| 6 | コード品質 | 常時 | inherit | 後述「コード品質観点の構成」 |
| 7 | セキュリティ | 常時 | sonnet | 後述「セキュリティ観点の構成」 |
| 8 | パフォーマンス | 常時 | sonnet | `references/performance.md` |
| 9 | テスト | テスト | inherit | `references/testing.md` |
| 10 | ドキュメンテーション | 常時 | sonnet | 後述「ドキュメンテーション観点」 |

#### サブエージェントへの共通プロンプト要件

各サブエージェントのプロンプトには必ず以下を含める。ルーブリックのパスは絶対パスで渡す（`references/` は本スキルのディレクトリ配下）。

1. **レビュー範囲**: `git diff main...HEAD` の差分。変更行に起因する問題のみを対象とし、変更していない行の既存問題は指摘しない
2. **ルーブリックの読み込み**: 指定されたルーブリックファイルを最初にReadし、その全項目を差分に適用する
3. **証拠の引用義務**: 各指摘には該当コードの引用（`file:line` + コード断片）を必ず添える。**該当コードを引用できない指摘は出力しない**
4. **重大度**: 🔴 Critical（マージ前必修正）/ ⚠️ Warning（修正または明示的合意が必要）/ 💡 Suggestion（検討推奨）のいずれかを付与
5. **信頼度**: 各指摘に0〜100のスコアを付与（基準は後述のスコアリング定義をプロンプトに含めて渡す）
6. **出力形式**: 指摘ごとに「観点 / 重大度 / 信頼度 / ファイル:行 / コード引用 / 問題の説明 / 修正案」

#### コード品質観点の構成

コード品質のサブエージェントには以下を順にReadさせる。

1. **開発規約**（プロジェクト `.claude/rules/` を優先、無ければ `~/.claude/rules/`、いずれも無ければ `~/.codex/AGENTS.md` の「開発原則」「Ruby on Rails コーディング原則」セクション）
   - `development-principles.md` — 開発原則
   - `rails-principles.md` — Rails のコーディング原則（バックエンド変更時）
   - どちらの場所にも存在しない場合は、その旨を報告に含めて当該規約に依存する判定をスキップ
2. **結合度**: modularity プラグインの balanced-coupling スキルを正本ルーブリックとしてReadさせる
   - パス探索: `~/.claude/plugins/cache/vladikk-modularity/modularity/*/skills/balanced-coupling/SKILL.md`（グロブで最新版を解決）
   - 適用手法: 変更が触れるコンポーネント間の統合を特定 → 共有知識の種別（intrusive / functional / model / contract）と距離を評価 → `BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY` により**不均衡かつ揮発性が高い統合のみ**指摘する。「全部疎結合にせよ」という指摘は禁止（分解は距離を上げる。強度が十分低い場合のみ推奨できる）
   - スキルが不在の場合の簡易チェック: (a) 別コンポーネントの実装詳細（プライベートメソッド・内部テーブル）への直接依存、(b) 同じ業務ルールの複数箇所への重複実装、(c) 頻繁に変わる箇所と遠い場所（別サービス・別層）との強い結合、の3点のみ確認
3. **TypeScript品質**（フロントエンド変更時のみ）: `references/quality-typescript.md`

#### セキュリティ観点の構成

1. サブエージェントにSkillツールで `security-review` スキル（ビルトイン）を起動させ、main...HEAD の差分を対象にレビューさせる。結果は共通の出力形式（重大度・信頼度・コード引用）にマッピングさせる
2. `security-review` が利用できない場合は `references/security.md` のフォールバックチェックリストを適用
3. フロントエンド変更がある場合は、上記に加えて `references/security.md` の「Next.js 固有の補足チェック」を必ず適用

#### ドキュメンテーション観点

- 複雑なロジックへのコメント: なぜそうしているかの説明が必要な箇所にコメントがあるか
- 公開API・メソッドのドキュメント: 外部から使われるものに説明があるか
- TODOコメント: 放置されたTODOがないか、またはチケット番号が紐付いているか
- CHANGELOG・コミットメッセージ: 変更の意図が追跡できるか（指摘は最小限に）

### 5. 検証パス

サブエージェントの指摘を集約した後、**メインエージェントが各指摘を実際のdiffと突き合わせて検証する**。

- 引用されたコードが実際にdiffの変更行に存在するか確認する。存在しない・変更行でない指摘は破棄
- 指摘内容が引用コードと整合しているか確認する。誤読・文脈の見落としがあれば破棄または信頼度を下げる
- 同一箇所への重複指摘は最も適切な観点に統合する。ただし統合してよいのは**同じ問題**を指す指摘のみ。同一行に独立した問題が複数ある場合（例: 未定義カラムへの代入という正確性の問題と、`Current.user` への依存という層違反）は、それぞれ別の指摘として残す

### 6. 信頼度スコアリング

各指摘に0〜100の信頼度スコアを付与する（サブエージェントの申告値を検証パスで補正した値）。

- **0**: 誤検知。精査に耐えない、または既存の問題
- **25**: やや自信あり。実際の問題かもしれないが誤検知の可能性もある
- **50**: 中程度の自信。実際の問題だが、軽微またはニッチ
- **75**: 高い自信。検証済みで、実際に影響がある問題
- **100**: 確実。二重チェック済みで、頻繁に発生する問題

スコア75未満の指摘はフィルタリングする。

### 7. 誤検知の除外

以下は誤検知として除外する:

- 変更前から存在していた問題（ただし「変更したロジックへのテスト欠落」は、テスト基盤が元から無い場合でも新規の問題として扱い、除外しない）
- リンター・型チェッカー・コンパイラが検出する問題
- CLAUDE.md / AGENTS.md で明示的に抑制されている問題（lint ignoreコメント等）
- 意図的な機能変更に対する指摘
- 変更していない行に対する指摘
- シニアエンジニアが指摘しないような些細なnitpick

### 8. 結果の出力

重大度（Critical → Warning → Suggestion）、同一重大度内は信頼度の降順でソートし、以下の形式でターミナルに出力する。**形式は以下のテンプレートに厳密に従い、日本語で出力する**（サブエージェントの出力言語に引きずられない）。

指摘がある場合:

```markdown
## コードレビュー結果

N件の指摘:

### 1. 🔴 [観点] 指摘の要約

**ファイル**: `path/to/file.rb:42`
**信頼度**: 85/100
**該当コード**:
```ruby
def complete!
  self.completed_by = Current.user
end
```
**詳細**: 具体的な問題の説明と改善案

### 2. ⚠️ ...
```

指摘がない場合:

```markdown
## コードレビュー結果

指摘なし。実行した観点: [実行した観点の一覧]
```

いずれの場合も、末尾に「実行した観点」と「スキップした観点（実行条件を満たさなかったもの）」を1行ずつで明記する。
