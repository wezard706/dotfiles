# モデリング（概念モデリング）レビュールーブリック

## 問い

データモデルはユーザーが業務を理解するためのメンタルモデルである〔杉本啓『データモデリングでドメインを駆動する』〕。
このdiffのモデル（クラス・エンティティ）は業務の写像として正しいか？
実装都合（偶有的複雑性 — 論理削除・テーブル分割・時系列の物理表現など）を概念モデルに持ち込んでいないか？

**適用範囲**: モデルクラス・スキーマ・マイグレーションの変更。テーブルに限らず通常のクラス設計にも適用する。
テーブルの論理構造そのもの（正規化・制約・NULL等）は別観点「論理設計」が扱う。

**注意**: DDDは戦略パターン（ユビキタス言語・境界づけられたコンテキスト）のみ参照する。
戦術パターン（集約・値オブジェクト・リポジトリ等）はRailsと相性が悪いため、この観点では扱わない。

## チェックリスト

### 1. 概念の抽出と命名〔kawasima / DDD戦略〕

- クラス名・テーブル名・カラム名に無意味語が含まれていないか: **情報・データ・処理・〜物・マスタ・記録・管理・履歴**（例: `UserInfo`, `order_data`, `ManagementRecord`）。無くても意味が通じる語は概念の解像度不足のサイン
- エンティティの3大要素が曖昧なまま実装されていないか
  - **単一性**: 何が1つのものか（全6回の連載記事は1記事か6記事か）
  - **同一性**: どこまで変わったら別物か
  - **カテゴリ**: それは何であるか、どんな分類で識別されるか
- 名前がドメインの言葉（ユビキタス言語）と一致しているか。コード上の名前と業務の呼び名の乖離

### 2. リソースとイベントの分類〔kawasima〕

- 新しいエンティティはリソースかイベントか判別できるか。基準: **「〜する」を付けて自然に成り立てばイベント**（注文する○ / 会員する×）。イベント＝業務活動の記録
- 「マスタ」「トランザクション」という曖昧な分類を設計・命名に使っていないか

### 3. 業務活動の混在〔kawasima〕

- 複数の独立した業務活動が1つの概念に押し込められていないか。**1つのエンティティに複数の日時属性**（注文日時・注文確認日時・発送日時の同居）はその典型サイン → 業務活動ごとのエンティティ分離（注文・注文確認・発送）を提案
- 始まりと終わりを持つ長い業務（入会: 申込→審査→会員証発行）は、詳細イベントを個別に表現した上で全体状態を持つ親（ロングタームイベント）で束ねているか

### 4. 隠れたイベントの発掘〔kawasima / zenn〕

- リソースに `updated_at` を持たせたい・更新機能を追加したいという変更は、**洗い出せていない業務活動のサイン**。「会員を更新する」の裏には「会員自身による情報変更」「オペレータによる強制退会」「復会」等の別々の業務が隠れている → それぞれをイベントとして概念化する提案
- 雑な「変更履歴」テーブル・監査ログで業務の解像度を潰していないか（イベントごとに記録すべき属性は異なる）
- ステータスカラムの上書き更新は、状態遷移という業務イベントが概念化されていないサイン

### 5. 区分とサブタイプ〔kawasima〕

- 業務上扱いが変わる区分（現役社員/退職社員、下書き/公開記事）が boolean フラグや散在する条件分岐に埋もれていないか → 区分＋サブタイプとして概念を明示する提案
- モデルに表現されない業務区分は、個々の画面・バッチのコードを読み解かないと分からなくなる

### 6. エンティティ間の存在依存〔kawasima〕

- 関連の設計がカージナリティ（1対多）だけで決められていないか。**存在依存**を確認する: 「部門がなくても社員は存在できるか？」「社員がいなくても部門は存在できるか？」
- 互いに独立して存在できる（非依存）のに一方が他方のIDを直接保持する設計は、「配属前の新入社員」「どの部門にも属さない役員」のような業務事実を表現できない → 関係そのもの（所属・配属）を独立した概念として表現する提案

### 7. コンテキスト境界〔DDD戦略〕

- 別の境界づけられたコンテキストの概念が混入していないか（例: 販売コンテキストの `Product` に在庫管理の属性が生えていく）
- 同じ語が別の意味で使われていないか（多義語の検出）。逆に同じ概念に複数の名前がついていないか
