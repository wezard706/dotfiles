# コード品質（TypeScript / React）レビュールーブリック

> フロントエンド（.ts / .tsx）変更がある場合にコード品質観点へ追加で適用する。
> Ruby側の品質規約は `.claude/rules/`（development-principles.md / rails-principles.md）が正本。

## 型安全性

- `any` の使用: `unknown` + 絞り込み、ジェネリクス、正しい型定義で置き換えられないか
- `as` によるキャストの濫用: 型システムへの嘘。特に `as unknown as T` は設計不備のサイン
- `!`（non-null assertion）: 実際にnullでないことをコードが保証しているか。型ガード・早期returnで表現できないか
- 戻り値型の明示: 公開関数・APIレイヤーの関数に戻り値型があるか（推論任せは意図しない型の伝播を生む）
- 型の重複定義: 同じ形の型がファイルごとに再定義されていないか（共有型への集約）
- `enum` より union 型 / `as const` が慣例（プロジェクト規約に従う）

## React / Hooks

- Hooksのルール: 条件分岐・ループ内での呼び出しがないか
- コンポーネントの責務: 1コンポーネントがデータ取得・変換・表示・フォーム制御を全部抱えていないか（カスタムフック・子コンポーネントへの分離）
- props drilling の深さ: 3階層を超えて素通しされるpropsは構成の見直しサイン（コンポジション・Context）
- 派生状態: propsやstateから計算できる値を `useState` + `useEffect` で二重管理していないか（レンダー中に計算するか `useMemo`）
- key に配列インデックスを使用していないか（並び替え・挿入で状態が壊れる）

## 命名・構造

- boolean は `is/has/can/should` プレフィックス、イベントハンドラは `handle`/`on` の対
- マジックナンバー・文字列リテラルの直書き（定数化・union型化）
- ファイル配置がプロジェクトの慣例（colocation / features / 共有lib）に従っているか

## アクセシビリティ（基本のみ）

- インタラクティブ要素にネイティブ要素を使っているか（`div` + `onClick` ではなく `button`）
- 画像の `alt`、フォーム入力と `label` の関連付け
- キーボード操作を妨げる実装（フォーカス不能な操作要素）がないか
