# テスト レビュールーブリック

## 問い

変更されたロジックに対するテストは存在するか？テストケースの設計とテストコードの実装は正本の原則に従っているか？

## 正本ファイル（RSpec の場合、レビュー前に必ずRead）

以下をプロジェクト → ユーザーの順で探索してReadし、ルーブリックとして適用する。内容はここに複製しない。

| 対象 | 正本 | 探索順 |
|---|---|---|
| テストケース設計の原則（リスクベース・境界値・委譲先を遡らない・振る舞いベース） | test-design スキルの SKILL.md | `.claude/skills/test-design/SKILL.md` → `~/.claude/skills/test-design/SKILL.md` |
| テスト実装の規約（AAA・let外出し・describe/context/it命名・モック規律） | test-implementation スキルの reference.md | `.claude/skills/test-implementation/reference.md` → `~/.claude/skills/test-implementation/reference.md` |

どちらも存在しない場合はその旨を報告に含め、下記の共通観点のみでレビューする。

## 共通観点（言語非依存）

- **カバレッジ**: 変更したロジックに対応するテストが存在するか。実装変更にテスト変更が伴わない場合、その理由が妥当か。**ロジックの変更にテストが1件も追加されていない場合は、それ自体を ⚠️ Warning 以上で必ず指摘する**。プロジェクトにテスト基盤が無い場合も「変更前から存在する問題」として除外せず、今回の変更分へのテスト欠落として報告する
- **ケースの過不足**: 正常系だけでなく異常系・境界値があるか。リスク（影響度×発生確率）に対してテストの厚みが釣り合っているか
- **委譲先を遡らない**: SUTが委譲しているオブジェクトの内部分岐まで重複して網羅していないか（委譲先の戻り値の区分ごとで十分）
- **振る舞いベース**: 内部実装（プライベートメソッド・内部状態）ではなく外から見た振る舞いをアサートしているか
- **独立性**: テスト間の順序依存・共有状態がないか
- **過剰モック**: モックだらけでテストが実装の写しになっていないか
- **明確さ**: テスト名から意図が読めるか

## フロントエンドテスト（Vitest / Jest + Testing Library 想定）

設計原則（リスクベース・境界値・振る舞いベース）は上記共通観点をそのまま適用する。実装面は以下を確認:

- **アクセシブルクエリ優先**: `getByRole` / `getByLabelText` を第一候補に。`getByTestId` の濫用や DOM構造（class・タグ階層）依存のクエリは実装詳細への結合
- **userEvent の使用**: `fireEvent` より `userEvent`（実際のユーザー操作に近いイベント列）
- **非同期の適切な待機**: `findBy*` / `waitFor` を使う。固定 `setTimeout` 待ち・`act` 警告の握りつぶしはflakyのもと
- **実装詳細のテスト回避**: コンポーネントの内部state・内部関数の呼び出し回数ではなく、ユーザーから見える出力（表示・属性・遷移）を検証
- **過剰モック**: 子コンポーネントやhooksを一律モックして「何もテストしていない」状態になっていないか。モックは外部境界（API・タイマー）に限定
- **MSW等によるAPI境界のモック**: fetch自体のモンキーパッチよりリクエストレベルのモックが望ましい（プロジェクトの慣例に従う）
