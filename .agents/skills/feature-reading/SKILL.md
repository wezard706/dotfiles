# feature-reading

大きなコードベースに新しく入ったエンジニアが、**プロダクトの機能とコードの脳内マッピング**を作るためのスキル。読み解いた内容を「抽象レベル」でNotionに蓄積する。

## このスキルの狙いと制約

**目的は脳内マッピングであって、実装ドキュメントではない。** 詳細（メソッドの中身・分岐の全網羅）を書くのはコードを読むのと変わらず価値が薄い。残すべきは「この機能は何で、どの登場人物が関わり、どのデータを触るか」という骨格。だから各セクションは抽象度を高く保つ。

**造語をしない（硬い制約）。** メモとダイアグラムの語彙は、極力「ユーザーが画面で常時目にする言葉（UI用語）」に揃える。AIは放っておくと「登録フロー」「連携処理」のような便利な造語を作りがちで、それだとユーザーが画面で見る言葉とズレて脳内マッピングにならない。UI用語は view テンプレートと i18n ロケールファイルから取得する。**対応するUI用語に確信が持てないときは、勝手に命名せず必ずユーザーに聞く。** ユーザーは「実行が止まること」より「用語が正しいこと」を優先する。

## 前提

- 対象は実アプリのリポジトリ（Rails想定。カレントの作業リポジトリで実行される）。
- 出力先は Notion のデータベース1つ。Notion MCP（`Notion:database-query` / `Notion:create-database-row` / `Notion:create-page` 等）を使う。
- 蓄積先DBがまだ決まっていない場合は、最初にユーザーに「どのデータベースに書き込むか（名前 or URL/ID）」を聞く。以後はそのDBを使い回す。

## ワークフロー

順序依存があるので飛ばさない。

### 1. 対象機能の特定

入力は「機能名（UI用語）」と「エントリポイント（Controller/route/URL/クラス名/関数名）」の両方を受け付ける。

- **機能名のみ**: Grep で該当コードの候補を探す。
  - 候補が1つ → そのまま採用
  - 複数 → `AskUserQuestion` で選択させる
  - 0件 → 「該当コードが見つかりません。画面名・URL・クラス名など手がかりを教えてください」と聞き返す
- **エントリポイントが明示されている**: それを起点に採用。
- トレース方向が曖昧なら（例: ジョブ名だけ渡された）`AskUserQuestion` で「enqueue元から辿るか／ジョブ内部のみか」を確認。

### 2. コード探索（Explore サブエージェント）

エントリポイントから副作用境界までの呼び出しグラフを Explore サブエージェントに構造化して収集させる。**メインコンテキストで直接 Grep/Read して読み込まない**（コンテキストを汚さない）。

サブエージェントに集めさせるもの:

1. **呼び出しグラフ**（正常系＝ハッピーパスのみ）。展開/停止ルールは `sequence-diagram` スキルの `references/explore-prompt.md` と同じ方針:
   - 展開: ビジネスロジック層・エントリポイント層・モデルの副作用メソッド
   - 停止点: DB操作・外部HTTP・ジョブenqueue・ネットワーク境界
   - 省く: 認証/認可・ログ/メトリクス・エラー/例外/rescue/バリデーション失敗などの異常系
2. **登場モデルとアソシエーション**: 呼び出しグラフに現れた ActiveRecord モデルと、その `has_many` / `belongs_to` / `has_one` 等の関連。ER図の材料。
3. **UI用語の材料**: エントリポイント周辺の view テンプレート（erb/haml/slim/jsx 等）のラベル文字列と、参照している i18n キー（`t('...')`）＋対応する ja.yml 等の値。

### 3. UI用語への対応付け（造語の歯止め）

探索で得たコード上の登場人物・機能名を、Step 2 で集めた view / i18n の UI用語に対応付ける。

- 対応が取れる → UI用語を採用（メモ・図のラベルに使う）。
- **対応が取れない・確信が持てない** → 勝手に翻訳・命名しない。
  - UIに現れない機能（バッチ・Webhook受信・内部API・管理者専用処理など）で画面用語が存在しないケースを含む。
  - この場合は `AskUserQuestion` 等で「この処理を何と呼ぶか」をユーザーに確認する。ユーザーが答えるまで造語で埋めない。

### 4. メモ草案の組み立て

以下のセクション構成で草案を作る。抽象度を高く保つ。

1. **概要** — この機能が何を実現するか。1〜3行、ビジネス言語（UI用語）で。実装の言い回しにしない。
2. **シーケンス図**（条件付き） — 下記「セクションの出し分け」参照。
3. **ER図**（条件付き） — 同上。
4. **疑問・要確認** — 読んでいて分からなかった点、確信が持てなかった用語、要確認の仮説。転職直後は「分からなかったことの記録」が後で効く。

> 機能↔コードの対応表は作らない。シーケンス図が登場人物と流れを示すため重複する。

#### セクションの出し分け

シーケンス図とER図は「多くの場合入るが常にではない」。機能の性質でAIが自動判定する（ユーザーが明示指定したらそれで上書き）:

- **フロー中心の機能**（複数の登場人物・非同期・外部連携がある）→ シーケンス図を入れる
- **データ中心の機能**（複数テーブルの関係理解が肝）→ ER図を入れる
- 両方該当 → 両方入れる／どちらも薄い → どちらも省いて概要と疑問だけにしてよい

#### シーケンス図の描き方

- Mermaid `sequenceDiagram`。**概念レベル**で描く。participant はクラス名ではなく UI用語・抽象的役割（例: `ユーザー` / 画面名 / `APIサーバー` / `データベース` / 外部サービス名）。
- **正常系（ハッピーパス）のみ**。異常系・例外・バリデーション失敗は描かない（骨格が見えなくなる）。
- 非同期処理も同じ図に1枚で描く。非同期境界は `--)` と `Note` で明示。トランザクション境界は `rect`。
- participant が8個以上／矢印が30本以上になりそうなら、描く前にユーザーに相談（スコープを絞るか抽象度を上げるか）。

#### ER図の描き方

- Mermaid `erDiagram`。**モデルのアソシエーション由来**で描く。
- **この機能が触れるテーブルのみ**（Step 2 で登場したモデル）。全テーブルは描かない。
- 各テーブルは「テーブル名 + リレーション + 代表カラムのみ」。代表カラム＝PK/FK・状態を表すkeyカラム程度。全カラムは列挙しない（schema.rb を見れば済む）。

### 5. Notion への書き込み

1. 組み立てた草案をユーザーに提示する（Notionに書く前に一度見せる）。
2. 蓄積先DBに対し、機能名で既存行を検索する（`Notion:database-query`）。
3. 分岐:
   - 同名・類似の行が見つかった → 「この行を更新しますか？ 新規で作りますか？」と提示して確認してから書く（誤マッチ・表記ゆれ対策）。
   - 見つからない → 新規行を作成する。
4. Mermaid はコードブロック（言語 `mermaid`）で本文に埋め込む（Notionはネイティブレンダリングする）。
5. 書き込み後、ユーザーに「作成/更新した行のリンク」「入れたセクション（シーケンス図/ER図の有無）」「省いた要素とその理由」「ユーザーに確認したい残りの疑問点」を返す。

#### Notion DB のプロパティ（推奨）

蓄積先DBには最低限これらのプロパティがあると引きやすい。無ければユーザーに用意を促す、または本文だけで運用する:

- **機能名**（タイトル / UI用語）
- **対象サブシステム / BC**（任意）
- **ステータス**（例: 読解中 / 理解済み、任意）
- 更新日時は Notion の自動プロパティを使う

## 出力例（イメージ）

```
# ユーザー登録

## 概要
新規ユーザーがメールアドレスとパスワードでアカウントを作成し、確認メールを受け取る機能。

## シーケンス図
（mermaid: ユーザー → 会員登録画面 → APIサーバー → データベース、確認メールは非同期）

## ER図
（mermaid: users, profiles, email_confirmations の3テーブルと関連・代表カラム）

## 疑問・要確認
- 確認メール未達時の再送導線がコード上見つからない。仕様？
- `CrmSyncJob` に対応するUI用語が不明（バックエンド処理）。要ヒアリング。
```
